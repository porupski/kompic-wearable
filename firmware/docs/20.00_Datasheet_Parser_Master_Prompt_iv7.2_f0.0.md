## ABSOLUTE RULES

1. **One chip at a time, one .md per chip.** File format: `20.XY_[ChipName]_Datasheet_Extract_iv7.2_f0.0_YYYY-MM-DD.md` where:
   - `20.XY` = sequential datasheet number (20.00 = master, 20.01 = first chip, 20.02 = second, …, 20.21 = last)
   - `[ChipName]` = part name (e.g., CO5300, CST9217)
   - `iv7.2_f0.0` = hardware version (locked at 7.2) + firmware version (0.0 porting)
   - `YYYY-MM-DD` = extraction date
   - Save to: `firmware/docs/datasheet_extracts/20.XY_*.md`

2. **Page numbers are mandatory.** Every extracted fact includes `[p. NN]`. If you find a spec, cite the page. If you can't find it, say so explicitly: `[not found in datasheet]`.

3. **Organize by functional area**, not by datasheet structure. Sections: **Pinout**, **I2C/Bus**, **Registers**, **Initialization**, **Operation**, **Interrupts**, **Power**, **Datasheet Notes**.

4. **Cross-reference v7.2, but datasheet is the source of truth.** At the top of your output, compare what the datasheet says against what v7.2 specifies. Flag conflicts. **If they disagree, the datasheet (manufacturer spec) is correct — v7.2 should be corrected**. Document the gap so Ivan can update the PCB doc.

5. **Current consumption is sacred.** Measure it in *every state* the chip supports. Example:
   - Initialization: `X mA [p. 45]`
   - Active polling: `Y mA [p. 47]`
   - Sleep mode: `Z µA [p. 52]`
   - Shutdown: `< 1 µA [p. 52]`

6. **Register maps are tables, not prose.** If a chip has ≤20 registers, include a full table. If > 20, include the "firmware-critical" ones (init, control, status, interrupt, data read) in a table; reference the datasheet appendix for the rest.

7. **No speculation.** If the datasheet is silent on something, say `[datasheet silent on X; recommend bench test]`. Do not guess.

8. **Defects and warnings go in a section at the bottom.** Format: `[WARN-NN] <title> [p. XX]` — any errata, silicon revisions, known issues, timings that are tight, or quirks.

## DEFECT LOG EXAMPLE

```
[WARN-01] FIFO threshold edge case | p. 47 | HIGH
The datasheet says "FIFO threshold 0 = no FIFO mode," but register default is 0. 
If a user never explicitly sets the threshold to 1+, the FIFO remains inactive and 
data is lost. Mitigation: firmware must write FIFO_THRESH = 1 during init; add 
assertion post-init to confirm.

[WARN-02] I2C timeout vs interrupt latency | p. 52, 43 | MED
Datasheet specifies max I2C timeout as 10 ms (p. 52) but INT pin latency can be 
15 ms under worst-case load (p. 43, 400 kHz bus). Mitigation: Use mutexes and 
priority inheritance; consider raising I2C speed to 1 MHz if the bus permits.
```
