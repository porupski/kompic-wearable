## Summary

The BQ25619 is the Mk I's entire power management surface: it handles USB-C charging, the +5V PMID boost that powers LED loads (MAX30101 LEDs, WS2812 status LED, flashlight), battery voltage readback over I2C, and ship-mode entry that drops the device's battery completely from the SYS rail. It does **not** include a coulomb-counting fuel gauge -- state-of-charge has to be derived from voltage. The chip is on its own I2C bus (bus 2, dedicated GPIO4/GPIO5) precisely so it doesn't share contention with the sensor-heavy bus 1.

The old design used the ETA6098 + an external ADC for battery voltage. **All of that is dead.** The BQ25619 is wire-different (I2C instead of ADC), feature-different (boost + ship-mode + fault detection), and protocol-different. There is no driver to port -- only the broker payload shape (`broker_battery_data_t`) carries forward, and even that is enriched with new fields (`charge_state`, `fault`, `boost_enabled`, `power_good`).

Today's deliverable is **driver code + tile + test harness + profiling hooks**, intentionally structurally complete but with every datasheet-derived register address tagged `[DSV]` (datasheet-verify) for bench bring-up. The single largest risk is that the register map below is family-conventional but not personally verified against the BQ25619 datasheet in this session.

### I2C / addresses (v7.2 §I2C Bus Assignment, line 264)

| Item | Value |
|---|---|
| Bus | I2C bus 2 (I2C_NUM_1 in firmware terms) |
| Speed | 400 kHz |
| Pull-ups | 5.1 kΩ -> 3V3 (external) |
| Address | 0x6A (fixed, 7-bit) |
| Co-resident on bus 2 | DRV2605 @ 0x5A (no collision -- different addresses) |

### Power topology (v7.2 §POWER ARCHITECTURE, lines 217-237)

```
USB-C 5V → BQ25619 (switching charger) → LiPo 3.7V
                            └→ SYS (battery or USB, whichever is higher) → TPS62840 → 3V3 to everything
                            └→ PMID boost (FW-gated) → +5V to MAX30101 LEDs / WS2812 / flashlight
```

- Charge current: 1.5 A max.
- Boost current: 1.0 A max @ 5 V.
- Boost / charge are **mutually exclusive** (hardware-enforced by the BQ).
- Battery voltage: read via I2C BATSNS (no external ADC).
- Temperature: 10 kΩ fixed (R14) + 10 kΩ 0402 NTC, NTC physically over the LiPo with thermal vias.

## Code audit

The old design: ETA6098 single-cell linear charger + external ADC on the ESP32 reading a divided BATSNS voltage. None of that code carries forward -- the BQ25619 is the entire stack now.

What DOES carry forward:
- The broker payload identifier `broker_battery_data_t` and the `BROKER_BATTERY_*` API surface. Existing consumers (`lvgl_ui/system_tile.c`, `lvgl_ui/ui_main_screen.c`) read `.voltage` (float V), `.percentage` (uint %), `.charging` (bool). Those fields are preserved verbatim so no consumer-side rework is needed. New fields are additive.
- The task entry point name `task_battery_fn` -- already in `boot_tasks.c` task table (line 83). Same name, new implementation behind it.

What's new in the broker payload:
- `power_good` (bool) -- USB or battery sourcing SYS.
- `charge_state` (uint8_t, `bq25619_charge_state_t`) -- idle / pre-charge / fast-charge / done.
- `fault` (uint8_t) -- raw `REG_FAULT` for diagnostic display.
- `boost_enabled` (bool) -- mirror of `REG_POC[5]`.

## Implementation

Five files in `components/bq25619/`:

### `bq25619.h`

- `BQ25619_ADDR = 0x6A`.
- Register addresses `BQ25619_REG_IINDPM` (0x00) through `BQ25619_REG_VBAT_ADC` (0x0B). **All [DSV]** -- standard BQ256xx family layout; bench bring-up confirms each.
- Bit-position defines for the subset of bits we touch today: `BATFET_DIS` (REG_MISC[5]), `BOOST_EN` (REG_POC[5]), `CHG_CONFIG` (REG_POC[4]), `CHRG_STAT` mask (REG_STATUS[4:3]), `PG_GOOD` (REG_STATUS[2]).
- `bq25619_charge_state_t` enum -- idle / pre-charge / fast-charge / done.
- `broker_battery_data_t` -- carries `.voltage`, `.percentage`, `.charging` (preserved field names) plus the new fields above. `BROKER_BATTERY_TIMEOUT_MS = 5000`.
- Public API:
  - `bq25619_init / deinit`
  - `task_battery_fn` (Core 0)
  - `bq25619_read_reg / write_reg` (caller-holds-mutex helpers)
  - `bq25619_read_vbat_mv` -- single helper for BATSNS readback
  - `bq25619_set_boost(bool)` -- PMID gate for LED loads
  - `bq25619_enter_ship_mode()` -- writes `BATFET_DIS = 1` to REG_MISC
  - `bq25619_soc_from_mv(uint16_t)` -- pure LUT lookup
  - `bq25619_soc_observe(mv, is_charging)` -- self-learning hook (stub today)
  - `bq25619_soc_get_observed_extremes(&min, &max)` -- diagnostics

### `bq25619.c`

- I2C read/write primitives mirror the pattern from `pcf85063.c`. Standard `i2c_cmd_link_*` API with a 20 ms ceiling per transaction.
- **SoC LUT** -- 11-point table from 4.20 V (100%) to 3.30 V (0%) with concave intermediate points modelled on a typical 3.7 V LiPo discharge curve. Linear interpolation between points. Pure function, no I/O.
- **Self-learning hook** -- `bq25619_soc_observe(vbat_mv, is_charging)` tracks observed min/max into static variables. The function is called every polling cycle; today it only updates extremes. Tomorrow (out of scope for Phase 1) it will:
  1. After N stable charge cycles, shift the LUT's 100% endpoint to the observed `max_mv` (within sanity bounds).
  2. Same for the 0% endpoint, gated on "we observed the device shut down on a discharge near this value".
  3. Persist the adjusted endpoints to NVS so the curve is sticky across reboots.
  4. Separate charge-curve vs discharge-curve LUTs (charging voltage reads higher than rest voltage; a single curve over-reports SoC during charging).

  The hook is ABI-stable today, so adding the algorithm later is a `.c` change only.

- `bq25619_init` probes by reading `REG_PART` (0x0A). Logs the raw byte. Once datasheet review pins the part-number / device-revision bit positions, we'll turn it into a real WHO_AM_I check.
- `bq25619_set_boost` is a read-modify-write of `REG_POC`. Holds `g_i2c_mutex` for the full RMW.
- `bq25619_enter_ship_mode` is read-modify-write of `REG_MISC` with `BATFET_DIS = 1`. Logs `WARNING` before the write since the device is about to lose power.
- `task_battery_fn` polls at 1 Hz. Each cycle: read STATUS / FAULT / POC / VBAT_ADC under the mutex (4 single-byte transactions), give mutex, compute SoC + decode bits, write broker. Notably uses `I2C_NUM_1` (bus 2) -- the rest of the system uses `I2C_NUM_0` (bus 1).

### `battery_tile.{c,h}`

LVGL settings tile sized for the CO5300 410x502 panel from day one (same constants as the resized `rtc_tile.c`). Six data rows:

```
[LED] BQ25619  Li-ion charger + PMID boost
─────────────────────────────────────────
Vbat:   3.92 V
SoC:    72 %
State:  Fast charge
Power:  USB OK
Fault:  0x00
Boost:  OFF
```

Status LED maps `sensor_status_t` -> colour: ONLINE green, NOTIF purple (low battery), OFFLINE red, STALE orange. Registered into the tile registry via `battery_tile_desc`.

### `test/test_bq25619.c`

Five-phase harness:
1. I2C bus 2 init at 400 kHz on GPIO4/GPIO5.
2. `bq25619_init` -- logs the raw `REG_PART` byte.
3. Register dump: STATUS, FAULT, POC, MISC.
4. Vbat read + SoC compute + charge-state decode + power-good check; runs `bq25619_soc_observe` and logs the running extremes.
5. Boost ENABLE -> POC dump -> Boost DISABLE -> POC dump.
6. **Ship-mode write is PRINTED ONLY.** The harness does not call `bq25619_enter_ship_mode()` -- a successful call drops the device off the bench. The byte that would have been written is logged so the operator can spot-check it.

Profiling hooks: `esp_timer_get_time()` around bus init, BQ init, Vbat read. Stack high-water + free heap logged at exit.

### `CMakeLists.txt`

`idf_component_register` with `REQUIRES driver freertos esp_timer data_broker lvgl esp_lvgl_port`. Mirrors the pcf85063 component shape (it owns both a driver and a tile, same as this one).

## Profiling

### Boot-time cost

| Phase | Component | Time | Method |
|---|---|---|---|
| Init | I2C bus 2 setup | **TBD (no hw)** | `esp_timer_get_time()` in test harness |
| Init | `bq25619_init` (single REG_PART read) | **TBD** | wall time around init call |
| Op   | First broker write (1-second tick latency) | ~1000 ms | governed by `BATTERY_POLL_MS = 1000` |

### Per-operation cost

| Operation | Time | Bus | Notes |
|---|---|---|---|
| Single register read (write 1 reg + read 1 byte) | **TBD** | 400 kHz | wire-time floor ~120 us; entire poll cycle = 4 reads = ~500 us |
| `bq25619_set_boost` (1 read + 1 write under mutex) | **TBD** | 400 kHz | ~240 us wire time + RMW logic |
| `bq25619_enter_ship_mode` | **TBD** | 400 kHz | one RMW; followed by a graceful ~10 s BATFET delay before actual shutdown |
| SoC LUT lookup | < 5 us | n/a | pure C, 11-element linear scan + interp |

### Memory

| Metric | Value | Notes |
|---|---|---|
| Driver state | ~16 bytes | static `s_vbat_min_mv` / `s_vbat_max_mv` + log tag |
| LUT | 11 × 3 bytes = 33 bytes | const, lives in flash |
| Battery task stack | 3072 bytes allocated, high-water **TBD** | small float math + 4 register reads -- expect ~1500 bytes used |
| Tile widgets | LVGL accounts for these in PSRAM; ~10 lv_label_t handles | per-tile cost ~600 bytes |

### Current draw estimate

| State | Current (mA) | Notes |
|---|---|---|
| Charge active | up to 1500 mA at chip input | depends on USB negotiation; BQ enforces ceiling |
| Boost active (LED loads) | up to 1000 mA on PMID @ 5 V | mutually exclusive with charge |
| Idle (no charge, no boost) | sub-mA chip standby | dominated by quiescent current of the buck + leakage |
| Ship-mode | sub-µA | BATFET open -- whole device is off the battery |

### Notes

- **The polling rate (1 Hz) is conservative.** Battery state doesn't change meaningfully faster than that. Cranking it up would just waste I2C bus 2 time, and bus 2 is shared with DRV2605 (haptic) which can be latency-sensitive.
- **The single most load-bearing number** for the user-facing battery indicator is the SoC LUT accuracy. The self-learning hook is the path to making that better over time; until then, we're at "+/- 5 percentage points" accuracy at best.
- **Boost is FW-gated.** Anything that turns on PMID-powered loads (flashlight, WS2812, HR LEDs) must call `bq25619_set_boost(true)` first and turn it off afterwards. Forgetting to disable wastes battery; enabling while a charge cycle is active is silently ignored by the chip (mutually exclusive).

## Defects discovered

- `[DEFECT-001] All register addresses tagged [DSV] -- datasheet-verify | components/bq25619/bq25619.h | CRITICAL | Every register address (0x00..0x0B) and most bit positions (BATFET_DIS, BOOST_EN, CHG_CONFIG, CHRG_STAT mask, PG_GOOD) are family-conventional but NOT verified against the BQ25619 datasheet in this session. Disposition: BLOCK BENCH BRING-UP until each is confirmed against TI SLUSDH8 / SLUSDS3. If even one differs, the polling task will read garbage and the SoC indicator will be wrong without any error.`
- `[DEFECT-002] SoC LUT is a first-pass approximation | components/bq25619/bq25619.c:s_soc_lut | HIGH | The 11-point Vbat -> SoC table is modelled on a generic 3.7 V single-cell LiPo discharge curve under light load. Real cells vary, and the device's load profile is dominated by the AMOLED panel which is content-dependent. Disposition: accept first-pass for v1, then plug the self-learning hook to refine. Expect ~5-10% error in the indicator until that lands.`
- `[DEFECT-003] Self-learning hook is a stub | components/bq25619/bq25619.c:bq25619_soc_observe | MED | The function tracks observed extremes but does not yet shift LUT endpoints or persist to NVS. The API is in place; the algorithm is not. Disposition: deferred to a Phase 1.5 follow-up. Not blocking v1.`
- `[DEFECT-004] Ship-mode entry not exercised in test harness | test/test_bq25619.c | MED | The test prints the byte that would be written but does not call bq25619_enter_ship_mode(). A successful call drops the device, and getting back requires USB-C insert or a GPIO16/QON long press. Disposition: exercise manually on bench with USB-C plugged in (USB power keeps the device alive even with BATFET open).`
- `[DEFECT-005] VBAT_ADC scaling assumed (20 mV step, 2304 mV offset) | components/bq25619/bq25619.c:bq25619_read_vbat_mv | HIGH | The conversion is a family-typical 20 mV step + 2304 mV offset. The BQ25619 datasheet may use different ADC scaling (12 mV step is also common in TI parts). Disposition: confirm on bench by sweeping a bench-supplied Vbat and reading the register; correct the conversion before trusting SoC. Test 4.20 V (should read ~100), 3.70 V (should read ~22), 3.30 V (should read ~0).`
- `[DEFECT-006] g_i2c_mutex via extern, not header | components/bq25619/bq25619.c | LOW (pre-existing) | Same pattern as cst9217.c / bme280_drv.c. Resolves when boot_hw_init.h lands.`

## Open questions

1. **BQ25619 datasheet locally available?** A single datasheet review pass would collapse [DEFECT-001] and [DEFECT-005] and downgrade most of this module's risk profile. Recommend: scheduling a 30-minute datasheet read for the next session.
2. **data_broker integration.** The carried-forward `data_broker.h` includes `app_battery.h` for `broker_battery_data_t`. That header does not exist in this tree; today the broker is presumably non-building. The clean fix is one line: change `data_broker.h` to include `"bq25619.h"` instead. Plus `data_broker/CMakeLists.txt` needs `REQUIRES bq25619`. Both are out of scope for this commit (they touch the also-untracked data_broker dir). Flag for the boot_hw_init integration pass.
3. **Boost gate ownership.** Today, `bq25619_set_boost(true/false)` is a public function. Anyone can call it. We probably want a thin reference-counted gate so flashlight / HR / WS2812 don't fight each other over PMID. Recommend: defer to Phase 4 when the actual consumers exist.
4. **Charging-aware SoC display.** A cell at 4.05 V under charge reads ~80% on the LUT, but reverts to ~70% within 30 s of unplug. UI should consider showing voltage + state ("Fast charge") more prominently than SoC during the charge phase, or use a separate charge LUT (per the [DEFECT-002] / [DEFECT-003] roadmap).
5. **Fault display.** The `fault` field surfaces the raw `REG_FAULT` byte. Should we decode the bits in `battery_tile` for user-friendly text ("NTC fault", "Watchdog timeout") or keep it as a 0x.. hex value for engineers? Recommend: hex for v1, decode later.
