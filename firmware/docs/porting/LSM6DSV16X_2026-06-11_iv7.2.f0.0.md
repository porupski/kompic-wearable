## Summary

The LSM6DSV16X is the Mk I's motion brain: 6-axis accel + gyro plus an embedded function block for sensor fusion, tap, tilt, free-fall, wake-up, and step-count features. It sits on I2C bus 1 at 0x6B and has a wake-capable INT1 line on GPIO8 -- the canonical "raise to wake" path. The driver replaces the v5 QMI8658 implementation at the chip layer while preserving everything above it: the broker payload shape, the complementary-filter math, the haptic-sweep resonance-detect path that reads `broker_imu_data_t.accel_z`, and the `imu_tile` consumer.

This is the cleanest possible swap. Everything that was chip-independent (the filter, the broker contract, the haptic dependency on accel_z) carries forward verbatim. Everything that was chip-specific (the WHO_AM_I value, the register addresses, the ODR / FS bit positions, the scaling constants) is new.

Today's deliverable: replaced driver code + identity-renamed-and-resized tile + standalone test harness + this doc. As with BQ25619, every register address and bit position is tagged `[DSV]` because the ST datasheet has not been personally audited in this session.

| Signal       | GPIO | Direction | Pull / boot | Notes |
|---|:--:|---|---|---|
| LSM_SDA      | 1    | OD bidi   | external 5.1 kΩ -> 3V3 | I2C bus 1 SDA, shared. |

| LSM_INT2     | --   | --        | --                  | Not routed on Mk I; v7.2 line 141 mentions LSM_INT2 listed only as a "noted-but-unused" pin. |

### I2C / addresses

| Item | Value |
|---|---|
| Bus | I2C bus 1 (`I2C_NUM_0`) |
| Speed | 400 kHz |
| Address | 0x6B (fixed, 7-bit) |
| WHO_AM_I | 0x70 at register 0x0F |

### Power topology (v7.2 lines 232-233)

| Rail | Connected | Notes |
|---|---|---|
| 3V3  | LSM6DSV16X VDD (analog)        | TPS62840 buck (always-on while battery present) |
| 1V8  | LSM6DSV16X V_IO (digital I/O) | XC6206 LDO; rules out direct 3.3V I/O |

The 1V8 V_IO rail is a small subtle gotcha: the ESP's I2C lines are 3V3, so the line-side pull-ups at 5.1 kΩ -> 3V3 are correct (the chip is open-drain and tolerates the higher pull-up voltage as long as its V_IO supply is correct). Bench bring-up should sanity-check that the chip's V_IO LDO is alive before talking to it.

## Code audit

### What dies

`components/lsm6dsv16x/qmi8658.{c,h}` -- deleted. The QMI8658 driver was the v5 carry-forward and is wire-incompatible with the LSM6DSV16X (different WHO_AM_I value, different register addresses for OUTX_L_G / OUTX_L_A / CTRL registers, different scaling constants, different INT routing semantics).

### What carries forward

- **`broker_imu_data_t` shape (exact field-for-field).** Defined now in `lsm6dsv16x.h` instead of `qmi8658.h`. Same accel_x/y/z + gyro_x/y/z + roll_deg + pitch_deg + temperature + bookkeeping. This means `haptic.c` (which reads `bd.accel_z` for LRA sweep resonance) and any other broker consumer build unchanged.
- **`task_imu_fn` -- the complementary-filter math.** The atan2-based accel-derived static angle, the gyro-integration + 0.98 alpha blend, the first-read seed. Verbatim from `qmi8658.c`. The filter doesn't care which chip produced the floats.
- **`imu_tile.{c,h}` -- the LVGL UI.** Only changes: (a) `#include "lsm6dsv16x.h"` instead of pulling identity strings from a `qmi8658_*` call, (b) literal "QMI8658C" / "6-axis IMU" header strings replaced with calls to `lsm6dsv16x_get_chip_name()` / `_get_chip_desc()`, (c) layout constants rescaled from 240x280 to 410x502.
- **Boot-tasks table entry** (`task_imu` row in `boot_tasks.c`). The extern declaration in that file already points to "qmi8658.c" in a comment -- pure cosmetic and out of scope today.

### What's new

- WHO_AM_I = 0x70 (was 0x05 for QMI8658).
- Register layout per ST DS13176: TEMP at 0x20, GYRO at 0x22, ACCEL at 0x28; CTRL1 = accel config, CTRL2 = gyro config, CTRL3 = BDU + IF_INC + reset.
- Scaling: 8192 LSB/g (datasheet) vs the empirical 4096 LSB/g the QMI8658 needed.
- Gyro range default ±2000 dps with 70 mdps/LSB sensitivity (was ±64 dps on QMI8658 at 512 LSB/dps -- a much narrower range).
- Temperature offset of +25 °C (LSM6DSV16X reports temp delta-from-25C in its register; QMI8658 reported temp directly).
- `lsm6dsv16x_install_int1_isr(TaskHandle_t)` -- new public API for wake-on-motion. Today's skeleton routes the embedded-function INT1 line as a falling-edge GPIO interrupt; the actual embedded-function configuration (WAKE_UP_THS, WAKE_UP_DUR, TAP_CFG, MD1_CFG) is **not done yet** -- pending the datasheet review.

## Implementation

### `components/lsm6dsv16x/lsm6dsv16x.h`

- Wire-level constants `LSM6DSV16X_I2C_ADDR = 0x6B`, `LSM6DSV16X_WHO_AM_I_VAL = 0x70`, `LSM6DSV16X_INT1_GPIO = GPIO_NUM_8`.
- Register-address `#define`s for the registers we touch today, all tagged `[DSV]`.
- Control-register bit defines for `CTRL3_BDU / IF_INC / SW_RESET`, `CTRL1_ODR_120HZ / FS_XL_4G`, `CTRL2_ODR_240HZ / FS_G_2000DPS`, `INT1_DRDY_XL / DRDY_G`, `MD1_INT1_WU`, `TAP_CFG0_LIR`, `TAP_CFG2_INT_EN`.
- Scaling constants `ACCEL_LSB_PER_G = 8192`, `GYRO_LSB_PER_DPS = 14.286`, `TEMP_LSB_PER_C = 256`, `TEMP_OFFSET_C = 25.0`.
- `broker_imu_data_t` -- field-for-field preserved.
- Public API: `lsm6dsv16x_init / get_chip_name / get_chip_desc / read / install_int1_isr` + `task_imu_fn`.

### `components/lsm6dsv16x/lsm6dsv16x.c`

- I2C helpers `write_reg / read_reg / read_regs` using the modern `i2c_master_write_to_device` / `write_read_device` APIs.
- `lsm6dsv16x_init` -- takes `g_i2c_mutex` internally, verifies WHO_AM_I, soft-resets via CTRL3.bit0, gives mutex while waiting 20 ms for reset, re-takes mutex to write CTRL3 (BDU + IF_INC), CTRL1 (120 Hz, ±4g), CTRL2 (240 Hz, ±2000 dps), INT1_CTRL (0 -- DRDY routing off by default).
- `lsm6dsv16x_read` -- 14-byte burst from 0x20 (TEMP + GYRO + ACCEL, ST register layout). Scaling identical to the public header constants.
- `lsm_int1_isr` -- IRAM, notify-only. `lsm6dsv16x_install_int1_isr` configures GPIO8 as pull-down posedge (LSM6DSV16X INT idles LOW), installs the GPIO ISR service idempotently, adds the handler. Pass NULL to uninstall.
- `task_imu_fn` -- structure verbatim from `qmi8658.c`'s implementation: read-before-write, 50 Hz poll, complementary filter with CF_ALPHA = 0.98. Static seeding from accel on first valid read.

### `components/lsm6dsv16x/imu_tile.c`

- One-time identity-string update: literal `"QMI8658C"` / `"6-axis IMU"` replaced with `lsm6dsv16x_get_chip_name()` / `_get_chip_desc()`.
- Layout constants extracted into named locals at the top of `imu_tile_init()` so the rescale is one block of `#define`s. Y positions scaled by ~1.79 (240x280 -> 410x502). X positions and switch size bumped proportionally. Old values preserved in comments.

### `components/lsm6dsv16x/CMakeLists.txt`

- `SRCS` updated: `qmi8658.c` -> `lsm6dsv16x.c`. `imu_tile.c` unchanged.

### `test/test_lsm6dsv16x.c`

Standalone harness:
1. I2C bus 1 init at 400 kHz on GPIO1/2.
2. `lsm6dsv16x_init` with WHO_AM_I check.
3. 10 successive `lsm6dsv16x_read` calls at 50 ms spacing, logging accel/gyro/temp + per-read I2C dt.
4. INT1 ISR install, 15 s observation window; each event logged with timestamp.
5. ISR uninstall, stack high-water + heap report, exit.

### Per-operation cost

| Operation | Time | Bus | Notes |
|---|---|---|---|
| WHO_AM_I read (1 byte) | **TBD** | 400 kHz | wire-time floor ~120 us |
| 14-byte burst read (TEMP + GYRO + ACCEL) | **TBD** | 400 kHz | wire-time floor ~420 us |
| Complementary filter (per cycle) | **TBD** | n/a | floats + 2x atan2 + 1x sqrt; expect ~50 us on the S3 with FPU |
| ISR -> task notify | **TBD** | n/a | FreeRTOS notify; 5-30 us |

### Memory

| Metric | Value | Notes |
|---|---|---|
| Driver state          | ~24 bytes | filter state (3 floats) + ISR task handle + bool |
| `task_imu` stack      | 4096 bytes allocated; high-water **TBD** | 14-byte raw buffer + atan2/sqrt locals |
| Tile widgets          | LVGL-accounted; ~14 lv_obj handles | per-tile ~800 bytes in PSRAM |

### Current draw estimate

| State | Current | Notes |
|---|---|---|
| Both sensors on, 120/240 Hz, no LPF | ~700-900 µA | datasheet-typical for an LSM6DSV at these rates |
| Accel only, 26 Hz, LP | ~75 µA | not used today, but available for power-save |
| Sleep + tilt wake armed | low single-digit µA | embedded-function-only mode; awaits config |

### Notes

- **The complementary filter is the only float-heavy code in the task.** With the FPU it's free; without it, it would be the bottleneck. ESP32-S3 has an FPU on each core, so we're fine.
- **Gyro at ±2000 dps is wide for a watch.** Reduces resolution from what ±125 dps would give. If we ever need sub-degree gyro precision (we won't, for tilt detection), tightening the range is a single CTRL2 write.

## Defects discovered

- `[DEFECT-001] All register addresses + bit positions tagged [DSV] -- datasheet-verify | components/lsm6dsv16x/lsm6dsv16x.h | CRITICAL | The register addresses (0x0F WHO_AM_I, 0x10 CTRL1, 0x11 CTRL2, 0x12 CTRL3, 0x20-0x2D OUT block) and the ODR / FS bit position layouts (e.g. CTRL1_FS_XL_4G = (0x02 << 0)) are ST-family conventional but NOT verified against the LSM6DSV16X datasheet (ST DS13176) in this session. Disposition: BLOCK BENCH BRING-UP until each is confirmed. If WHO_AM_I or CTRL3 reset semantics differ, the init returns ESP_FAIL and the broker stays SENSOR_OFFLINE.`
- `[DEFECT-002] Embedded-function wake-up not configured | components/lsm6dsv16x/lsm6dsv16x.c:lsm6dsv16x_init | HIGH | The brief specifies "raise-to-wake gesture (tilt detection, wakeup-on-motion)". Today's init does not write WAKE_UP_THS / WAKE_UP_DUR / TAP_CFG[012] / MD1_CFG. The INT1 ISR install path is functional but the chip is not configured to ASSERT INT1 on motion. As a result, the test harness's "tilt window" will see zero events on real hardware until the embedded-function block is configured. Disposition: defer to a Phase 2.5 follow-up after datasheet review. Place the embedded-function setup behind a dedicated `lsm6dsv16x_enable_wake_on_motion(threshold_mg)` API once known.`
- `[DEFECT-003] INT1 polarity assumed posedge | components/lsm6dsv16x/lsm6dsv16x.c:lsm6dsv16x_install_int1_isr | MED | The ISR is configured for posedge with pull-down (assumes INT idles LOW per LSM6DSV family default). The LSM6DSV16X can be inverted via INT_NOTIFICATION / INT_CFG1.H_LACTIVE; if a previous power-up state left that flipped, our edge sense is wrong. Disposition: confirm on bench by scoping GPIO8 during a known-motion event.`
- `[DEFECT-004] Gyro range default ±2000 dps wider than QMI8658 was | components/lsm6dsv16x/lsm6dsv16x.h:CTRL2_FS_G_2000DPS | LOW | Old chip ran ±64 dps; we now run ±2000 dps. Resolution drops from 512 LSB/dps to ~14 LSB/dps. For tilt/orientation that's fine; for delicate gesture detection it might not be. Disposition: defer; tighten to ±500 dps later if gesture detection requires it.`
- `[DEFECT-005] V_IO 1V8 rail not exercised by firmware | components/lsm6dsv16x/lsm6dsv16x.c | LOW | The LSM6DSV16X's digital interface runs from a separate 1V8 rail (XC6206 LDO, v7.2 line 233). Firmware has no way to verify the LDO is alive prior to the WHO_AM_I read; if the rail is dead the I2C transactions will fail and we'd attribute it to a chip issue. Disposition: documentation-only; bench bring-up should verify the LDO with a scope before any IMU debugging.`
- `[DEFECT-006] g_i2c_mutex via extern | components/lsm6dsv16x/lsm6dsv16x.c | LOW (pre-existing) | Same pattern as cst9217.c / bq25619.c. Resolves when boot_hw_init.h lands.`

## Open questions

1. **LSM6DSV16X datasheet locally available?** A single datasheet review pass would collapse [DEFECT-001] / [DEFECT-002] / [DEFECT-003] and downgrade this module from CRITICAL to MED.
2. **Tilt-vs-wake-up.** The LSM6DSV family has TWO motion-detection paths: the simple "wake-up" function (programmable threshold + duration in WAKE_UP_THS/DUR) and the more elaborate "tilt" function (state machine in the embedded function block). For raise-to-wake on a watch, "tilt" gives better UX (only fires on a deliberate gesture), but it requires more config writes. Recommend: implement wake-up first (simpler, faster to ship), upgrade to tilt later.
3. **Embedded function block usage.** LSM6DSV16X also exposes 6D orientation, free-fall, single-tap, double-tap, step counter, significant motion, and a full sensor-fusion algorithm with quaternion output. Today we use none of it. Question: do we want any of these as Phase 2.5? Step counter is particularly cheap (one IIR filter the chip runs without ESP intervention).
4. **Qvar (ECG) block.** Per the master prompt Phase 5, the LSM6DSV16X's QVar block can read electrical body impedance for a basic ECG signal. This is the "ECG via IMU" Phase 5 entry. Out of scope today; flagged.
5. **Read-before-write contention.** `task_imu_fn` reads from the broker, then writes back -- the `enabled` field is only modified by Core 1 (tile callback). Between the read and the write, Core 1 could change `enabled`. The window is small (<1 ms) but non-zero. Same pattern was present in QMI8658 and has not caused observable issues; flagging for completeness.
