## Summary

The PCF85063A is the Mk I's time authority. It is battery-backed (ML621 cell + 1N4148 trickle charge), holds time across full-system power loss, and is the source from which the watch face draws its truth. GPS is a periodic calibration source; the RTC chip is what the firmware actually reads on every UI tick.

This module is **the smoothest port in the queue**. The chip is identical to the v5 design, the driver carries forward verbatim, and the broker / tile integration is already in place. Today's deliverable is therefore split:

1. **Audit + document** the carried-forward driver -- confirm it satisfies v7.2 and Blueprint 8 §2 (UTC-only storage), record what was kept and what changed.
2. **Extend** the driver with a hardware-alarm path on GPIO15 -- the v7.2 callout names GPIO15 as the "Wake: alarm" source, but the existing carried-forward code had no alarm support. This is new.
3. **Resize** `rtc_tile.c` from the old 240x240 ST7789 sizing to the new 410x502 CO5300 panel, per the brief.

The hardware alarm is **orthogonal to the existing software alarm** (`components/alarm/`), which polls broker_rtc and is the in-session UX path. The hardware path is intended for future deep-sleep wake-on-alarm and is wired in but not consumed by application code today.

## Code audit

The `components/pcf85063/` tree was already in place when this port started:

- `pcf85063.c` -- 230 lines, carry-forward from v5.
- `pcf85063.h` -- public API, broker struct, Phase-15 contract docs.
- `rtc_tile.c` -- LVGL settings tile, sized for the old 240x240 ST7789.
- `rtc_tile.h` -- tile public API + tile_desc_t registration.
- `CMakeLists.txt` -- idf_component_register, REQUIRES driver / data_broker / lvgl / esp_lvgl_port / boot_logic.

**What carries forward intact (no edits required):**

- BCD helpers (`bcd_to_dec`, `dec_to_bcd`).
- 7-byte burst register read/write spanning REG_SECONDS..REG_YEARS.
- Oscillator-stop-flag detection at probe time (bit 7 of REG_SECONDS).
- `pcf85063_get_time` / `pcf85063_set_time` -- straight I2C, valid bit set on read failure.
- `pcf85063_sync_utc` -- the Phase-15 fix that replaced `pcf85063_sync_from_gps()`. Writes UTC directly, computes weekday via Tomohiko Sakamoto's pure-integer algorithm. Honours Blueprint 8 §2 ("RTC ALWAYS stores UTC").
- `task_rtc_fn` -- pinned to Core 0 (Blueprint 1 §1), 1 Hz poll, takes `g_i2c_mutex` (50 ms ceiling) for each read, writes `broker_rtc_data_t` to the broker. Honours the "lock held for memcpy only" rule.
- Broker integration -- `broker_rtc_data_t` + `BROKER_RTC_TIMEOUT_MS` (3000) defined in pcf85063.h per the "thin broker, fat driver" pattern (Blueprint 4 §2). The broker macros generate the read/write/status/enabled functions in data_broker.c.

**Where the brief diverges from the existing code:**

The brief says: "Boot-time UTC seed: `timegm()` (not mktime + TZ env hack)." The existing driver does **not** use `timegm()`. Instead, it uses Tomohiko Sakamoto's integer-only day-of-week algorithm. This is functionally **equivalent for what the RTC needs** (correct weekday for any Gregorian date, no libc dependency, no TZ env state) and arguably better -- it avoids libc time-zone state entirely, where `timegm()` is a glibc extension (not POSIX) and on some embedded toolchains is either absent or implemented via `setenv("TZ", "UTC")` + `mktime` + `unsetenv`, which is the very hack the brief calls out as wrong.

Disposition: **keep Sakamoto**, document the divergence, escalate as Open question Q1.

## Implementation

### Driver -- unchanged (audit only)

The carried-forward `pcf85063.c` / `pcf85063.h` time-read/write/sync paths were audited line-by-line against v7.2 and Blueprint 8. No changes required.

### Driver -- new hardware alarm path

Three new public functions in `pcf85063.h`, implemented in `pcf85063.c`:

- `pcf85063_set_alarm(i2c, h, m, s)` -- writes the H/M/S alarm registers (REG_HOUR_ALARM / REG_MINUTE_ALARM / REG_SECOND_ALARM, BCD, AEN bit cleared = enabled), masks out day-of-month and weekday (AEN bit set = disabled, fires every day at the given time), sets `AIE` (CTRL_2 bit 7), clears `AF` (CTRL_2 bit 6). Takes `g_i2c_mutex` internally.
- `pcf85063_clear_alarm(i2c)` -- masks out every alarm register's AEN bit and sets CTRL_2 to 0. Disables AIE, clears AF.
- `pcf85063_clear_alarm_flag(i2c)` -- read-modify-write CTRL_2 to clear AF while preserving AIE. The application calls this after handling an alarm to re-arm the INT line.
- `pcf85063_install_alarm_isr(task)` -- configures GPIO15 as pull-up + falling-edge input, installs `gpio_install_isr_service` (idempotent), registers a `vTaskNotifyGiveFromISR`-style handler that notifies the given task. Passing NULL uninstalls.

Two new private helpers (`i2c_write_reg` / `i2c_read_reg`) for single-byte register I/O, since the carried-forward code only had the 7-byte burst path.

ISR is `IRAM_ATTR`, does nothing except notify-and-yield. No logging, no I2C, no malloc.

### `rtc_tile.c` -- resize for 410x502

Pixel positions and the divider width were rescaled. Constants now live in named locals at the top of `rtc_tile_init()` so the next reshuffle is one edit, not five:

| Constant | Old (240x240) | New (410x502) | Rationale |
|---|---:|---:|---|
| X_MARGIN | 12 | 20 | proportional + small headroom for round-panel masking |
| LED_SIZE | 12 | 16 | bigger panel, bigger pixel; LEDs at 12 px would look thin |
| HEADER_X | 30 | 44 | LED_X + LED_SIZE + 8 px |
| DIVIDER_W | 216 | 360 | scaled 1.67x; leaves ~25 px on each side |
| DIVIDER_Y | 34 | 50 | scaled vertically to leave room under header |
| ROW0_Y | 44 | 64 | scaled |
| ROW_STEP | 22 | 32 | proportional + room for new (likely larger) UI_FONT_LABEL |

The tile is **not** round-mask aware. If `lvgl_ui.c` applies a circular mask to the tile container (likely, for the CO5300 round window), corners are clipped automatically. If not, the corners at y > ~380 may sit outside the inscribed circle. Open question Q3.

### Test harness -- `test/test_pcf85063.c`

Standalone `app_main()` covering four phases:

1. **Probe + init.** I2C bus init (400 kHz on GPIO1/2), `pcf85063_init`, wall time logged.
2. **Round-trip.** Write 12:34:56 2026-06-10 Wed, read back, log, check fields match (allow second to advance by 1 due to test latency).
3. **Midnight rollover.** Set 23:59:55 2026-06-10 Wed, sleep 8 s, read, confirm date advanced to 2026-06-11 and the chip's weekday counter advanced by 1.
4. **Hardware alarm.** Install ISR notifying the test task, arm alarm for now+5 s, wait up to 8 s for INT, log arm-to-fire dt, clear AF, clear alarm, uninstall.

Profiling hooks: `esp_timer_get_time()` around bus init, RTC init, every `set_time`, every `get_time`, the arm-to-fire window. Stack high-water and heap delta logged at exit.

### Current draw estimate

| State | Current (mA) | Notes |
|---|---|---|
| Normal (main rail up, RTC ticking) | TBD | dominated by chip standby draw (datasheet: ~150 nA typical) |
| Backup (main rail down, ML621 powering RTC) | sub-µA | per v7.2, ~4 years on 5.5 mAh -- limit set by self-discharge, not draw |
| Alarm armed (AIE set, no match) | TBD | should be identical to "normal" -- AIE only affects the INT pin |
| Alarm firing (INT held LOW until AF cleared) | TBD | INT current is set by the pull-up + external load; pin sources nothing on its own |

### Notes

- **Backup-cell life dominates the power story.** Anything other than reliable sub-µA standby is a defect. The chip handles this in hardware; firmware just has to not poke it.
- **The 1 Hz poll is enough.** RTC second resolution is the data; polling faster wastes I2C bus time without improving the broker payload. Confirmed against Blueprint 5 §3.
- **The hardware alarm is dead weight today.** It is plumbed in but not consumed. Worth keeping for the deep-sleep wake path; if v1 ships without ever exiting deep sleep, this can be dead code with no cost.

## Open questions

1. **Sakamoto vs `timegm()`.** The brief specifies `timegm()`. The carried-forward driver uses Tomohiko Sakamoto's day-of-week algorithm. Sakamoto is correct, libc-free, and side-effect-free; `timegm()` is glibc-only and frequently faked via TZ-env munging on embedded toolchains -- which is the exact failure mode the brief warns against. **Confirm: keep Sakamoto?** If Ivan wants a Unix-epoch helper for some other reason, that's a separate function, not a replacement for the DoW computation.
2. **Hardware alarm wake.** Today the alarm INT plumbing exists but no consumer registers a handler. Question: do we want a default registration (e.g., wired into `boot_power` so it can wake from deep sleep), or leave this for whoever brings up deep-sleep mode? Recommend: leave it -- deep-sleep is out of Phase 0.2 scope.
3. **Round-panel mask.** Does `lvgl_ui.c` apply a circular mask to tile containers automatically, or does each tile own its own clipping? If the latter, this tile's last row may sit outside the visible circle on the CO5300. Today's resize assumes the former.
4. **CO5300 + RTC integration.** The legacy `main.c` still calls `boot_display_init()` + `boot_cst816d_configure()` which refer to the ST7789 path. The CO5300 driver exists at `components/co5300/` but is not yet integrated into the boot flow. The tile resize is therefore correct for the eventual integration, but until `main.c` switches over, the tile will be rendered on an ST7789-shaped canvas (240x240) and the new positions will clip. Status: documented mismatch; not blocking this batch.
5. **task_alarm collision.** `task_alarm_fn` in `components/alarm/` is the in-session software alarm. The hardware-alarm ISR installed here notifies a *different* task by design (the consumer registers itself). Worth deciding: should the software alarm task also be the consumer of the hardware INT (for "wake the system from light sleep on alarm")? If so, we add a `pcf85063_install_alarm_isr(xTaskGetHandle("task_alarm"))` call in `alarm_init()`. **Recommend: yes, but in a follow-up batch.**
