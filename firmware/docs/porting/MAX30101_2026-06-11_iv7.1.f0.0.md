## Summary

The MAX30101 replaces the v5 MAX30102 at the HR/PPG slot. They are register-compatible at the I2C level (same handshake, same FIFO layout, same SpO2 mode, same Part-ID 0x15), so all the hard-won Phase 14/15 bug fixes in the legacy driver -- empty-FIFO watchdog, FIFO settle delay, shutdown readback, the P0 read-before-write bug -- carry forward verbatim. The only chip-level deltas are:

1. **LED3 (green) channel** at REG 0x0E plus MULTI_LED_CTRL1/2 slot config registers (0x11 / 0x12). Phase 3 stays in two-LED SpO2 mode; the green channel is explicitly set to PA=0 (off) so we draw the same power as the MAX30102. `max30101_setup_multi_led_mode()` is exposed in the header for Phase 4+ PPG work.
2. **INT pin newly routed to GPIO7.** v5 had no MAX_INT line; Mk I gives us an RTC-capable wake source. Phase 3 installs a falling-edge ISR that counts edges and notifies the HR task. The wake-on-FIFO path for background HR during sleep is plumbed (task notifications instead of polling) but the actual sleep-mode integration is Phase 2+ work.

The `broker_hr_data_t` shape carries forward field-for-field so `health_tile.c` builds with only an include + identity-call rename.

Today's deliverable: replaced driver (register-compatible swap plus LED3 plumbing plus INT ISR), `health_tile` identity rename, standalone test harness with INT-driven FIFO read, dated `.md`, old `max30102.{c,h}` deleted.

## Code audit

### What dies

`components/max30101/max30102.{c,h}` -- deleted. The MAX30102 driver was a v5 carry-forward. All of it is replaced; the register-compatible parts are duplicated verbatim into the new driver (renamed only) and the chip-specific bits (Part-ID expectation, LED3, INT routing) are new.

### What carries forward

- **`broker_hr_data_t` shape, field-for-field.** Defined now in `max30101.h`. `health_tile.c` builds with only the include + identity-call rename (3 sites: chip name + chip desc in tile init + one comment in the header block).
- **FIFO drain logic.** The whole `task_hr_fn` body -- wake transition with shutdown readback + 10 ms settle, sleep transition with raw-struct dump, empty-FIFO watchdog (50 ticks = 2 s before re-init), first-sample gate, P0 read-before-write fix -- carries forward verbatim. Only the chip prefix is renamed.
- **Beat detector.** `max30101_check_for_beat()` is character-for-character identical to the MAX30102 implementation. Pure CPU, no I2C, no chip-specific state.
- **Signal-quality math.** `(|AC| * 100) / DC_estimate` clamped to 100. Chip-independent.

### What's new

- **LED3 (green) plumbing.** New REG 0x0E define. `max30101_setup_hr_mode()` explicitly writes 0x00 to it (off) for parity with the MAX30102 power profile.
- **MULTI_LED mode entry point.** `max30101_setup_multi_led_mode()` -- programs all three LEDs + the slot-assignment registers. Not used in Phase 3 but exposed for Phase 4+ green-channel PPG work.
- **9-byte FIFO read.** `max30101_read_fifo(... bool multi_led)` -- reads 6 bytes (Red+IR) in SpO2 mode or 9 bytes (Red+IR+Green) in MULTI_LED mode. The Green channel is 0 in SpO2 mode.
- **INT ISR on GPIO7.** Falling-edge, IRAM, increments `s_int_count` + `vTaskNotifyGiveFromISR()` to `s_notify_task`. `max30101_install_int_isr(NULL)` cleanly uninstalls. `gpio_install_isr_service()` is idempotent across siblings (same convention as MAX-M10S 1PPS install).
- **Part-ID warning vs hard-fail.** The legacy driver hard-failed on unexpected Part-ID. We warn and continue, because the same family ID 0x15 is used by both MAX30102 and MAX30101; if a future part bumps to 0x16 we don't want to brick on a value that's still register-compatible.

## Implementation

### `components/max30101/max30101.h`

- Wire constants: `MAX30101_ADDR = 0x57`, `MAX30101_I2C_PORT = I2C_NUM_0`, `MAX30101_INT_GPIO = 7`.
- Register-address `#define`s for the full map. LED3 + MULTI_LED_CTRL1/2 are new vs the MAX30102 set; everything else preserved.
- Mode / SMP_AVE / SR / PW / ADC constants kept identical to the MAX30102 driver (register-compatible).
- `INTR_A_FULL`, `INTR_DATA_RDY`, `INTR_ALC_OVF` bit defines added for the INT-driven path.
- `max30101_sample_t` -- gains a `green` field; otherwise unchanged from the MAX30102 layout.
- `max30101_beat_detector_t` -- unchanged shape (no static-local re-entrancy bug to fix this time; that was the v15 fix).
- `broker_hr_data_t` -- field-for-field preserved.
- Public API: lifecycle (`init/set_shutdown/setup_hr_mode/setup_multi_led_mode/clear_fifo`), FIFO access (`read_fifo/get_fifo_available`), ISR (`install_int_isr/get_int_count`), beat detector, `task_hr_fn`.

### `components/max30101/max30101.c`

- I2C `read_reg / write_reg` helpers via the cmd-link API (same as MAX30102 -- proven on the hardware).
- `max30101_init()` -- Part-ID probe (warn on mismatch), soft reset, sleep.
- `max30101_setup_hr_mode()` -- SpO2 mode, SMP_AVE=4 (25 Hz effective), SR=100 Hz, PW=411 us (18-bit), ADC=4096 nA. Red/IR LED PA set by argument; Green PA forced to 0. FIFO cleared.
- `max30101_setup_multi_led_mode()` -- MULTI_LED mode, all three LEDs PA set by argument, MULTI_LED_CTRL1 = `(2 << 4) | 1` (slot2=IR, slot1=Red), MULTI_LED_CTRL2 = `(0 << 4) | 3` (slot4=none, slot3=Green). The slot numbering and LED# encoding are tagged `[DSV]`.
- `max30101_read_fifo()` -- 6 or 9 byte burst from REG 0x07. 18-bit masks preserved.
- `max30101_int_isr` -- IRAM, count + notify. `pps`-style idempotent ISR service install.
- `task_hr_fn` -- carries forward the entire MAX30102 task body with chip-prefix rename. All Phase 14/15 fixes preserved.

### `components/max30101/health_tile.c`

- `#include "max30102.h"` -> `#include "max30101.h"`.
- `max30102_get_chip_name/desc` -> `max30101_get_chip_name/desc`.
- Header docstring updated to mention the third LED and the new "HR / SpO2 / PPG" desc.

### `components/max30101/CMakeLists.txt`

- `SRCS max30102.c health_tile.c` -> `SRCS max30101.c health_tile.c`. REQUIRES unchanged.

### `test/test_max30101.c`

Six-phase standalone harness:

1. **Phase 1** -- I2C bus 1 init.
2. **Phase 2** -- `max30101_init` (Part-ID + soft reset + sleep).
3. **Phase 3** -- Wake + `setup_hr_mode` (SpO2, Red+IR @ 0x1F PA), 100 ms settle.
4. **Phase 4** -- `install_int_isr(self)` -- ISR notifications come to the test task.
5. **Phase 5** -- 4 s observation. `ulTaskNotifyTake(40 ms)` waits for either an INT notification or the polling fallback. Each notify reads up to 4 FIFO samples and feeds the beat detector. Logs the first 8 raw samples + total reads + total beats + INT count.
6. **Phase 6** -- Sleep, ISR uninstall, stack + heap report.

### Current draw estimate

| State | Current | Notes |
|---|---|---|
| Shutdown (REG_MODE_CONFIG bit 7) | < 1 uA | -- |
| SpO2 mode @ 100 Hz, PA=0x1F (~12 mA) | ~25 mA average | dominated by Red + IR LEDs at moderate drive |
| MULTI_LED mode @ 100 Hz with Green PA=0x1F | ~37 mA average | three LEDs at moderate drive |

### Notes

- **The MAX30101 supports up to 50 mA peak LED drive.** PA=0x1F maps to ~12 mA, which is the legacy MAX30102 driver setting and a sensible starting point for wrist PPG. Drive too low and the signal is noisy; drive too high and you saturate the photodiode and burn power.
- **INT-driven vs polled.** Phase 3 keeps the 40 ms polling loop and uses the INT only as a wake hint (notification with 40 ms timeout). When sleep modes land, polled mode can be turned off and the INT becomes the sole driver.
- **Empty-FIFO watchdog stays.** The MAX30102 had a behaviour where the chip silently lost its mode config after a marginal I2C transaction; the watchdog re-runs the full wake sequence after 2 s of empty FIFO. We keep it because we have no evidence the MAX30101 doesn't share the quirk.

## Defects discovered

- `[DEFECT-001] Register addresses + bit positions tagged [DSV] | components/max30101/max30101.h | HIGH | The MULTI_LED_CTRL1/2 layout (slot encoding 1=Red 2=IR 3=Green 0=disabled), LED3 register position (0x0E), and Part-ID expectation (0x15) come from Maxim 19-7411 conventions and the legacy driver. They have NOT been personally verified against the MAX30101 datasheet in this session. Disposition: BLOCK SpO2/PPG bench bring-up until each is confirmed. If Part-ID expectation differs, init logs a warning but continues -- the rest of the registers should still work.`
- `[DEFECT-002] LED_VCC rail assumption | components/max30101/max30101.c | MED | We assume LED_VCC is shared with 3V3 (per v7.2 §POWER). If the schematic ends up with a separate LED rail that needs explicit power-up before LED PA writes do anything visible, the driver doesn't know. Disposition: confirm on bench. The fix would be a power-up sequence in init(); not in scope today.`
- `[DEFECT-003] INT pin polarity assumed open-drain active-LOW | components/max30101/max30101.c:max30101_install_int_isr | MED | The MAX30101 INT pin is open-drain on most package variants but the chip has no software polarity inversion. We pull up with the ESP internal pull-up + falling-edge ISR. If a previous firmware (or strap) put the chip in an inverted state, we miss every interrupt. Disposition: scope GPIO7 during a known FIFO-A-FULL trigger to confirm polarity.`
- `[DEFECT-004] MULTI_LED slot encoding [DSV] | components/max30101/max30101.c:max30101_setup_multi_led_mode | MED | The slot assignment writes (2 << 4)|1 to MULTI_LED_CTRL1 and (0 << 4)|3 to MULTI_LED_CTRL2. The encoding (1=Red 2=IR 3=Green 0=none) is the documented Maxim convention but has not been verified against this datasheet rev. Phase 3 doesn't actually call this function, so the worst case is a dead Phase 4 work item -- not a Phase 3 blocker.`
- `[DEFECT-005] Empty-FIFO watchdog carried over | components/max30101/max30101.c:task_hr_fn | LOW | The MAX30102 had a quirk where the chip silently lost its mode config after marginal I2C. We keep the 50-tick watchdog because we have no evidence the MAX30101 doesn't share it. If bench shows the chip is rock-solid, this code is dead weight; if bench shows it still happens, it's load-bearing. Disposition: keep until bench data tells us either way.`
- `[DEFECT-006] g_i2c_mutex via extern | components/max30101/max30101.c | LOW (pre-existing) | Same as elsewhere.`

## Open questions

1. **MAX30101 datasheet review.** A single datasheet read collapses DEFECTs 001, 003, 004.
2. **SpO2 algorithm.** Today we only compute HR (a peak detector on the IR channel). True SpO2 requires the Red/IR ratio computed across a windowed sample of both channels. Question: do we want SpO2 in v1, or stub-only? Recommend: stub-only; revisit when fusion-layer work starts. Maxim publishes an example SpO2 implementation under a permissive license.
3. **Green-channel PPG.** The MAX30101's third LED unlocks "watch-style" green-LED PPG (much better motion rejection than IR). Question: when does the Phase 4+ green-channel work land? Until then the green LED is idle and consumes no power.
4. **INT-driven vs polled.** Phase 3 keeps the polling loop with INT as a notification hint. When the watch goes into deep sleep, we want INT-driven background HR (chip stays in SpO2 mode at low ODR, FIFO threshold ISR wakes the ESP just to drain the buffer). Question: confirm the wake-from-GPIO7 path through the ESP-IDF sleep machinery is functional on this hardware. Recommend: validate before depending on it for power-save.
5. **LED current calibration.** PA=0x1F is the legacy setting. Question: should the watch auto-tune LED current based on signal quality (lower current in low-ambient, higher when AC/DC ratio drops)? Recommend: not in v1.
6. **TMP117 + MAX30101 share I2C bus 1.** Both are read by the health tile. No address collision (0x48 vs 0x57); just bus-time competition. Documented in the next module's .md.
