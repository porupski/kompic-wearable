## Summary

The CST9217 is the round-AMOLED-glass capacitive touch controller for the Mk I. It sits on I2C bus 1 alongside the PCF85063 RTC and most of the sensor stack, raises a pin on every touch event, and exposes its data through a fixed-base 16-bit register (`0xD000`) whose first byte is a `0xAB` ACK marker. The driver's job is to translate that into a stream of `cst9217_point_t` records on a lock-free, depth-1 queue that the LVGL indev callback drains. Latency, not throughput, is the load-bearing performance number -- a user has zero tolerance for a swipe that lags.

The old design used the Espressif-managed `esp_lcd_touch_cst816s` component (CST816D family). The CST9217 is from the same broad register-protocol family but uses a **16-bit register address** instead of the CST816's 8-bit address, lives at I2C 0x5A instead of 0x15, and has a different (and not fully documented in this brief) report layout. The ISR/queue pattern and the lock-free fast path are reusable -- the wire-level bytes are not.

This module is implemented as a custom component in `components/cst9217/` rather than going back to the Espressif managed-component path. Reasons: (a) the v7.2 brief explicitly specifies the wire-level facts (16-bit reg, ACK 0xAB, base 0xD000) we own, and (b) there is no published `esp_lcd_touch_cst9217` package, so we'd be writing it anyway.

Today's deliverable is **driver code + ISR + queue + test harness + profiling hooks**, not a fully-debugged bring-up. The first real measurement happens when the FPC lands on the bench.

## Hardware spec from v7.2

### Pinout (v7.2 GPIO Quick Pinout, line 81 onwards; line 152 callout)

| Signal      | GPIO | Direction | Pull / boot | Notes |
|---|:--:|---|---|---|
| TP_SDA      | 1    | OD bidi   | external 5.1 kΩ -> 3V3 | I2C bus 1 SDA. Shared. |
| TP_SCL      | 2    | OD bidi   | external 5.1 kΩ -> 3V3 | I2C bus 1 SCL. Shared. |
| TP_INT      | 6    | input     | internal pull-up; idles HIGH | Falling-edge on touch event. **RTC-wake source (tap).** |
| TP_RST      | 44   | output    | none (boot-tolerant, driven LOW->HIGH on init) | Active-low. v7.2 warns against I2C on 43/44, but RST is fine. |

**Critical reminders from v7.2:**
- **GPIO43/44 boot-log:** never an I2C line. RST is driven and tolerates the boot-log activity, so it stays here (v7.2 line 138).
- **GPIO6 is RTC-wake:** keep ISR pattern clean -- no level-triggered interrupts; falling-edge only, no holdoff that could swallow a wake event.

### I2C / addresses (v7.2 §I2C Bus Assignment, lines 247-266)

| Item | Value |
|---|---|
| Bus | I2C bus 1 (I2C_NUM_0 in firmware terms) |
| Speed | 400 kHz |
| Pull-ups | 5.1 kΩ -> 3V3 (external) |
| Address | 0x5A (fixed, 7-bit) |
| Collision check | 0x5A also exists on bus 2 (DRV2605) -- **different bus, no collision** (v7.2 line 266). |
| Register address width | 16-bit, MSB first |
| Register base | 0xD000 |
| Probe ACK byte | 0xAB at `0xD000` |

### Power (v7.2 §POWER ARCHITECTURE, line 232)

Single 3V3 rail (TPS62840 buck), no separate analog/digital split. No hardware enable on the touch IC -- it powers up with the rest of the device. Standby control, if any, has to be a software command (not exercised in this skeleton; see Open question Q4).

## Code audit

The old project used Espressif's managed component `espressif/esp_lcd_touch_cst816s @ 1.0.3` (referenced in `main/idf_component.yml`) plus a thin `boot_cst816d_configure()` helper called from `main.c`. That component (a) does its own I2C transactions through the ESP-IDF managed API, (b) plugs into LVGL via the `esp_lcd_touch` interface, and (c) assumes an 8-bit register address.

**What carries over:**
- The "ISR -> notify -> task -> burst read -> lock-free queue" pattern. This is the structural skeleton of any usable touch driver and survives intact.
- The "INT pin is the only hot path; polling is dead weight" stance.
- The integration shape (touch out -> LVGL indev). The CST816 managed component drives that via a callback registered with `esp_lcd_touch_register_interrupt_callback()` and a `esp_lcd_touch_read_data()` call. Our replacement will expose `g_touch_q` and let `lvgl_ui` register the indev callback that drains it. The CST816 managed component itself is no longer used.

**What dies:**
- The 8-bit register protocol. CST9217 uses 16-bit register addresses (MSB first). This is the single biggest wire-level change.
- The CST816 device address (0x15). CST9217 is at 0x5A.
- The CST816-specific gesture byte values. The gesture enum in `cst9217.h` mirrors the CST816 family naming but the exact byte codes are **not yet datasheet-confirmed for the CST9217** -- see `[DEFECT-001]`.

**What's new:**
- Explicit 0xAB ACK byte at the report base. This is now the WHO_AM_I-equivalent and is checked at every probe.
- Explicit RST GPIO (the CST816 path didn't drive a reset pulse, just trusted the chip to come up cold).

## Implementation

Three files were created in `components/cst9217/`:

### `cst9217.h`

- Wire-level constants pinned in code: `CST9217_I2C_ADDR = 0x5A`, `CST9217_REG_BASE = 0xD000`, `CST9217_ACK_VALUE = 0xAB`, `CST9217_INT_GPIO = 6`, `CST9217_RST_GPIO = 44`.
- `cst9217_gesture_t` enum mirroring CST816 naming (NONE / SWIPE_*  / SINGLE_TAP / DOUBLE_TAP / LONG_PRESS). Values are placeholders -- see `[DEFECT-001]`.
- `cst9217_point_t` -- x, y, fingers, gesture, t_us (esp_timer_get_time at task wakeup).
- Public queue handle: `extern QueueHandle_t g_touch_q;` (depth 1, `xQueueOverwrite` semantics).
- Function signatures: `cst9217_init`, `cst9217_deinit`, `task_touch_fn` (Core 0 task), `cst9217_probe_ack`, `cst9217_read_report` (caller-holds-mutex helpers).
- No data_broker reference. Touch latency is too tight to round-trip the broker mutex; touch stays on its own lock-free queue. (Blueprint 4 §6.)

### `cst9217.c`

- `cst9217_isr()` -- IRAM, falling-edge handler. Single job: `vTaskNotifyGiveFromISR()` and yield. No I2C, no logging.
- `i2c_read_16bit_reg()` -- 2-byte register address (MSB first) followed by a repeated-START, then `len` data bytes. Burst-mode where possible. Caller holds `g_i2c_mutex`.
- `cst9217_hw_reset()` -- LOW for `CST9217_RST_LOW_MS` (5 ms), HIGH for `CST9217_RST_HIGH_MS` (50 ms). Pulse widths are first-pass conservative; datasheet confirmation pending (`[DEFECT-003]`).
- `cst9217_int_install()` -- pull-up + falling-edge config, `gpio_install_isr_service()` (idempotent -- returns INVALID_STATE if another driver beat us to it, which is benign), `gpio_isr_handler_add()`.
- `cst9217_init()` -- creates queue (depth 1), runs reset, probes ACK under `g_i2c_mutex`, installs INT. Logs total boot time via `esp_timer_get_time()`.
- `task_touch_fn()` -- pinned to Core 0 (Blueprint 1 §1). Waits on `ulTaskNotifyTake(pdTRUE, portMAX_DELAY)`. On notification: takes `g_i2c_mutex` (50 ms ceiling), burst-reads 8 bytes at `0xD000`, gives mutex, parses, `xQueueOverwrite(g_touch_q, &p)`.

### `CMakeLists.txt`

Standard `idf_component_register` with `REQUIRES driver freertos esp_timer`. Mirrors `components/pcf85063/CMakeLists.txt` style.

### `test/test_cst9217.c`

Standalone `app_main()` that:
1. Logs chip identity strings.
2. Creates `g_i2c_mutex` if the host bootstrap hasn't.
3. Initialises I2C bus 1 at 400 kHz on GPIO1/2 -- logs wall time.
4. **Phase 1, sync probe:** takes mutex, reads 1 byte at 0xD000, expects 0xAB, logs wall time + result.
5. **Phase 1, sync report dump:** takes mutex, burst-reads 8 bytes, prints them in hex. Useful for "what does the chip actually say?" on cold boot.
6. **Phase 2, ISR loop:** calls `cst9217_init()`, spawns `task_touch_fn` on Core 0, then drains `g_touch_q` for 30 seconds. Each event prints x/y/fingers/gesture and a task-to-queue dt (a coarse proxy for end-to-end latency).
7. At exit: logs heap delta and the running task's stack high-water.

Test is intentionally a flat sequence -- no menus, no parametrization -- so first bring-up is dead simple to read.

## Profiling

### Boot-time cost

| Phase | Component | Time | Method |
|---|---|---|---|
| Init | I2C bus 1 setup | **TBD (no hw)** | `esp_timer_get_time()` around `i2c_param_config` + `i2c_driver_install` in `test_cst9217.c` |
| Init | RST pulse | **5 + 50 = 55 ms (planned)** | wall time around `cst9217_hw_reset()` |
| Init | Probe + INT install | **TBD (no hw)** | wall time around `cst9217_init` minus RST |
| Op   | First report read after INT | **TBD (no hw)** | task wakeup -> queue post, logged per event |

### Per-operation cost

| Operation | Time | Bus | Notes |
|---|---|---|---|
| ACK probe (write 2-byte reg + read 1 byte) | **TBD** | 400 kHz | wire-time floor: ~80 us (3 bytes write + 2 bytes read with addr re-START) |
| Full report read (write 2-byte reg + read 8 bytes) | **TBD** | 400 kHz | wire-time floor: ~280 us, plus driver overhead |
| ISR -> task wakeup latency | **TBD** | n/a | dominated by FreeRTOS notify path; expect 5-30 us on Core 0 |
| ISR -> queue post end-to-end | **TBD** | 400 kHz | upper bound = task wake + mutex acquire (worst case 50 ms ceiling) + I2C report + queue post; expect < 1 ms typical |

### Memory

| Metric | Value | Notes |
|---|---|---|
| Driver state | ~0 bytes | only `s_task_hdl` + `s_isr_installed` globals (~8 bytes) |
| Queue allocation | 1 × `sizeof(cst9217_point_t)` = ~12 bytes | created in `cst9217_init`, freed in `cst9217_deinit` |
| Stack high-water (touch task) | **TBD (no hw)** | task is sized 3072 in test harness (no float, no large locals) |
| Heap delta (init+deinit) | **0 expected** | logged at end of test harness |

### Current draw estimate

| State | Current (mA) | Notes |
|---|---|---|
| Idle (no touch) | TBD | dominated by IC standby + I2C pull-up bias |
| Active (single touch event burst) | TBD | I2C transaction is brief; chip-side draw is the long pole |
| Sleep (if supported via software command) | TBD | not exercised in this skeleton; see Open question Q4 |

### Notes

- **Latency is the load-bearing perf number, not throughput.** A 100 ms tap-to-feedback is the difference between "responsive" and "broken". The end-to-end budget is roughly: chip -> INT (us) + ISR (us) + task notify (us) + mutex acquire (typically << 1 ms, ceiling 50 ms) + I2C burst (~280 us at 400 kHz) + queue post (us). Anything that pushes the typical case above ~1 ms is worth investigating.
- The 50 ms `g_i2c_mutex` ceiling is the worst-case starvation under load from the env / IMU drivers. If we ever see tactile lag during a BME688 read, **this is the first suspect**, not the chip. Disposition: raise to defect if observed.
- ISR is fall-edge only and `IRAM_ATTR`. Do not log from it. Do not call any non-ISR-safe FreeRTOS APIs from it. The only legal action is `vTaskNotifyGiveFromISR` + yield.

## Defects discovered

- `[DEFECT-001] Gesture byte codes unverified | components/cst9217/cst9217.h:cst9217_gesture_t | MED | The gesture enum values (0x01-0x0C) are inherited from the CST816 family and have NOT been confirmed against a CST9217 datasheet. The driver code does not act on the gesture byte today; it just propagates it. Disposition: do not consume the gesture byte in UI logic until codes are verified on bench. If we end up only using x/y, the enum can be dead code.`
- `[DEFECT-002] INT polarity assumption | components/cst9217/cst9217.c:cst9217_int_install | MED | The driver configures GPIO6 as pull-up + falling-edge, matching the CST816 family idle-HIGH / pulse-LOW convention and the v7.2 RTC-wake-tap callout. The CST9217 datasheet has not been audited to confirm polarity. Disposition: verify on first bench bring-up by scoping GPIO6 during a touch event.`
- `[DEFECT-003] RST pulse widths conservative, not datasheet-confirmed | components/cst9217/cst9217.c:cst9217_hw_reset | LOW | 5 ms low / 50 ms high are first-pass values from generic touch-IC practice. Disposition: confirm against CST9217 datasheet, tighten if longer-than-needed.`
- `[DEFECT-004] Report layout assumed CST816-style | components/cst9217/cst9217.c:task_touch_fn (parse block) | HIGH | The 8-byte burst at 0xD000 is decoded as [ACK | fingers | gesture | reserved | x_hi | x_lo | y_hi | y_lo]. This layout is the CST816 family convention, applied here on the assumption that the CST9217 inherits it. The brief specifies 0xD000 as the report base and 0xAB as the first byte but does NOT specify the layout of bytes 1..7. Disposition: dump raw bytes on first bench bring-up (test_cst9217 Phase 1 step 5 does this); reconcile against actual chip output; correct parser if needed.`
- `[DEFECT-005] g_i2c_mutex resolved via extern, not header | components/cst9217/cst9217.c | LOW | The driver extern-declares g_i2c_mutex rather than #including boot_hw_init.h, because boot_hw_init.h does not yet exist in this tree. Same pattern as bme280_drv.c and max30102.c. Disposition: switch to #include "boot_hw_init.h" when that file lands.`

## Open questions

1. **CST9217 datasheet:** is a copy available locally, or do we go off whatever the CST816 family doc says? `[DEFECT-001]`, `[DEFECT-002]`, `[DEFECT-003]`, `[DEFECT-004]` all collapse if we have the actual datasheet.
2. **Multi-touch:** the brief and the v7.2 doc both speak in single-point language. CST9217 family chips can usually do 2-finger. Do we want the driver to expose a second point now, or wait until the UI calls for it? Today's skeleton is single-point only.
3. **Gesture consumption:** should gestures (swipe, double-tap, long-press) be consumed in the firmware (e.g., for screen wake / nav events), or do we want LVGL to derive them from raw x/y/down/up events? Recommend the latter for v1 -- it keeps the driver dumb and the gesture surface uniform.
4. **Low-power mode:** the CST816 family supports a software-issued "deep sleep" register write. Do we want one for v1? If yes, on what wake -- screen-off, IMU-stillness, both? Today's skeleton has no sleep path.
5. **GPIO6 wake responsibility:** v7.2 names tap on GPIO6 as an RTC-wake source. Does the wake-from-sleep ISR re-use this same handler, or do we need a separate deep-sleep wake path? Out of scope for today; tag for the eventual boot_power refactor.
6. **`boot_hw_init.h` integration:** the file is referenced by drivers across the tree but does not exist yet. When it lands, this driver should switch from `extern SemaphoreHandle_t g_i2c_mutex;` to `#include "boot_hw_init.h"`. Same for adding `task_touch_fn` to the `boot_tasks.c` task table.
7. **LVGL indev hookup:** the queue is in place. The actual `lv_indev_drv_register()` call belongs in `lvgl_ui.c` (or a new `touch_indev.c`) and is out of scope for this module. We should sketch it as a deliverable for the lvgl_ui refresh.
