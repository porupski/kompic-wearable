## Summary

Vibration alarm module — up to 3 user-configurable slots, persisted in the
`alarm_cfg` NVS namespace. Core 0 task polls RTC every second, fires a haptic
burst pattern with intensity escalation on time-match, supports snooze and
auto-dismiss, and routes UI overlays via `g_ui_event_q`.

Phase 6 v7.2 changes:
- **`buzzer` field repurposed to `led_strobe`.** v5 reserved GPIO42 as a piezo
  buzzer; v7.2 uses GPIO42 as the WS2812 status LED. The persisted bit is kept
  (NVS key `"buzz"` retained for backwards-compatible loads) but the semantic
  changes — when set, the alarm task drives `ws2812_set_state(WS2812_STATE_ALERT)`
  for the firing window and restores the pre-fire LED state on STOP /
  auto-dismiss.

No chip-header swaps required: alarm reads RTC + battery via the broker and
calls into DRV2605 via the `haptic.h` API, both of which are chip-driver
internals that stayed put.

## Pinout / bus from v7.2

- **RTC (PCF85063)** on I2C bus 1 — read via broker, no direct contact.
- **DRV2605 (haptic)** on I2C bus 2 — driven via `haptic.h`.
- **WS2812 (status LED)** on GPIO42 (RMT TX) — driven via `ws2812.h` for
  ALERT strobe when a slot's `led_strobe` flag is set.

## Code audit (v5 → v7.2)

- `alarm.h`: `bool buzzer` → `bool led_strobe` with a doc comment explaining the
  v5-to-v7.2 rename rationale.
- `alarm.c`:
  - `#include "ws2812.h"` added.
  - `task_alarm_fn` gains two locals: `bool firing_led_strobe` (tracks whether
    LED was switched for the active fire) and `ws2812_state_t led_state_pre_fire`
    (the state captured before the strobe so we can restore on dismiss).
  - On both fire-entry points (snooze re-fire path and normal time-match scan),
    the per-slot `led_strobe` flag gates a `ws2812_get_state()` capture and a
    `ws2812_set_state(WS2812_STATE_ALERT)` transition.
  - On the three fire-exit paths (external STOP, AUTO_DISMISS, snooze
    activation — snooze re-fire keeps the LED on, but the STOP branch in the
    UI flow handles dismissal), the captured state is restored via
    `ws2812_set_state(led_state_pre_fire)`.
  - NVS key `"buzz"` is preserved so old v5 saves load cleanly; new saves
    write the same key with the new semantic. Log line updated:
    `buzzer=` → `strobe=`.
- `alarm_tile.c`:
  - `.buzzer = false` literal initialisers in slot constructors → `.led_strobe = false`.
  - File-header doc updated: the LED-strobe field is persisted but currently
    not exposed in the create/edit overlay; defaults to off. UI toggle is a
    Phase 6.6 / v2 follow-up.
- `CMakeLists.txt`: `ws2812` added to REQUIRES (between `pcf85063` and
  `app_logic` for readability).

## Implementation

The firing flow now looks like:

```
TIME MATCH
  ad.slots[i].led_strobe == true →
      led_state_pre_fire = ws2812_get_state();
      ws2812_set_state(WS2812_STATE_ALERT);
  task_alarm_fn keeps firing burst+pause+escalation as before.

EXIT (STOP from UI, AUTO_DISMISS at 10 min, or snooze activation)
  if (firing_led_strobe)
      ws2812_set_state(led_state_pre_fire);
```

ALERT is the existing WS2812 state machine entry for "red square-wave blink
2 Hz" — visible across a darkened room, audible on the LRA, restorable to
whatever the LED was doing before (OFF/IDLE/CHARGING).

## Test harness

`firmware/test/test_alarm.c`:

1. Boot broker + cross_driver + ws2812 + (faked) DRV2605 + (faked) RTC.
2. Spawn `task_alarm_fn`.
3. **Test A — slot fires with strobe off:** write slot[0] with `armed=true`,
   `led_strobe=false`, time = (next-minute). Advance the broker RTC.
   Assert `ad.firing == true`, assert WS2812 state did NOT change.
4. **Test B — slot fires with strobe on:** repeat, `led_strobe=true`. Assert
   firing == true and `ws2812_get_state() == WS2812_STATE_ALERT`.
5. **Test C — STOP restores prior state:** while firing, set
   `broker_alarm.firing = false` (simulating UI STOP). Within 2 ticks, assert
   the WS2812 state is restored to pre-fire value.
6. **Test D — AUTO_DISMISS restores prior state:** repeat fire, then advance
   `now_sec` by `ALARM_AUTO_DISMISS_SEC + 1`. Assert `ad.firing == false` and
   LED restored.
7. **Test E — NVS round-trip:** save a slot with `led_strobe=true`, simulate
   reboot (`alarm_nvs_load_all()`), assert the bit comes back.
8. **Test F — v5 compatibility load:** seed NVS by writing key `"slot0_buzz"`
   with value `1`, run `alarm_nvs_load_all()`, assert `led_strobe == true`
   (legacy persisted bit carries over without migration).

### Memory
| Metric | Value | Notes |
|--------|-------|-------|
| Task stack | 4096 bytes allocated | high-water TBD; pattern table is static |
| `broker_alarm_data_t` | sum: 3 × (3 u8 + 2 bool) + 4 + 1 + 2 + 4 + 1 ≈ 30 bytes | mostly slots |
| Pattern table (k_patterns) | static, ~240 bytes (3 patterns × 5 steps × 4 bytes + meta) | const |

### Current draw
| State | Current (mA) | Notes |
|-------|--------------|-------|
| Idle (no firing) | low | one RTC I2C transaction/sec |
| Firing — haptic LRA | medium | DRV2605 LRA pulses, see DRV2605 profiling |
| Firing — LED strobe | small additive | WS2812 red at 2 Hz, ≤ 12 mA peak |

### Notes
- The LED strobe sits on top of haptic; it does not gate or coordinate the LRA
  rhythm. Visually, you get a red blink at 2 Hz while the LRA hums in its own
  burst/pause cadence — two independent cues. By design.
- Snooze re-fire path (re-entering firing after 10 min) also re-asserts ALERT
  if the slot's `led_strobe` was set. Re-capturing pre-fire state at this
  re-entry means a USB-plug-in during snooze (which would set CHARGING) won't
  be clobbered.

## Defects discovered

[DEFECT-001] LED strobe toggle not yet exposed in the slot-edit UI | components/alarm/alarm_tile.c | LOW | New slots default to `led_strobe = false`; legacy v5 slots carry their `buzz` bit forward, but no LVGL control lets the user flip the bit interactively. Disposition: add to the overlay in a follow-up alongside the Phase 6.6 lvgl_ui pass — three rollers are already busy, a small checkbox or icon-button row below is the natural slot.

[DEFECT-002] Pattern effect IDs assume v5 DRV2605 effect library | components/alarm/alarm.c:46-67 | LOW | `k_strong_buzz_steps`, `k_alert_pulse_steps`, `k_gentle_ramp_steps` reference DRV2605 internal effect IDs (1, 10, 14, 47). Same chip family, so IDs should resolve, but the Phase 20 datasheet pass should cross-check against the DRV2605 datasheet's effect library appendix to confirm none were renumbered between revisions.

[DEFECT-003] `g_tz_offset_hours` extern in alarm.c | components/alarm/alarm.c:33 | LOW | The TZ offset for local-time comparison lives in `ui_broker.c` (Core 1 only, NVS-loaded) but `alarm.c` is Core 0. Reading a Core-1-owned volatile from Core 0 is OK on ESP32 (32-bit reads are atomic) but the ownership comment in data_broker.h could be tightened to call this read out explicitly. Disposition: cosmetic; revisit when ownership rules are next audited.
