## Summary

New output device in Mk I -- no v5 carryforward. One WS2812B pixel (not a string) reports system status via colour + animation. The driver uses the ESP32-S3 RMT peripheral with the IDF v5 bytes-encoder API:

- RMT TX channel on GPIO42 at 10 MHz resolution (100 ns/tick).
- 0-bit symbol: 0.4 us high + 0.8 us low.
- 1-bit symbol: 0.8 us high + 0.4 us low.
- Both well inside the WS2812B +/- 150 ns datasheet tolerance.
- 24 bits per pixel sent as 3 bytes in **GRB** order (WS2812B convention, not RGB).
- > 50 us low between frames latches the pixel.

A 50 ms `esp_timer` periodic callback drives the animations off a small state machine: OFF / IDLE (dim white) / CHARGING (sinusoidal blue pulse, 1 Hz) / CHARGED (steady green) / ALERT (red square wave, 2 Hz). For ad-hoc colours, `ws2812_set_color(r,g,b)` sets a manual-override flag that suppresses the animation tick until the next `ws2812_set_state()` call.

Today's deliverable: new `components/ws2812/` (driver + animation tick + color macros), standalone test harness with 8-color sweep + state machine demo, dated .md.

|---|:--:|---|---|
| LED_Din | 42 | output | RMT TX. Drives the single WS2812B pixel. |

### Pixel

| Item | Value |
|---|---|
| Count | 1 (not a string) |
| Wire order | GRB (24 bit) |
| Driver | RMT TX, 10 MHz resolution |

### Timings (WS2812B datasheet, +/- 150 ns)

| Bit | High | Low |
|---|---|---|
| 0  | 0.4 us | 0.85 us |
| 1  | 0.8 us | 0.45 us |
| Reset (latch) | -- | > 50 us low |

Our 4/8-tick choices at 100 ns/tick land at 0.4/0.8 high and 0.8/0.4 low respectively -- inside the +/- 150 ns spec on both edges.

### Power

3V3 rail. Single pixel typical current is < 20 mA at full white; the driver caps `WS2812_WHITE` at 0x808080 (half-intensity) because full white on a 1-pixel status indicator at 50 mm viewing distance is genuinely uncomfortable.

## Code audit

### What dies

Nothing -- new component.

### What's new

- **RMT bytes-encoder configuration.** One-shot setup via `rmt_new_bytes_encoder` with the 0/1 symbol shapes. The IDF handles per-bit translation; we push 3 raw bytes per frame.
- **Animation state machine.** 5 states, sinusoidal CHARGING and square-wave ALERT animations driven by elapsed-time math against `esp_timer_get_time() - s_state_start_us`. No FreeRTOS task -- esp_timer is lighter for 50 ms cadence.
- **Manual override.** `ws2812_set_color()` writes immediately and sets `s_manual_override = true`; the animation tick early-exits while the override is in effect. `ws2812_set_state()` clears the override.
- **`WS2812_RGB(r,g,b)` macro** + named color macros (RED, GREEN, BLUE, YELLOW, MAGENTA, CYAN, WHITE half-intensity, OFF). Used both in the driver internals and in the test harness.

## Implementation

### `components/ws2812/ws2812.h`

- Pin: `WS2812_GPIO = GPIO_NUM_42`.
- RMT resolution: 10 MHz (`WS2812_RMT_HZ`).
- Pixel count: 1.
- `ws2812_state_t` enum: OFF / IDLE / CHARGING / CHARGED / ALERT.
- Identity: `"WS2812B"` / `"Status LED (1 px)"`.
- Public API: `ws2812_init` / `_deinit` / `_set_color` / `_set_state` / `_get_state`.
- Color macros: `WS2812_RGB(r,g,b)`, 7 named + OFF.

### `components/ws2812/ws2812.c`

- `install_encoder()` -- one-shot RMT bytes encoder setup, MSB-first, the 4/8 tick symbol pairs.
- `ws2812_push(r,g,b)` -- assembles the 3-byte GRB frame and pushes via `rmt_transmit`. Loop-count 0 (single shot). The next call waits for the channel to be idle, which is trivial at our 50 ms cadence.
- `ws2812_anim_cb` -- 50 ms tick. Early-exits on manual override. CHARGING uses `cosf((age_us % 1e6) / 1e6 * 2pi)` mapped to a 0..1 intensity; ALERT uses `(age_us / 250000) & 1` for a 2 Hz square wave.
- `ws2812_set_color` -- pushes immediately, sets override flag.
- `ws2812_set_state` -- updates state + resets `s_state_start_us` so animations start from a known phase, clears override, calls the animation callback once for an immediate update.
- `ws2812_init` -- RMT TX channel + bytes encoder + enable + force-LED-off + animation timer @ 50 ms.
- `ws2812_deinit` -- stops timer, drives LED dark, disables + deletes the RMT channel and encoder.

### `components/ws2812/CMakeLists.txt`

REQUIRES: `driver` (legacy GPIO compat), `esp_driver_rmt` (RMT TX channel + bytes encoder), `esp_timer`, `freertos`.

### `test/test_ws2812.c`

Four-phase standalone harness:

1. **Phase 1** -- `ws2812_init`, wall time.
2. **Phase 2** -- 8-step color sweep at 500 ms each via `ws2812_set_color`.
3. **Phase 3** -- State machine demo: IDLE 2 s -> CHARGING 4 s (visible sinusoidal pulse) -> CHARGED 2 s -> ALERT 4 s (visible square wave) -> OFF 1 s.
4. **Phase 4** -- `ws2812_deinit`, stack high-water + heap.

Primary verification signal is visual; for rigour, a logic analyzer on GPIO42 will confirm the 800 kHz nominal bit rate.

## Profiling

### Boot-time cost

| Phase | Component | Time | Method |
|---|---|---|---|
| Init | RMT TX channel + bytes encoder + enable | **TBD (no hw)**, < 1 ms expected | wall time |
| Init | esp_timer create + start | < 100 us | -- |

### Per-operation cost

| Operation | Time | Notes |
|---|---|---|
| `ws2812_push` (24 bit frame) | ~36 us wire time + RMT setup | 24 bits * 1.25 us nominal + > 50 us reset |
| Animation tick (no-op state) | < 5 us | early-exit on OFF / IDLE |
| Animation tick (CHARGING, with cosf) | < 30 us | one cosf on the FPU |

### Memory

| Metric | Value | Notes |
|---|---|---|
| Driver state | ~24 bytes | handles + state + override flag + timestamps |
| RMT TX channel + encoder | ~500 bytes | IDF internal |
| Animation timer | ~80 bytes | esp_timer internal |

### Current draw estimate

| State | Current | Notes |
|---|---|---|
| OFF      | < 1 mA (just the chip's own quiescent) | -- |
| IDLE (dim white 0x404040) | ~6 mA | half intensity all channels |
| CHARGING (sinusoidal blue, avg) | ~5 mA average | peaks ~10 mA at the pulse maximum |
| CHARGED (steady green 0xC0) | ~10 mA | green channel mid-high |
| ALERT (square wave red, 50% duty avg) | ~9 mA average | peaks ~18 mA on |
| Worst case (full white) | ~50 mA | not used; capped to half intensity by WS2812_WHITE macro |

### Notes

- **WS2812B is a current hog at full intensity.** ~20 mA per channel = 60 mA per pixel at full white. The driver intentionally caps WS2812_WHITE at 0x808080 to halve that.
- **Animation cadence is 50 ms.** Animation updates at 20 Hz feel smooth to the eye; the WS2812B latches a new frame much faster than that.
- **RMT idle behaviour matters during sleep.** When the watch sleeps, the RMT channel should be torn down to avoid the wire idling HIGH (which the WS2812B may interpret as bit data). `ws2812_deinit` does the right thing -- but the sleep path needs to call it. Out of scope today.

## Defects discovered

- `[DEFECT-001] WS2812B timings [DSV] | components/ws2812/ws2812.c:install_encoder | LOW | The 4/8 tick choices at 10 MHz target the published WS2812B timings. They have NOT been verified on a logic analyzer in this session. Disposition: scope GPIO42 during a known color update; tolerance is +/- 150 ns. If the chip is actually an SK6812 clone or a knockoff with different timing, this is where the symptom shows up.`
- `[DEFECT-002] No sleep-time cleanup hook | components/ws2812/ws2812.c | MED | When the watch enters deep sleep, the RMT channel stays alive and the GPIO42 line idles. WS2812B latches any state > 50 us low, so the chip enters an undefined state for the duration of sleep. On wake, the first ws2812_push may produce a brief wrong color before locking in. Disposition: add a ws2812_pre_sleep() / _post_wake() pair when the sleep machinery lands; or call ws2812_deinit() / _init() across sleep transitions.`
- `[DEFECT-003] No charging-source-aware state choice | components/ws2812/ws2812.c | LOW | The driver exposes states but does not subscribe to broker_battery_data_t. Some other code (boot_power? a dedicated coordinator?) must call ws2812_set_state() at the right times. Disposition: this is by design -- the driver is a pure output, the policy lives elsewhere. Flagged for the integrator.`
- `[DEFECT-004] Single shared esp_timer thread | components/ws2812/ws2812.c | LOW (pre-existing IDF behaviour) | esp_timer callbacks all run on the same low-priority task. If something else hogs that task, the 50 ms animation tick may jitter visibly. Disposition: keep simple; revisit if animation jitter is observed.`
- `[DEFECT-005] WS2812_WHITE capped at 0x808080 | components/ws2812/ws2812.h | LOW | Documented in summary. If a future use case wants full white (camera fill light?), it would set color directly via ws2812_set_color(0xFF, 0xFF, 0xFF) and the cap is bypassed. The macro just protects the casual user. Disposition: documented.`
- `[DEFECT-006] cosf() in animation tick | components/ws2812/ws2812.c:ws2812_anim_cb | LOW | The S3 has an FPU; cosf is ~20 cycles. At 50 ms cadence the overhead is negligible. Flagged only because it's the kind of thing CR comments tend to obsess over. Disposition: keep.`

## Open questions

1. **Where does the LED policy live?** Some entity has to call `ws2812_set_state(CHARGING)` when the BQ25619 reports charging. boot_power? battery tile? a new led_coordinator component? Recommend: boot_power, since it already owns the battery+power state machine and has the right context.
2. **GPIO42 conflict on the dev board?** GPIO42 is on the OSC line on many ESP32-S3 dev boards. Confirm v7.2 hardware has no conflict (the spec says LED_Din which suggests dedicated).
3. **ALERT colour gate.** Should ALERT respect a "do not disturb" toggle? Today it blinks unconditionally. Recommend: gate when the LED policy layer lands.
4. **Power-budget under animation.** CHARGING animation averages ~5 mA at 1 Hz sinusoidal -- not nothing but tolerable. ALERT square wave averages ~9 mA. Question: long-running ALERT (e.g. notification not dismissed for 10 minutes) drains 1.5 mAh. Worth flagging in user-facing UX so they don't leave alerts on overnight.
5. **Diagnostic mode.** A "rainbow test" or "all colors at full intensity" mode for production verification. Not needed today; could be a tile in Phase 5+ debug menus.
