## Summary

Same chip as v5 -- no chip-layer rewrite. Two surgical changes get the existing driver up on Mk I hardware:

1. **Move from I2C bus 1 to bus 2.** Mk I isolates the haptic sequencer on its own I2C bus (GPIO4 SDA / GPIO5 SCL) so the 100-200 ms sweep bursts in `drv2605_sweep_step()` and the per-effect register-write storm in `drv2605_play_effect()` can't starve the touch / sensor traffic on bus 1. The driver becomes mutex-aware via `g_i2c2_mutex` (extern, expected to land in boot_hw_init.c alongside `g_i2c_mutex`) and uses `I2C_NUM_1` instead of `I2C_NUM_0`.

2. **GPIO0 DRV_EN ownership.** The DRV2605's hardware EN pin is now routed to ESP GPIO0. Per the brief, `boot_power.c` owns the boot-time drive-LOW; this driver does NOT touch GPIO0 in production. The test harness drives it LOW directly so that standalone test runs work before boot_power lands.

Everything else carries forward verbatim from the v5 driver:
- LRA open-loop init sequence (FEEDBACK = 0xBA, CONTROL1-4 tuned for Apple Taptic).
- Auto-calibration (known to fail DIAG on the Taptic, kept for diagnostics).
- Sweep calibration with IMU-assisted peak-amplitude latch.
- Library-effect playback + RTP mode.
- NVS persistence of LRA period + UI effect.
- Phase 11/13 fixes intact.

The brief says "copy as-is". This .md captures the minimum delta to make it work on Mk I.

Today's deliverable: bus-1-to-bus-2 refactor in `haptic.c` (10 mutex sites, 12 I2C_NUM sites), header docstring updated, standalone test harness for bus 2, dated .md.

## Hardware spec from v7.2

### Pinout

| Signal     | GPIO | Notes |
|---|:--:|---|
| DRV_SDA    | 4 | I2C bus 2 SDA |
| DRV_SCL    | 5 | I2C bus 2 SCL |
| DRV_EN     | 0 | Active-LOW. Chip in HW shutdown when HIGH. Internal pull-up on the chip; boot_power.c drives LOW after V_LRA is stable. |

### I2C

| Item | Value |
|---|---|
| Bus | I2C bus 2 (`I2C_NUM_1`) |
| Speed | 400 kHz |
| Address | 0x5A |
| WHO_AM_I | 0xE4 (unfused part; clone-typical) |

### Motor

- ELV1411A Apple Taptic Engine clone on a daughter PCB.
- Resonant frequency: ~157 Hz nominal (period reg 64 at 98.46 us/LSB).
- Calibration: auto-cal fails DIAG; IMU-assisted sweep calibration in `haptic.c::run_sweep()` is the working path. Latched value lives in NVS `haptic_cfg/lra_period`.

### Power

| Rail | Connected | Notes |
|---|---|---|
| V_LRA | DRV2605 VDD via LDO | Brought up by boot_power; DRV_EN should stay HIGH until V_LRA is stable. |

## Code audit

### What dies

Nothing. The driver, the haptic task, the haptic tile, the NVS layout, the broker shape -- all carry forward verbatim from v5.

### What changes (bus-1 -> bus-2)

- **`haptic.c`** -- 22 token-level edits:
  - `g_i2c_mutex` -> `g_i2c2_mutex` (10 sites).
  - `I2C_NUM_0` -> `I2C_NUM_1` (12 sites, including the inline `i2c_master_write_to_device()` in the RTP_AMP path).
- **`drv2605.h`** docstring updated: interface note now says "I2C_NUM_1 (bus 2) -- caller must hold g_i2c2_mutex".
- **`haptic.c`** file-header comment block now flags the Phase 4 bus move.

### What doesn't change

- `drv2605.c` -- the chip-layer driver takes an `i2c_port_t` argument; it does not name the bus. Zero edits.
- `haptic_tile.c` -- Core 1 / LVGL only, no I2C references. Zero edits.
- `haptic_tile.h` -- zero edits.
- `CMakeLists.txt` -- no source changes; REQUIRES list unchanged.
- `broker_haptic_data_t` -- unchanged.
- `boot_tasks.c` task wiring -- unchanged (haptic task entry point is `task_haptic_fn`).

### GPIO0 DRV_EN -- who owns what

| Responsibility | Owner | This driver's stance |
|---|---|---|
| Configure GPIO0 as output, drive LOW at boot | `boot_power.c` (per brief) | Not touched by drv2605.c / haptic.c. |
| Drive HIGH on shutdown (low-power) | `boot_power.c` | Not touched. |
| Drive HIGH during a "force-reset" recovery | TBD | Could be added to `haptic_init()` if a "kick the chip" recovery is wanted -- not in scope today. |
| Test-harness standalone setup | `test_drv2605.c` | The harness configures GPIO0 + drives LOW + waits 5 ms before issuing I2C, so the test runs before boot_power lands. |

## Implementation

### `components/drv2605/haptic.c` (modified)

The diff is mechanical: `s/g_i2c_mutex/g_i2c2_mutex/g` and `s/I2C_NUM_0/I2C_NUM_1/g`. File header comment block gains a Phase 4 note explaining the rationale. The `boot_hw_init.h` extern path is preserved -- once that header lands, it must declare both `g_i2c_mutex` (bus 1) and `g_i2c2_mutex` (bus 2).

### `components/drv2605/drv2605.{c,h}` (header docstring only)

`drv2605.c` is bus-agnostic by design; zero functional edits. `drv2605.h` docstring updated to say bus 2.

### `components/drv2605/haptic_tile.c` (no edits)

Core 1 / LVGL only. The tile calls `haptic_play(haptic_get_ui_effect())` and `haptic_request_calibration()` -- both queue-based, the underlying chip layer is opaque to the tile.

### `components/drv2605/CMakeLists.txt` (no edits)

REQUIRES still references `data_broker`, `lvgl`, `lvgl_ui`, `esp_lvgl_port`, `driver`, `freertos`, `esp_timer`; PRIV_REQUIRES still references `boot_logic`. Bus number isn't part of the build graph.

### `test/test_drv2605.c`

Six-phase standalone harness:

1. **Phase 1** -- I2C bus 2 init on GPIO4/5 at 400 kHz.
2. **Phase 2** -- GPIO0 DRV_EN configured as output and driven LOW; 5 ms settle.
3. **Phase 3** -- `drv2605_init` (chip-layer config: LRA library, FEEDBACK, CONTROL1-4, rated voltage, OD clamp).
4. **Phase 4** -- Play four named effects (CLICK, DOUBLE_CLICK, STRONG_BUZZ, SHARP_TICK) at 500 ms spacing. Logs per-write I2C time so the bus-2 latency can be compared against the bus-1 baseline once bench data exists.
5. **Phase 5** -- 100 ms RTP burst at period reg 64 (~157 Hz) via `drv2605_sweep_step()`. Confirms the sweep path works end-to-end.
6. **Phase 6** -- Stop, DRV_EN back HIGH (hardware shutdown), stack + heap report.

### Notes

- **Bus 2 isolates haptic from sensor traffic.** The legacy bus-1 layout meant a 200 ms sweep burst could push touch latency above 200 ms; bench measurement on Mk I should show that's gone.
- **DRV_EN under boot_power's control is the right move.** It keeps haptic init independent of V_LRA timing -- the chip stays in HW shutdown until the rail is stable.
- **Test harness is intentionally standalone.** Sets up GPIO0 itself so the bench technician can flash `test_drv2605.c` to a bare ESP32 + the daughter board and validate the LRA without the rest of the firmware.

## Defects discovered

- `[DEFECT-001] g_i2c2_mutex via extern through boot_hw_init.h | components/drv2605/haptic.c | LOW (pre-existing pattern) | The driver declares the bus-2 mutex via the same extern pattern that all of Phase 2/3 uses for g_i2c_mutex. boot_hw_init.h does not exist yet; once it lands it must declare BOTH g_i2c_mutex and g_i2c2_mutex. Disposition: documentation; resolves when boot_hw_init.h lands.`
- `[DEFECT-002] Test harness owns GPIO0 directly, bypassing boot_power | test/test_drv2605.c | LOW | In production, boot_power.c configures GPIO0 and drives it LOW. The standalone test harness sets it up itself so it can run before boot_power exists. If the production firmware and the test harness are ever both run on the same board without a reset between them, there's a brief window where the GPIO direction is ambiguous. Disposition: documented; not a real risk because the test is bare-metal app_main.`
- `[DEFECT-003] DRV2605 clone WHO_AM_I = 0xE4 is non-standard | components/drv2605/drv2605.c | LOW (pre-existing) | The TI datasheet says WHO_AM_I should be 0xE0 for an unfused part; the Mk I clone reports 0xE4. The driver doesn't read WHO_AM_I in init, so this is informational only. Disposition: doc-only; flagged here because future bring-up engineers asking "is the chip alive?" will read 0xE4 and worry.`
- `[DEFECT-004] Auto-calibration always fails DIAG on Apple Taptic | components/drv2605/drv2605.c:drv2605_calibrate | MED (pre-existing) | The Apple Taptic Engine's BEMF signature is incompatible with the DRV2605's auto-cal algorithm. drv2605_calibrate() returns ESP_FAIL with STATUS.DIAG_RESULT set. The IMU-assisted sweep in run_sweep() is the working path; haptic_request_calibration() is kept for diagnostic + future ERM/standard-LRA support. Disposition: known limitation, well-documented in driver comments.`
- `[DEFECT-005] Sweep range may not bracket Apple Taptic resonance | components/drv2605/drv2605.h:DRV2605_SWEEP_REG_MIN..MAX | LOW (pre-existing) | Sweep window 50..90 corresponds to ~113-203 Hz. Apple Taptic resonance is ~157 Hz (reg 64) -- comfortably inside the range. If a future LRA SKU has resonance outside this band, the sweep auto-select would land on a boundary value. Disposition: cosmetic; current range is safe for the planned hardware.`
- `[DEFECT-006] RTP_AMP path uses i2c_master_write_to_device directly, not drv2605.c | components/drv2605/haptic.c:HAPTIC_CMD_RTP_AMP | LOW (pre-existing) | The RTP amplitude write bypasses the drv2605.c API (write_reg is file-static). Functionally identical but couples haptic.c to the chip address. Disposition: minor; refactor when the broader haptic refactor lands.`

## Open questions

1. **Does boot_power.c reset GPIO0 across deep sleep?** If GPIO0 floats LOW during sleep, the chip might glitch awake. Recommend: boot_power should hold DRV_EN HIGH during sleep and only release LOW when the haptic task is requested. Out of scope today.
2. **Is V_LRA always-on or sequenced?** If it's sequenced, boot_power should bring up V_LRA, wait ~10 ms, then drop DRV_EN. The legacy v5 had a fixed rail so the question never came up. Confirm on bench.
3. **Bus-2 stretching / pull-up sizing.** I2C bus 2 has only the DRV2605 on it today (per brief), so pull-up sizing is unlikely to be a problem; the 5.1 kOhm convention from bus 1 is fine. If Phase 5+ adds more bus-2 devices, revisit.
4. **Should sweep calibration block UI events?** A full sweep is ~5 s countdown + ~8 s sweep = 13 s. During that time the haptic task is busy and other haptic_play() calls queue up. Recommend: documented behaviour; UI tile gates the sweep button while sweep_active.
5. **Persisted UI effect default vs first boot.** First-boot reads no NVS key, defaults to HAPTIC_EFFECT_CLICK (id=1). Is CLICK the right "out of the box" feel? Recommend: keep -- it's the most universally pleasant of the named effects.
