## Summary

New module -- no v5 carryforward. Mk I exposes a microSD socket on a three-wire SDMMC interface: CLK / CMD / DAT0 only. DAT1-3 are unrouted (v7.2 §GPIO ASSIGNMENT) which forces **1-bit SDMMC** at the host layer. A SD UHS-I card runs at SDR12/SDR25 in 1-bit mode -- ~12.5 MB/s peak -- which is **three orders of magnitude** above what any sensor or audio capture path will throw at it. The bottleneck will be the FAT layer + the application's write cadence, not the wire.

The card is **mounted on demand** and unmounted when not in use. SDMMC host quiescent current sits at ~5-15 mA depending on the card; a watch that logs an hour a day should keep the host parked the other 23.

Three deliverables:

1. **`components/sdcard/sdcard.{c,h}`** -- thin wrapper over `esp_vfs_fat_sdmmc_mount()` plus a session-style log API (one-session-at-a-time, `_NNNN.csv` rotation, recursive mutex for sequential ordering across tasks).
2. **`test/test_sdcard.c`** -- 6-phase bench harness: init / mount / 256 KiB sequential write with per-chunk throughput / read-back verify first+last 16 bytes / one-shot blob / unmount.
3. **This .md** -- pin map, profiling skeleton, defects, open questions.

**SD card-detect (SD_Cd)** -- per v7.2 the SD_Cd signal lives on a **test pad only**, not a GPIO. The driver cannot detect whether a card is physically present until it tries to mount. `sdcard_mount()` returns `ESP_FAIL` or `ESP_ERR_NOT_FOUND` cleanly when the slot is empty; the caller decides what to do about it (typically: surface a "no card" tile and move on). v7.2 §Open Issues item #4 already flags "SD card-detect polarity -- verify when the physical socket arrives".

## Hardware spec from v7.2

### Pinout

| Signal | GPIO | Direction | Notes |
|---|:--:|---|---|
| SD_CLK   | 38 | host out | RTC-incapable. East edge. |
| SD_CMD   | 39 | bidir | RTC-incapable. East edge. IDF enables internal pull-up. |
| SD_DAT0  | 40 | bidir | RTC-incapable. East edge. IDF enables internal pull-up. |
| SD_Cd    | -- | -- | **Test pad only -- not on a GPIO.** Cannot detect card insertion in firmware. |

### Bus

| Item | Value |
|---|---|
| Mode | 1-bit SDMMC (forced; no DAT1-3) |
| Max clock | 40 MHz (`SDMMC_FREQ_HIGHSPEED`); falls back if card rejects |
| Peak throughput | ~12.5 MB/s on UHS-I cards; well below what 4-bit would deliver |
| Mount point | `/sd` |
| Allocation unit | 16 KiB (good FatFS default for sequential log writes) |
| Max open files | 4 (one active log + room for diagnostics) |

### Power

3V3 domain. The card itself can spike 100+ mA during writes (charge-pump + flash erase); the regulator on the 3V3 rail (TPS62840, 750 mA) has plenty of headroom (v7.2 §POWER ARCHITECTURE).

## Code audit

### What dies

Nothing -- new component.

### What's new

- **On-demand mount.** `sdcard_init()` only creates the mutex. `sdcard_mount()` powers up the host, probes the card, and mounts FatFS. `sdcard_unmount()` reverses both.
- **1-bit slot config.** `slot.width = 1`; `slot.flags = SDMMC_SLOT_FLAG_INTERNAL_PULLUP` (no external pull-ups on the Mk I PCB).
- **Session API.** Open / write / flush / close pattern with `name_NNNN.csv` rotation. One session at a time deliberately -- callers that need concurrent sessions must build a higher layer.
- **Recursive mutex.** `xSemaphoreCreateRecursiveMutex()` -- callers can do open+write+close inside a critical section without re-acquiring.
- **`name_is_safe()` whitelist.** Sessions accept only `[A-Za-z0-9_]` characters; no path traversal, no leading dots, no spaces. Defense in depth: even if a hostile caller passes "..", it never reaches `fopen`.
- **One-shot helper.** `sdcard_write_oneshot()` is open + write + close in one call, for callers that don't need streaming.

### What's intentionally NOT here

- **No auto-format.** If the card is unformatted, the mount fails loudly. The user has to format it externally. This is deliberate -- silently formatting a user's card is a feature only a hostile firmware would ship.
- **No background flush task.** Writes are buffered by stdio; `sdcard_flush()` is explicit when needed. A periodic flush would burn idle current; the application knows when durability matters.
- **No retry on mount failure.** If the card isn't there, the caller decides what to do.
- **No `_tile.c`.** Storage is policy-driven by other modules; no standalone "SD" tile is in scope. A future debug UI could wrap `sdcard_get_free_mib()` etc.

## Implementation

### `components/sdcard/sdcard.h`

- Pin defines: `SDCARD_GPIO_{CLK,CMD,DAT0}` = `GPIO_NUM_{38,39,40}`.
- Tunables: `SDCARD_MOUNT_POINT = "/sd"`, `SDCARD_MAX_OPEN_FILES = 4`, `SDCARD_ALLOC_UNIT_SIZE = 16 KiB`.
- Path limits: `SDCARD_SESSION_NAME_MAX = 24`, `SDCARD_PATH_MAX = 64`.
- Identity: `"SD Card"` / `"SDMMC 1-bit, FatFS @ /sd"`.
- Public API: `sdcard_init` / `_mount` / `_unmount` / `_is_mounted` / `_get_capacity_mib` / `_get_free_mib` / `_open_session` / `_write` / `_flush` / `_close_session` / `_write_oneshot`.

### `components/sdcard/sdcard.c`

- File-static state: `s_lock` (recursive mutex), `s_card` (IDF card handle), `s_mounted`, `s_session_f` (FILE*), `s_session_path` (last-opened path).
- `take()` / `give()` recursive-mutex helpers with 2 s timeout.
- `next_session_suffix(name)` -- linear stat() probe 0..9999, returns lowest unused suffix.
- `sdcard_mount()` -- `SDMMC_HOST_DEFAULT()` + 1-bit `SDMMC_SLOT_CONFIG_DEFAULT()`, FatFS mount cfg, on success log card name + capacity.
- `sdcard_unmount()` closes any active session, then `esp_vfs_fat_sdcard_unmount()`.
- `sdcard_get_free_mib()` uses `f_getfree("0:", ...)` -- the raw FatFS API.

### `components/sdcard/CMakeLists.txt`

REQUIRES: `driver`, `sdmmc`, `fatfs`, `vfs`, `esp_timer`, `freertos`.

### `test/test_sdcard.c`

Six-phase bench harness:

1. **Phase 1** -- `sdcard_init`, wall time.
2. **Phase 2** -- `sdcard_mount`, wall time, log capacity / free MiB.
3. **Phase 3** -- Open `test_NNNN.csv`, write 256 KiB in 64 chunks of 4 KiB each with a LCG pattern, flush. Log per-chunk timing and aggregate MiB/s.
4. **Phase 4** -- Read-back verification: re-open the file, compare the first + last 16 bytes against the synthesis pattern.
5. **Phase 5** -- One-shot: 1 KiB blob via `sdcard_write_oneshot("oneshot", ...)`.
6. **Phase 6** -- `sdcard_unmount`, stack high-water + heap.

Bench operator should run this with both a freshly-formatted card and a near-full card. The 256 KiB write size is chosen so the test takes ~50-200 ms on a half-decent card; longer if FatFS has to extend the cluster chain frequently.

## Profiling

### Boot-time cost

| Phase | Component | Time | Method |
|---|---|---|---|
| Init | Mutex create | < 50 us | wall time |
| Init | `sdcard_mount` (SDMMC host + card probe + FatFS mount) | **TBD (no hw)**, ~100-300 ms expected | wall time |
| Init | Card probe alone | ~50-150 ms | dominated by the card's ACMD41 / CMD17 sequence |

### Per-operation cost

| Operation | Time | Notes |
|---|---|---|
| `sdcard_open_session` (with N existing files) | O(N) stat() calls | one stat per probed suffix; cache-friendly |
| `sdcard_write` (4 KiB chunk, stdio-buffered) | **TBD**, < 1 ms typical | hits stdio buffer; actual SD write is deferred |
| `sdcard_flush` (4 KiB pending) | **TBD**, ~5-20 ms | dominated by card's program time |
| `sdcard_close_session` | **TBD**, < 50 ms | flush + close + FatFS metadata update |
| Aggregate write throughput (sequential, 4 KiB chunks) | **TBD**, ~2-8 MiB/s expected | varies wildly by card; UHS-I "U3" cards are the floor |

### Memory

| Metric | Value | Notes |
|---|---|---|
| Driver state | ~120 bytes | mutex handle + card ptr + bools + path buffer |
| FatFS internal | ~5-10 KiB | per-mount overhead (DBR + per-file structs) |
| Per-open file | ~600 bytes | FATFS_FIL + stdio FILE |
| Allocation unit | 16 KiB | configured; matches typical SD erase block |
| Test harness chunk buffer | 4 KiB | `malloc` from internal heap |

### Current draw estimate

| State | Current | Notes |
|---|---|---|
| Host unmounted | ~0 mA at the card slot | card un-powered (no soft-power switch on Mk I) |
| Host mounted, idle | ~5-15 mA at the card | quiescent for a typical SD card |
| Host active, writing | ~50-150 mA peaks | card's internal program current; brief |
| Host active, sustained write | ~20-40 mA average | depends on the card's wear-leveling activity |

### Notes

- **Card power is NOT switched.** v7.2 does not document a soft-power gate on the SD slot; the card stays powered as long as 3V3 is up. `sdcard_unmount()` puts the host into low-power but the card's own quiescent current is unavoidable. If the watch needs deeper sleep with the slot occupied, a future hardware rev should add a load-switch -- flagged in DEFECT-006.
- **Mount latency dominates first-write UX.** 100-300 ms is enough to be perceptible. The watch should mount **eagerly** when the user starts an action that will need persistence (e.g., starting a workout) rather than lazily on the first write.
- **DAT0-only is the binding constraint.** 4-bit mode would quadruple throughput; the Mk II forward-ideas list (v7.2 §KOMPIC̄ Mk II) is the right place to revisit, not Mk I.

## Defects discovered

- `[DEFECT-001] No card-detect (SD_Cd not on a GPIO) | components/sdcard/sdcard.c | MED | v7.2 §Signals NOT on a GPIO confirms SD_Cd is on a test pad only. The firmware cannot detect card insertion / removal without trying to mount. If a user pulls the card mid-write, FatFS gets stuck and the next write may hang for the I/O timeout (~5 s typical). Disposition: documented; the application should treat "open / write failed" as a possible card-removed signal and unmount.`
- `[DEFECT-002] format_if_mount_failed = false swallows unformatted cards | components/sdcard/sdcard.c:sdcard_mount | LOW | An unformatted (or wrong-FS) card returns ESP_FAIL. We deliberately do NOT auto-format -- formatting silently destroys user data. The UI must surface "card not readable" clearly. Disposition: design choice; flagged for UX.`
- `[DEFECT-003] No power-gate on the card slot | hardware-side | MED | v7.2 has no soft-power switch on the SD; the card draws ~5-15 mA whenever 3V3 is up, even unmounted. For a watch that mostly sleeps, this is real money in the battery budget. Disposition: defer to Mk II (v7.2 §KOMPIC̄ Mk II).`
- `[DEFECT-004] Path traversal blocked by name whitelist | components/sdcard/sdcard.c:name_is_safe | LOW | sdcard_open_session rejects names with anything outside [A-Za-z0-9_]. Defense in depth even though the only callers are internal. Disposition: documented behaviour.`
- `[DEFECT-005] next_session_suffix() is O(N) stat per probe | components/sdcard/sdcard.c | LOW | In practice the suffix counter rarely climbs above 100; even at 10000 the cost is bounded. If logs are rotated daily and never cleaned up, this becomes noticeable. Disposition: revisit if the file count routinely exceeds 1000.`
- `[DEFECT-006] No background flush -- written data is at risk on sudden power loss | components/sdcard/sdcard.c | MED | stdio buffers a few KiB; if power dies between flushes, the last write window is lost. Recommend: callers that care about durability (alarm log, error log, OTA progress) call sdcard_flush() at session boundaries or after every significant record. Disposition: documented; callers are responsible.`
- `[DEFECT-007] f_getfree may be slow on near-full large cards | components/sdcard/sdcard.c:sdcard_get_free_mib | LOW | First call after mount scans the whole FAT, which on a 64 GB card with FAT32 can take a couple of seconds. Subsequent calls are cheap (cached). Disposition: avoid calling in a hot path; once-per-mount is fine.`

## Open questions

1. **Who calls `sdcard_init()` / `_mount()` at boot?** The brief implies on-demand; recommend boot does only `sdcard_init()` (cheap) and lets the first logger trigger `sdcard_mount()`. Confirm.
2. **Where does session naming policy live?** Today the caller picks the base name. Should there be a shared `kompic_session_id()` (e.g., "20260612_1330") for consistent naming across modules? UX call.
3. **Does the SD card need a "force-unmount on low-battery" hook?** When battery hits 5%, should we proactively flush + unmount so a sudden brownout doesn't corrupt FatFS metadata? Recommend: YES; coordinate with boot_power.
4. **Should sdcard_mount() retry on transient failure?** First-mount on a freshly-inserted card sometimes needs 2-3 retries while the card finishes its internal init. Recommend: a single retry after 50 ms if the first attempt fails with `ESP_ERR_TIMEOUT`.
5. **Logging format -- CSV or binary?** Brief mentions both. CSV is human-readable + slow + bulky; binary is fast + compact + opaque. Recommend: CSV for low-rate (sensor minutes), binary framing for high-rate (PDM mic, ECG). Caller chooses.
