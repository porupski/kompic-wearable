## Summary

The CO5300AF-51 is the round 2.06" AMOLED panel (PY206-W38-V2) that is the Mk I's only display. It is driven over QSPI from SPI2/FSPI on the ESP32-S3, runs natively at 410 x 502, and consumes RGB888 pixels packed 3 bytes per pixel. It is the user's window into everything the firmware does -- watch face, menus, sensor data, alerts. Until it boots, nothing else matters.

This module is **new for Mk I**. The old design used an ST7789 over SPI2 at 20 MHz with RGB565, which is a fundamentally different protocol, color depth, and wiring. There is no driver code to port; the work is to (a) lift the wire-level facts from v7.2 + the CO5300 datasheet, (b) lay down a skeleton driver with the full register map and a correct SPI framing model, and (c) leave a standalone test harness ready to flash the day the panel lands on the bench.

Today's deliverable is **skeleton + test harness + profiling hooks**, not a fully-debugged driver. The real bring-up cannot happen without hardware.

## Code audit

**CO5300 is new for Mk I.** The old project used an ST7789 over SPI2 at 20 MHz with RGB565 -- fundamentally different protocol, color depth, and wiring. No code to reuse. The only carryover from the old design is the general idea of "panel driver as an ESP-IDF component" -- structurally similar to `components/pcf85063/`. The wire-level details (command opcodes, COLMOD, QIO framing, RGB888 packing, sleep/wake sequence, column offset) are all fresh.

## Implementation

Three files were created in `components/co5300/`:

### `co5300.h`

- Full register / opcode map (NOP, SWRESET, RDDID, RDDST, SLPIN, SLPOUT, INVOFF/ON, DISPOFF/ON, CASET, RASET, RAMWR, TEON/OFF, MADCTL, IDMOFF/ON, COLMOD, WRDISBV, RDDISBV, WRCTRLD).
- Wire-prefix constants for the two framings: `CO5300_WIRE_INSTRUCT = 0x02`, `CO5300_WIRE_PIXELS = 0x32`.
- COLMOD value pinned: `CO5300_COLMOD_RGB888 = 0x77`.
- Panel geometry constants (`CO5300_PANEL_WIDTH/HEIGHT/COL_OFFSET/PIXEL_BYTES`) and the pixel-frame address (`CO5300_PIXEL_ADDR = 0x003C00`).
- `co5300_config_t` with every QSPI pin + clock freq, plus a `CO5300_CONFIG_DEFAULT()` macro pre-filled to the v7.2 wiring.
- Function signatures: `co5300_init`, `co5300_deinit`, `co5300_write_command`, `co5300_write_command_with_data`, `co5300_write_pixels`, `co5300_set_brightness`, `co5300_set_window`, `co5300_sleep`, `co5300_wake`, `co5300_get_status`.

### `co5300.c`

- Opaque `co5300_dev_s` device struct (config copy, SPI handle, awake flag, last brightness).
- Two SPI plumbing helpers, **stubbed** with TODO blocks containing the exact ESP-IDF call shape that will go in once hardware is on the bench:
  - `spi_send_instruction()` -- mirrors a `spi_transaction_ext_t` with `cmd=0x02`, `cmd_bits=8`, `addr_bits=24`, opcode encoded in the upper byte of `addr`, payload on D0 (1-wire).
  - `spi_send_pixels()` -- mirrors a `spi_transaction_ext_t` with `cmd=0x32`, `cmd_bits=8`, `addr_bits=24`, `SPI_TRANS_MODE_QIO` flag, payload on D0..D3.
- All entry points wrap `esp_timer_get_time()` before/after the call and `ESP_LOG` the delta -- profiling hooks live in the driver, not bolted on by callers.
- Init sequence is laid out in code in the order v7.2 specifies for wake: `SLPOUT -> 5 ms delay -> COLMOD 0x77 -> DISPON`. Reset pulse is in a commented block (no GPIO calls in the stub -- can't drive a pin on hardware that isn't there).
- `co5300_set_window()` already applies the `+22` column offset so callers do not have to.
- `co5300_get_status()` returns `ESP_ERR_NOT_SUPPORTED` and is wired to `[DEFECT-001]`.

### `CMakeLists.txt`

Standard `idf_component_register` with `REQUIRES driver freertos esp_timer`. Mirrors the `pcf85063` component style.

### `test/test_co5300.c`

Standalone `app_main()` that:
1. Logs chip identity strings.
2. Bookmarks SPI-bus-init wall time (fused into `co5300_init()` in the skeleton).
3. Calls `co5300_init()`, logs delta + heap delta.
4. Calls `co5300_get_status()` and logs the (currently NOT_SUPPORTED) result.
5. Sends `DISPON`, then `set_brightness(50)`, then `set_brightness(100)`, logging each.
6. Cycles `sleep()` / `wake()` to exercise the SLPIN/SLPOUT path.
7. Pushes 1 synthetic RGB888 pixel through the QIO path.
8. Tears down and logs final heap delta.

Test is intentionally a flat sequence -- no menus, no parametrization -- so the first bring-up output is dead simple to read.

### Per-operation cost

| Operation | Time | Bus | Notes |
|---|---|---|---|
| Instruction frame, no payload (DISPON) | **TBD** | QSPI 40 MHz (1-wire cmd phase) | overhead dominated by SPI setup, not bytes |

| State | Current (mA) | Notes |
|---|---|---|
| Init / idle (panel awake, black frame) | TBD | depends on COLMOD + display content; AMOLED so dark frame is cheap |

### Notes
- The wire-time floor for a full frame at 40 MHz (~123 ms) is the load-bearing performance number. If we want >10 fps full-screen, we have to raise the QSPI clock or move to partial updates.
- AMOLED behavior means power is content-dependent. UI design should default to dark backgrounds.
- The TODO blocks in `co5300.c` cover the only path to actual wire traffic; everything else is structurally complete.

## Defects discovered

- `[DEFECT-001] WHO_AM_I semantics not certain | components/co5300/co5300.c:co5300_get_status | LOW | The CO5300 datasheet does not advertise a stable WHO_AM_I-style chip ID register. RDDID (0x04) and RDDST (0x09) exist, but the returned bytes are not specified to be unique to the CO5300. Disposition: the test harness logs whatever the chip returns on the first bench bring-up; once we know the actual response, we pin it as a constant and treat it as the handshake. Until then, `co5300_get_status()` returns ESP_ERR_NOT_SUPPORTED.`
- `[DEFECT-002] TE pin (GPIO45) is firmware-unused | components/co5300/co5300.h:pin_te | MED | GPIO45 is wired to QSPI_TE as a VDD_SPI strap and stays that way permanently (v7.2 line 165). Firmware in v1 does not consume the TE pulse, so we have no tearing sync -- any partial-frame writes during refresh can produce visible tearing. Disposition: acceptable for v1 (full-frame updates dominate). Revisit when partial-update paths land.`
- `[DEFECT-003] Reset pulse not exercised in skeleton | components/co5300/co5300.c:co5300_init | LOW | The reset block is commented out because there is no hardware to drive. The exact pulse width (currently planned: 10 ms low, 120 ms wait after release) is from generic OLED practice, not the CO5300 datasheet. Disposition: confirm timing against the CO5300 datasheet, then enable on first bench bring-up.`
- `[DEFECT-004] Pixel-clock 40 MHz unverified | components/co5300/co5300.h:freq_hz | LOW | 40 MHz is the kickoff brief's starting point. Maximum stable QSPI clock on the actual PCB trace lengths is unknown. Disposition: bench-sweep clock once the panel is alive, log the highest stable rate.`
- `[DEFECT-005] No git repo in working dir | repo root | MED | The project at /home/ivan/esp/ESP32-S3/Kompic_Mk1_v7.2.0.0/ is not a git repository (.git is absent). ## Open questions

1. **Status / WHO_AM_I:** is there a known-good chip-ID byte sequence for the CO5300 that we can pin in code, or do we treat "any non-zero RDDST response" as proof of life? (Blocks `co5300_get_status` real implementation.)
2. **Init sequence completeness:** v7.2 lists SLPOUT, COLMOD, DISPON. The Waveshare-style reference boards for similar panels also send a long manufacturer-specific init block (gamma, charge-pump, panel-specific timing). Does the CO5300 boot to a sensible default without that block, or do we need the full vendor init list?
3. **Reset pulse timing:** is the 10 ms low / 120 ms post-release rule correct for CO5300, or does the datasheet specify a tighter / looser pulse? (Blocks enabling the reset code in `co5300_init`.)
