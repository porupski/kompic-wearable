## Summary

The LIS3MDLTR is the watch's compass. It sits on I2C bus 1 at 0x1C, runs continuously in 10 Hz / ±4 G mode, and feeds heading + cardinal direction into `broker_mag_data_t`. The chip is wire-incompatible with the v5 QMC5883P at every byte but the broker contract, the hard-iron min/max sweep calibration, and the heading-from-atan2 math all carry forward.

The interesting new wrinkle is the on-board LRA. v7.2 line 282 + line 432 flag that the haptic actuator sits ~6-7 mm below the magnetometer, contributing an expected DC magnetic offset of ~1 gauss (= 100 µT) -- about twice the Earth field magnitude. Standard hard-iron calibration sweeps that out, but a dedicated `lis3mdl_measure_lra_offset()` API is added so the first-boot calibration can isolate "this is the LRA" from "this is the surrounding environment".

Today's deliverable: replaced driver code + renamed-and-resized tile + LRA-offset API + standalone test harness + this doc.

| Alternate | 0x1E (SA1 = VDD_IO via JP6 cut+bridge) |

### Power

| Rail | Connected | Notes |
|---|---|---|
| 3V3 | LIS3MDLTR VDD | TPS62840 buck, always-on with battery |
| 1V8 | LIS3MDLTR VDD_IO | XC6206 LDO |

Like the LSM6DSV16X, the LIS3MDLTR's digital interface runs from the 1V8 rail. Open-drain I2C lines pulled to 3V3 are fine; the chip itself handles the voltage difference.

### LRA proximity (v7.2 line 282 / line 432)

The DRV2605 LRA sits on the daughter board below the main board, ~6-7 mm from the LIS3MDLTR. v7.2 line 432 says: "measure field on the prototype; separate further only if it saturates a range." With FS = ±4 G (default) we have ~3 G of headroom over the LRA's DC offset + Earth's ~0.5 G -- comfortable. If we ever measure >2 G offset, we drop to FS = ±8 G in CTRL2.

## Code audit

### What dies

`components/lis3mdl/qmc5883p.{c,h}` -- deleted. The QMC5883P used a completely different register layout (REG_DATA_X_LSB = 0x01, CTRL1 = 0x0A, no WHO_AM_I check, no auto-increment bit), different address (0x2C vs 0x1C), and a different sensitivity (3000 µT / 32768 LSB vs LIS3MDLTR's 6842 LSB/G).

### What carries forward

- **`broker_mag_data_t` shape, field-for-field.** Now defined in `lis3mdl.h`. Same `x_ut / y_ut / z_ut / heading_deg / cardinal / calibrated / calibrating / cal_countdown / bookkeeping` set. `compass_tile.c` builds unchanged (modulo the include path swap and identity-function rename).
- **Hard-iron min/max sweep.** `lis3mdl_start_calibration / update_calibration / finish_calibration` mirror the old `qmc5883p_*` API one-for-one. The calibration storage struct is renamed `qmc5883p_calibration_t` -> `lis3mdl_calibration_t` (only consumed inside this component and by boot_hw_init for NVS seed/restore).
- **Heading math.** `lis3mdl_calculate_heading(x, y, declination)` uses `atan2f(y, x)`, normalises to 0-360, adds declination. Identical math to qmc5883p_calculate_heading. The housing-rotation correction comment is preserved as a TODO/marker.
- **Task structure.** `task_mag_fn` (10 Hz poll, read-before-write, apply offset + scale, compute heading + cardinal) and `task_mag_cal_fn` (figure-8 sweep, 30 s countdown, broker `calibrating` flag handshake) both carry forward.
- **NVS calibration seed.** `lis3mdl_seed_calibration(...)` is called once at boot from `boot_hw_init.c` to restore the last successful calibration.

### What's new

- **WHO_AM_I check.** QMC5883P had none. LIS3MDLTR has 0x3D at REG 0x0F.
- **Auto-increment bit.** Burst-reads on the LIS3MDLTR require the sub-address to have bit 7 set (`reg | 0x80`). This is the most common LIS3MDLTR bring-up footgun.
- **Continuous-mode write-order.** CTRL1 (rates) -> CTRL2 (FS) -> CTRL4 (Z axis mode to match X/Y) -> CTRL5 (BDU) -> CTRL3 (start). Writing CTRL3 last is important because that's what kicks the chip out of power-down.
- **`lis3mdl_measure_lra_offset(...)`** -- new API. 20 samples over ~1 second, simple average. The result is intended to be subtracted from the hard-iron offset during first-boot calibration so the LRA component is separated from the environment component.

## Implementation

### `components/lis3mdl/lis3mdl.h`

- Wire-level constants `LIS3MDL_ADDR = 0x1C`, `LIS3MDL_WHO_AM_I_VAL = 0x3D`, `LIS3MDL_AUTO_INC = 0x80`.
- Register-address `#define`s for CTRL1..CTRL5, STATUS, OUT_X_L, OUT_TEMP_L, INT_CFG. All [DSV].
- Control-register bit defines: `CTRL1_OM_UHP_XY`, `CTRL1_DO_10HZ`, `CTRL2_FS_4G`, `CTRL2_SOFT_RST`, `CTRL3_MD_CONTINUOUS`, `CTRL3_MD_POWER_DOWN`, `CTRL4_OMZ_UHP`, `CTRL5_BDU`.
- Sensitivity constant: `LIS3MDL_LSB_PER_UT = 68.42` (= 6842 LSB/G at ±4 G FS).
- `broker_mag_data_t` -- field-for-field preserved.
- `lis3mdl_calibration_t` -- renamed from `qmc5883p_calibration_t`. Same fields.
- Public API: identity, init/deinit, task_mag_fn, task_mag_cal_fn, read_raw, calculate_heading, set_standby, wake, seed_calibration, start/update/finish_calibration, **measure_lra_offset**.

### `components/lis3mdl/lis3mdl.c`

- I2C helpers using the modern `i2c_master_write_to_device` / `write_read_device` APIs.
- `lis3mdl_init`: take mutex, read WHO_AM_I, write CTRL2.SOFT_RST, release mutex + delay 10 ms, re-take, write CTRL1/CTRL2/CTRL4/CTRL5/CTRL3 in that order, release, delay 20 ms. Log.
- `lis3mdl_read_raw`: 6-byte burst from `OUT_X_L | AUTO_INC`, scale via `LIS3MDL_LSB_PER_UT`. Caller holds mutex.
- `lis3mdl_calculate_heading`: pure atan2 + declination + 0-360 normalisation. No side effects.
- Calibration code: identical structure to qmc5883p.c -- min/max trackers + offset/scale computation + memcpy into module-static state.
- `lis3mdl_measure_lra_offset`: 20 samples × 50 ms = ~1 s; returns averaged XYZ.
- `task_mag_fn`: 100 ms poll, read-before-write, apply hard-iron offset + (if calibrated) scale normalisation, compute heading + cardinal, broker write.
- `task_mag_cal_fn`: poll for `bd.calibrating == true`; on rising edge, run a 30-second figure-8 with 1 Hz countdown via `bd.cal_countdown`. Cancellable by Core 1 flipping `calibrating` back to false. Verbatim port of the existing qmc5883p flow.

### `components/lis3mdl/compass_tile.c`

- `#include "qmc5883p.h"` -> `#include "lis3mdl.h"`.
- Two identity-function call sites renamed: `qmc5883p_get_chip_name/desc` -> `lis3mdl_get_chip_name/desc`.
- Layout constants pulled to named locals at the top of `compass_tile_init()` so the 240x280 -> 410x502 rescale is one block of `#define`s. Rose grown from 70px to 120px. Cal button widened to 280×40.

### `components/lis3mdl/CMakeLists.txt`

- `SRCS qmc5883p.c` -> `lis3mdl.c`. `compass_tile.c` unchanged.

### `test/test_lis3mdl.c`

Standalone harness:
1. I2C bus 1 init at 400 kHz on GPIO1/2.
2. `lis3mdl_init` with WHO_AM_I check.
3. 10 raw reads at 100 ms spacing, printing XYZ + magnitude + heading + per-read dt.
4. `lis3mdl_measure_lra_offset` -- log result and warn if |B| < 30 µT (the magnetometer is supposed to see Earth's field + LRA offset).
5. Stack high-water + free heap report.

### Notes

- **The LRA offset is the load-bearing measurement risk.** If we see >2 G on the prototype, we have to widen CTRL2.FS or move the magnetometer. v7.2 line 432 explicitly flags this.
- **10 Hz is the right number.** Heading doesn't change faster than the wrist moves; faster sampling wastes I2C bus 1 time that could go to the IMU.
- **The auto-increment bit is the most common bring-up footgun.** If burst reads return wraparound nonsense, that's the first thing to check.

## Defects discovered

- `[DEFECT-001] All register addresses + bit positions tagged [DSV] | components/lis3mdl/lis3mdl.h | CRITICAL | Register addresses (0x0F WHO_AM_I, 0x20-0x24 CTRL1..5, 0x27 STATUS, 0x28 OUT_X_L) and the OM/DO/FS/MD bit positions in CTRL1/CTRL2/CTRL3/CTRL4 are family-conventional but NOT verified against ST DM00075867 in this session. Disposition: BLOCK BENCH BRING-UP until each is confirmed.`
- `[DEFECT-002] Sensitivity constant assumed for ±4 G FS only | components/lis3mdl/lis3mdl.h:LIS3MDL_LSB_PER_UT | HIGH | The driver hardcodes 68.42 LSB/uT (= 6842 LSB/G at ±4 G FS). If we ever switch to FS = ±8 G via a CTRL2 write, the scaling becomes wrong by a factor of 2. Disposition: when an FS-change API is added, also flip the scaling constant. Today there's no public FS-change API, so the constant is correct as long as the driver owns the FS bits.`
- `[DEFECT-003] LRA offset measurement is a single one-shot | components/lis3mdl/lis3mdl.c:lis3mdl_measure_lra_offset | MED | The function takes 20 samples over ~1 second with no warm-up, no outlier rejection, no integration with the hard-iron calibration. It is intended as a diagnostic, not a production calibration step. Disposition: refine after bench data shows what the LRA offset actually looks like (constant, drifting with temp, drifting with charging state, etc.).`
- `[DEFECT-004] Housing rotation correction is identity | components/lis3mdl/lis3mdl.c:lis3mdl_calculate_heading | MED | The heading math assumes the chip's X axis points to the crown / 12 o'clock. The Mk I PCB orientation has not been verified against this assumption. If the LIS3MDL is rotated relative to the watch crown, the heading is offset by a constant and "North" points the wrong way. Disposition: verify on bench by physically pointing the watch at a known direction and reading bd.heading_deg.`
- `[DEFECT-005] CTRL write order assumed | components/lis3mdl/lis3mdl.c:lis3mdl_init | LOW | The driver writes CTRL1 -> CTRL2 -> CTRL4 -> CTRL5 -> CTRL3, with CTRL3 (mode select) last. The datasheet does not strictly mandate this order, but it's a sane convention (configure everything, then enable). Disposition: confirm on bench that the chip actually reaches continuous mode (read CTRL3 back).`
- `[DEFECT-006] g_i2c_mutex via extern | components/lis3mdl/lis3mdl.c | LOW (pre-existing) | Same pattern as elsewhere.`

## Open questions

1. **LIS3MDLTR datasheet locally available?** Single review collapses [DEFECT-001], [DEFECT-002], [DEFECT-005].
2. **Declination source.** Today `lis3mdl_calculate_heading` accepts a declination parameter; the task calls it with `0.0f`. Real declination is GPS-position-dependent (varies from ±20° across the globe). Question: do we want a NVS-stored user-set declination, or a GPS-position-driven lookup, or just leave it at 0?
3. **LRA offset baked into the hard-iron calibration?** Today the hard-iron sweep captures EVERYTHING (Earth + LRA + local environment); the LRA contribution is implicit, not separated. The new `lis3mdl_measure_lra_offset` is exposed but not used in `task_mag_fn`. Recommend: keep it diagnostic-only until we have bench data on how stable the LRA offset is.
4. **6D / 9D fusion with LSM6DSV16X.** The brief (line 142) suggests "LIS3MDLTR references LSM6DSV16X in compass_tile for coordinated calibration (optional Phase 2.5)". Today no such reference exists. Recommend: defer; tilt compensation can come later via the fusion task.
5. **PCB orientation.** What direction does the LIS3MDLTR X axis physically point relative to the crown? Until that's confirmed, [DEFECT-004] is open.
