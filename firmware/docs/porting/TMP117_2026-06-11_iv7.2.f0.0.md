## Summary

New chip in Mk I -- no legacy v5 driver to port. The TMP117 reads skin-contact temperature through the back of the case at 1 Hz, +/- 0.1 C accurate across the human-body range. Out of the four Phase 3 modules this is by far the simplest: one I2C read every 5 s, no calibration, no FIFO, no interrupt routing.

The non-trivial part is **broker integration**. There is no existing `broker_skin_data_t` channel; this module introduces it in `tmp117.h`. The corresponding `broker_skin_*` proxy API must be added to `data_broker.{c,h}` as a follow-up step (same pattern that Phase 2 left behind for `boot_hw_init.h` and the now-orphaned `bh1750.h` / `qmi8658.h` includes). The driver itself is complete; the broker shim is a 1-line `BROKER_MODULE_IMPL(skin, SKIN, BROKER_SKIN_TIMEOUT_MS, true)` macro invocation away.

The master prompt's note that TMP117 "shares an address with VEML" is **incorrect** -- VEML6030 is at 0x10 (per Phase 2's VEML6030 .md) and TMP117 is at 0x48. Distinct addresses; no collision. Worth flagging since it's the kind of misconception that ends up wasting a bench session.

Today's deliverable: new driver (init + read + 5 s polling task), broker_skin_data_t declared, standalone test harness with skin-range sanity check, dated .md.

## Hardware spec from v7.2

### Pinout

| Signal | GPIO | Notes |
|---|:--:|---|
| TMP_SDA   | 1 | I2C bus 1 SDA, shared |
| TMP_SCL   | 2 | I2C bus 1 SCL, shared |
| TMP_ALERT | -- | Not routed on Mk I |

### I2C / addresses

| Item | Value |
|---|---|
| Bus | I2C bus 1 (`I2C_NUM_0`) |
| Speed | 400 kHz |
| Address | 0x48 (ADD0 = GND -- factory default) |
| Alternates | 0x49 (ADD0 = VDD), 0x4A (ADD0 = SDA), 0x4B (ADD0 = SCL) -- not used |
| Device ID | low 12 bits = 0x117 at REG 0x0F |

The master prompt §Phase 3 says "confirm no I2C collisions (same addr as VEML)". This is wrong -- VEML6030 is at 0x10 and TMP117 is at 0x48. Flagged in summary; no action needed.

### Power

| Rail | Connected | Notes |
|---|---|---|
| 3V3 | VDD | TPS62840 buck, always-on |

Single rail. Datasheet says VDD range 1.8-5.5 V; 3V3 is comfortably in the middle.

### Sensor placement

Per v7.2 §SKIN TEMP, the TMP117 sits against the inside surface of the back case so it reads through metal/plastic to skin contact. The 0.1 C accuracy is the spec at the sensor; the metal interface adds some lag (seconds) but does not degrade accuracy meaningfully. We poll at 5 s because the metal interface alone makes faster polling pointless.

## Code audit

### What dies

Nothing -- new chip.

### What's new (everything)

- **Driver shape.** `tmp117_init` + `tmp117_read_temp_c` + `task_skin_fn` + identity getters. 110 lines of C, no surprises.
- **`broker_skin_data_t` shape.** New broker channel: `skin_temp_c (float)`, `valid (bool)`, plus the mandatory `last_update_ms` + `enabled` bookkeeping.
- **5 s polling task.** Rather than the 200 ms / 500 ms / 2 s cadences of other sensors. Skin temperature changes on the order of minutes when worn, so 5 s is the right resolution.
- **Read-before-write pattern.** Same convention as the rest of the Phase 2/3 drivers. The `enabled` field is Core-1-written; the task preserves it.

## Implementation

### `components/tmp117/tmp117.h`

- Wire constants: `TMP117_ADDR = 0x48`, `TMP117_I2C_PORT = I2C_NUM_0`.
- Register map (TI SBOS719C) for the registers we actually touch: TEMP_RESULT (0x00), CONFIG (0x01), DEVICE_ID (0x0F). EEPROM / limit registers are defined but unused today.
- `TMP117_CONFIG_RESET = (1 << 1)` -- soft reset bit.
- `TMP117_LSB_PER_C = 128` -- scaling constant (1 LSB = 1/128 C = 0.0078125 C).
- `TMP117_POLL_MS = 5000`.
- `broker_skin_data_t` -- defined here (driver owns the broker shape).
- `BROKER_SKIN_TIMEOUT_MS = 15000` -- 3x the poll period.
- Identity strings `"TMP117"` / `"Skin temperature"`.
- Public API: `tmp117_init`, `tmp117_read_temp_c`, `tmp117_get_last_temp_c`, `task_skin_fn`.

### `components/tmp117/tmp117.c`

- `read_reg16 / write_reg16` -- 16-bit MSB-first via the modern `i2c_master_write_read_device` / `write_to_device` API.
- `tmp117_init`:
  - Takes `g_i2c_mutex` internally (caller passes through).
  - Reads DEVICE_ID; warns on mismatch but does not hard-fail.
  - Writes `CONFIG = TMP117_CONFIG_RESET`. The reset bit self-clears; datasheet says "wait 1.5 ms". We wait 5 ms.
  - Returns ESP_OK.
- `tmp117_read_temp_c`:
  - Reads TEMP_RESULT (raw uint16, MSB first).
  - Casts to int16_t (two's complement) and divides by 128.0 (datasheet scaling).
  - Caller holds the mutex; the function does not touch it.
- `task_skin_fn`:
  - 5 s vTaskDelayUntil cadence.
  - Gates on `broker_skin_hw_alive() && broker_skin_get_enabled()`. **These broker functions do NOT exist yet** -- see DEFECT-002.
  - Mutex-protected `tmp117_read_temp_c`, mutex held only for the wire transaction (~120 us at 400 kHz).
  - Read-before-write: `broker_skin_read(&bd)`, mutate `skin_temp_c` + `valid`, `broker_skin_write(&bd)`.
- `s_last_temp_c` -- file-static last successful reading; exposed via `tmp117_get_last_temp_c()` for the test harness.

### `components/tmp117/CMakeLists.txt`

New component, registered with REQUIRES driver + data_broker + boot_logic (for the eventual `g_i2c_mutex`).

### `test/test_tmp117.c`

Five-phase standalone harness:

1. **Phase 1** -- I2C bus 1 init.
2. **Phase 2** -- `tmp117_init` (DEVICE_ID probe + soft reset).
3. **Phase 3** -- 6 reads at 1 s spacing. Each read logs temperature + per-read I2C wire time.
4. **Phase 4** -- Sanity flag: any reading in the 20-40 C window is tagged as "skin range"; 10-50 C as "ambient-ish". Counts how many reads land in skin range. Useful for "is this thing alive and on a wrist?".
5. **Phase 5** -- Stack high-water + heap.

## Profiling

### Boot-time cost

| Phase | Component | Time | Method |
|---|---|---|---|
| Init | I2C bus 1 setup | **TBD (no hw)** | wall time |
| Init | DEVICE_ID probe (1 reg16 read) + soft reset (1 reg16 write) + 5 ms wait | ~5.5 ms | architectural; the wait dominates |

### Per-operation cost

| Operation | Time | Notes |
|---|---|---|
| TEMP_RESULT read (1 byte addr + 2 byte data) | **TBD**, ~120 us wire floor | at 400 kHz |
| Two's complement decode + divide by 128 | < 5 us | one int16 cast + one fdiv |

### Memory

| Metric | Value | Notes |
|---|---|---|
| Driver state | 4 bytes (s_last_temp_c) | -- |
| `task_skin` stack | 2048 bytes recommended; high-water **TBD** | task does almost nothing |

### Current draw estimate

| State | Current | Notes |
|---|---|---|
| Continuous conversion (default, 8-sample avg @ 1 Hz) | ~3.5 uA average | datasheet typical |
| Shutdown mode | < 0.5 uA | not used; the chip happily runs continuous |

### Notes

- **5 uA is below the measurement floor** of most bench supplies. We will not see the TMP117 contribute meaningfully to the system current draw; the I2C bus traffic alone is more expensive than the chip.
- **5 s poll is intentional.** Skin temperature has a ~30 s time constant on a watch; polling faster wastes bus time without adding information.

## Defects discovered

- `[DEFECT-001] Master prompt says TMP117 shares VEML's I2C address -- it does NOT | docs/10_KOMPIC_MK1_FIRMWARE_PORTING_MASTER_PROMPT.md:149 | LOW | The master prompt's Phase 3 entry for TMP117 says "confirm no I2C collisions (same addr as VEML)". VEML6030 is at 0x10 (per Phase 2 deliverable), TMP117 is at 0x48. No collision. Disposition: cosmetic doc nit. Worth fixing in the master prompt to prevent confusion in future review.`
- `[DEFECT-002] broker_skin_* API does NOT exist yet | components/tmp117/tmp117.c:task_skin_fn | HIGH | The driver references broker_skin_hw_alive(), broker_skin_get_enabled(), broker_skin_read(), broker_skin_write(). None of these are declared in data_broker.h yet. Link will fail until data_broker.{c,h} is updated to add the broker_skin channel. Same Phase 2 pattern as g_i2c_mutex via extern. Disposition: add the broker shim as a follow-up (1-line BROKER_MODULE_IMPL macro invocation in data_broker.c plus the API block in data_broker.h). Cannot be done from this module in isolation because data_broker.h pulls headers from multiple components.`
- `[DEFECT-003] DEVICE_ID check accepts low-12-bit mismatch with a warning | components/tmp117/tmp117.c:tmp117_init | LOW | Like the MAX30101 Part-ID check, we warn but continue if the value isn't 0x?117. The TMP117 family is a single SKU, so a mismatch almost certainly means wrong I2C address (we're reading something else) -- but we let the rest of the init run anyway. Disposition: keep as warning; the read-temperature path will produce nonsense and the SENSOR_OFFLINE state will reflect that.`
- `[DEFECT-004] Soft-reset wait is 5 ms vs datasheet 1.5 ms | components/tmp117/tmp117.c:tmp117_init | LOW | We sleep 5 ms after writing the reset bit; datasheet says 1.5 ms is enough. 3.5 ms of extra boot time is fine. Disposition: cosmetic.`
- `[DEFECT-005] No ALERT pin handling | components/tmp117/tmp117.c | LOW | The TMP117 has a threshold-based ALERT output. v7.2 says it's not routed in Mk I, so we ignore it. If a future PCB rev routes it, we'd add a wake-on-temperature path. Disposition: not blocking.`
- `[DEFECT-006] g_i2c_mutex via extern | components/tmp117/tmp117.c | LOW (pre-existing) | Same as elsewhere.`

## Open questions

1. **Broker integration follow-up.** When do we add `broker_skin_*` to `data_broker.{c,h}`? Recommend: bundle with the next batch (probably Phase 4 actuators, since the data_broker.h will need attention anyway for any other new channel).
2. **Calibration.** The TMP117 ships factory-calibrated to +/- 0.1 C. The watch case adds a temperature gradient between the sensor and skin -- worth measuring once on bench (compare against a clinical thermometer worn for 30 min) and applying a single-offset correction in the broker write if the gradient is significant. Probably 0.5-1.5 C low.
3. **Use in fusion / alerts.** Skin temp on a watch is interesting for: detecting fever onset (slow rise over hours), detecting cold-extremity stress, calibrating the MAX30101 IR reading (LED current vs ambient drift). Question: do we want any of these in v1? Recommend: just display the number for now; revisit when fusion-layer work starts.
4. **Subtile in health_tile.** The batch doc says "no tile of its own -- subtile in health_tile". Implementing that subtile is out of scope today (it's a UI / lvgl_ui concern, not a driver concern). When ready, health_tile would gain a small "skin: 36.2 C" row reading from broker_skin_read().
