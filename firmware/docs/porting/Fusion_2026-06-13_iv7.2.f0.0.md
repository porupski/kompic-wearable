## Summary

Tier-3 derived-data aggregator. Reads multiple broker slots, writes a single
`broker_fusion_data_t`. Current responsibilities:
- **Altitude arbitration:** GPS fix → BME688 baro → last-known GPS → none.
- **Activity classification:** stub (UNKNOWN/0%).

Runs 1 Hz on Core 0, priority 2, stack 2048. Reads only — never fires
cross-driver events.

## Pinout / bus from v7.2

N/A. Reads only from broker slots populated by the chip drivers.

## Code audit (v5 → v7.2)

The v5 fusion module shipped with the BME280 read path **commented out** because
the broker did not yet expose `broker_env_read`. With BME688 in the v7.2 broker,
the read path activates.

Additionally, the v5 fusion held its own `baro_altitude_m()` helper duplicating
the hypsometric formula. The v7.2 BME688 driver already computes
`broker_env_data_t.altitude_m` inside its own task (P0 = 1013.25 hPa,
documented in `bme688_drv.h:44`). Fusion now reads that field directly and the
local helper is removed. This:
- eliminates duplicated floating-point work,
- aligns with "fat driver / thin broker" (the chip layer owns the math),
- centralises any future sea-level recalibration in one place (BME688 driver),
- drops the `newlib` PRIV_REQUIRES dependency from CMakeLists.

## Test harness

`firmware/test/test_fusion.c`:

1. Boot broker + cross_driver init; spawn fusion task.
2. **Scenario A** — GPS fix present:
   - `broker_gps_set_hw_status(true) + set_enabled(true)`
   - Write `broker_gps_data_t{.fix=GPS_FIX_3D, .position_valid=true, .altitude_m=712.4}`
   - Wait ≥ 1.1 s for one fusion tick.
   - Read `broker_fusion_data_t`; assert `altitude_source == ALTITUDE_SRC_GPS`,
     `altitude_m ≈ 712.4`.
3. **Scenario B** — no GPS, BME688 valid:
   - `broker_gps_set_hw_status(false)`
   - `broker_env_set_hw_status(true) + set_enabled(true)`
   - Write `broker_env_data_t{.pressure_hpa=998.0, .altitude_m=125.0}`
   - Wait ≥ 1.1 s.
   - Assert `altitude_source == ALTITUDE_SRC_BARO`, `altitude_m ≈ 125.0`.
4. **Scenario C** — no GPS, no ENV, previous GPS present:
   - Pre-load broker_fusion with `altitude_gps_m=712.4`.
   - Disable GPS + ENV.
   - Wait ≥ 1.1 s.
   - Assert `altitude_source == ALTITUDE_SRC_LAST`, `altitude_m ≈ 712.4`.
5. **Scenario D** — nothing:
   - Disable GPS + ENV; pre-load broker_fusion with `altitude_gps_m=0`.
   - Wait ≥ 1.1 s.
   - Assert `altitude_source == ALTITUDE_SRC_NONE`.
6. Stack high-water check: `uxTaskGetStackHighWaterMark` after 10 ticks must
   confirm the 2048-byte allocation has substantial headroom (target > 512
   bytes free).

### Notes
- 1 Hz is a deliberate choice: derived data does not need higher cadence (GPS
  updates at 1 Hz typical, BME688 at 0.5 Hz). Raising this would just consume
  mutex windows without producing newer data.
- The `baro_altitude_m` removal eliminates a duplicate `powf()` call per tick.
  Phase 20 datasheet pass should confirm BME688's internal pressure-to-altitude
  conversion matches expectations (and exposes a sea-level pressure setter if
  one exists in the chip's IIR-filter config).
- Activity classification stays a stub. Reason: meaningful activity detection
  needs IMU step-count + HR + dwell-time signals, none of which are wired to
  fusion yet. Deferred to v2.

## Defects discovered

[DEFECT-001] Sea-level reference is hard-coded in BME688 driver | components/bme688/bme688_drv.h:44 + driver source | LOW | The hypsometric formula uses P0 = 1013.25 hPa with no runtime calibration. Altimetry in regions of high/low pressure (or near sea-level vs mountain) will drift. The `on_gps_fix_valid` hook in fusion is the natural place to call into BME688 and update P0 when a GPS altitude is available. Disposition: Phase 20 (datasheet pass). The BME688 datasheet may expose an in-chip pressure-offset register that is cleaner than a software P0.

[DEFECT-002] No safeguard against repeated `prev` carry-forward | components/fusion/fusion.c (altitude arbitration block) | LOW | When GPS and ENV both go missing for a long time, ALTITUDE_SRC_LAST keeps carrying the stale GPS altitude forever. There's no decay or invalidation. UI consumers can detect this by checking `altitude_source == ALTITUDE_SRC_LAST` but no time bound is signalled. Disposition: acceptable today (UI tile can colour the tile orange when source = LAST); revisit if user feedback says altitude is "stuck".
