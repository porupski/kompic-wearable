## Summary

New module -- no v5 carryforward. Mk I has a single PDM MEMS mic wired to I2S0 (GPIO47 CLK, GPIO48 DATA). v7.2 §SOLDER JUMPERS JP9 selects the L/R slot at the bare-die level; default is **GND = left channel**, so the IDF channel is configured for `I2S_PDM_SLOT_LEFT` to match.

Capture-only driver for firmware v1 -- no VAD, no Speex, no compression. The brief is explicit: "capture only; preprocessing in Phase 2+". The decimation from 1-bit PDM to 16-bit PCM is done by the ESP32-S3 I2S0 hardware PDM-to-PCM block; the driver just exposes the post-decimation 16-bit samples.

Three deliverables:

1. **`components/mic_pdm/mic_pdm.{c,h}`** -- IDF v5 i2s_pdm RX channel, 6 DMA frames * 320 samples = 120 ms total buffer in PSRAM, blocking `_read()` API + a convenience `_capture(int16_t *, n_frames)`.
2. **`test/test_mic_pdm.c`** -- 5-phase bench harness: init / start / 50-frame DC-peak-RMS analysis / convenience capture / stop+deinit. Bench operator looks for non-zero RMS in a quiet room (the noise floor) and a peak jump when tapping near the port.
3. **This .md** -- pin map, profiling skeleton, defects, open questions.

Frame size choice: **20 ms (320 samples)** is the de-facto speech-frame standard (every codec from G.711 to Opus uses something in the 10-30 ms range). At 16 kHz / 16-bit that's 640 bytes per frame; 6 frames in flight = ~3.8 KiB DMA backing. The IDF v5 PDM RX driver uses PSRAM for these buffers automatically when `CONFIG_SPIRAM_USE_MALLOC` is on (it is for this project per v7.2's 8 MB octal PSRAM).

### Solder jumper interaction

| Jumper | Default | Driver expects |
|---|---|---|
| JP9 (Mic L/R) | GND (left) | `I2S_PDM_SLOT_LEFT` |

If JP9 is moved to VDD (right) on a bench unit, the driver will run but the data slot will be empty (all zeros). **Symptom: RMS=0 across every frame in the test harness.** -- see DEFECT-001.

### Power

3V3 domain (mic VDD). The mic's typical current is < 1 mA at 16 kHz. The I2S0 controller adds a few mA of its own. Negligible vs the rest of the system.

## Code audit

### What dies

Nothing -- new component.

### What's new

- **IDF v5 PDM RX channel.** `i2s_new_channel` + `i2s_channel_init_pdm_rx_mode`, with the `i2s_pdm_rx_config_t` matching v7.2 wiring.
- **DMA topology.** 6 frames * 320 samples in a ring -- IDF moves them in HW; the app calls `i2s_channel_read()` when it wants the next batch.
- **Lifecycle separation.** `_init` allocates the channel but doesn't start DMA. `_start` enables; `_stop` disables. This matters because the channel allocation is heavy (hundreds of KiB of PSRAM in theory; ~4 KiB in practice for our config) and shouldn't be paid every recording.
- **Convenience capture.** `mic_pdm_capture(dst, n_frames, timeout_ms)` -- starts if not running, reads N frames, restores prior state. The 90% caller pattern.
- **Mutex around lifecycle, not reads.** The IDF channel handle is single-consumer; we don't lock around `_read` to avoid contention on the hot path.

### What's intentionally NOT here

- **No VAD, no Speex, no SNR estimate.** The brief is explicit -- "capture only; preprocessing in Phase 2+".
- **No SD save in this driver.** The harness shows the pattern; an application that wants to record to disk composes mic_pdm + sdcard.
- **No 24-bit option.** 16-bit covers the chip's analog SNR; 24-bit would just inflate the DMA traffic for noise.
- **No stereo mode.** v7.2 has one mic. Configuring stereo and discarding the right slot would double DMA + halve effective rate -- worse on every axis.

## Implementation

### `components/mic_pdm/mic_pdm.h`

- Pins: `MIC_PDM_GPIO_CLK = GPIO_NUM_47`, `MIC_PDM_GPIO_DIN = GPIO_NUM_48`.
- Capture: 16000 Hz, 16-bit, 20 ms frames = 320 samples = 640 bytes, 6 DMA frames.
- Identity: `"PDM Mic"` / `"I2S0 PDM RX 16kHz/16b, GPIO47/48"`.
- Public API: `_init` / `_deinit` / `_start` / `_stop` / `_is_running` / `_read` / `_capture`.

### `components/mic_pdm/mic_pdm.c`

- File-static state: `s_lock` (mutex), `s_rx_chan` (`i2s_chan_handle_t`), `s_initialised`, `s_running`.
- `_init` -- `i2s_new_channel(NUM_0, MASTER)` then `i2s_channel_init_pdm_rx_mode()` with the slot mask set to LEFT.
- `_deinit` -- disable if running, `i2s_del_channel`.
- `_start` / `_stop` -- `i2s_channel_enable` / `_disable`.
- `_read` -- direct passthrough to `i2s_channel_read()` (no mutex; the channel handle is single-consumer).
- `_capture` -- preserves start/stop state, loops `_read` for `n_frames` frames, short-reads logged.

### `components/mic_pdm/CMakeLists.txt`

REQUIRES: `driver`, `esp_driver_i2s`, `esp_timer`, `freertos`.

### `test/test_mic_pdm.c`

Five-phase bench harness:

1. **Phase 1** -- `mic_pdm_init`, wall time.
2. **Phase 2** -- `mic_pdm_start`, wall time.
3. **Phase 3** -- 50-frame loop (= 1 s of audio). Each frame: read via `mic_pdm_read`, compute DC offset / peak |sample| / RMS. Log every 10th frame; aggregate report at the end.
4. **Phase 4** -- `mic_pdm_capture(25 frames)` convenience path into a static buffer.
5. **Phase 5** -- `_stop` + `_deinit`, stack high-water + heap.

Operator's eyeball test: quiet room -> RMS ~50-200; tap near the mic port -> peak jumps into the thousands.

| First-sample latency | DMA start -> first complete frame in HW | ~20-40 ms | one frame + IDF queue handoff |

### Per-operation cost

| Operation | Time | Notes |
|---|---|---|
| `mic_pdm_read` (one 640-byte frame, hot DMA) | < 50 us | memcpy from DMA buf to caller; the DMA work is parallel |
| `mic_pdm_read` blocking (no data ready) | up to timeout_ms | typical: 20 ms (one frame period) |
| `mic_pdm_capture` (25 frames) | ~500 ms | wall time = n_frames * 20 ms by definition |
| DC / peak / RMS analysis (320 samples) | < 100 us | trivial FP work; cosf-free |

### Memory

| Metric | Value | Notes |
|---|---|---|
| Driver state (mic_pdm.c statics) | ~24 bytes | mutex handle + chan handle + bools |
| DMA buffers (PSRAM) | 6 * 640 = 3.84 KiB | I2S_PDM RX ring |
| IDF channel state | ~1 KiB | descriptors + per-channel state |
| Test harness stack frame | ~4 KiB (CAPTURE_FRAMES_PHASE4 * 640 + locals) | static -- not on the task stack |

### Current draw estimate

| State | Current | Notes |
|---|---|---|
| Mic powered, channel idle (init only) | < 1 mA at the mic + a few mA in I2S0 quiescent | -- |
| Mic powered, capturing | ~1-2 mA at the mic + 5-10 mA in I2S0 active | dominates over the mic itself |
| Mic deinit (channel released) | ~0 mA at I2S0 | mic still draws its own quiescent off 3V3 |

### Notes

- **First-frame latency is the UX concern.** ~20-40 ms from `mic_pdm_start()` to first useful data. Any push-to-talk flow should account for this -- show the recording indicator before the first frame is meaningful.
- **PDM uses internal pull-up on CLK during init.** The clock isn't actively driven until the channel is enabled; the mic's clock-domain may take a few cycles to settle. The IDF driver handles this.
- **No software AGC.** A speech-recognition pipeline will probably want one in Phase 2+. The 16-bit headroom leaves room for ~30-40 dB digital gain before clipping.

## Defects discovered

- `[DEFECT-001] JP9 mismatch yields silent capture (all zeros) | hardware + components/mic_pdm/mic_pdm.c | LOW | If JP9 is moved to VDD (right channel) on a bench unit, the LEFT-slot configuration returns zeros forever. Symptom: every frame has RMS = 0. Driver cannot detect this; bench technician must verify JP9 visually. Disposition: documented; test harness flags it via "if RMS=0 across every frame, suspect JP9".`
- `[DEFECT-002] No mic VDD power gate | hardware-side | LOW | The mic stays powered as long as 3V3 is up; we cannot save its sub-mA quiescent during sleep. For a watch this is sub-noise vs the rest of the system. Disposition: noted; not a v1 problem.`
- `[DEFECT-003] PDM clock = ~640 kHz radiated | hardware/firmware | LOW | The PDM CLK at 16 kHz sample rate runs at 64x = 1.024 MHz nominal (clock multiple depends on IDF defaults). The line is short (chip-to-mic), no shield. If the radiated clock couples into the analog sensor lines (TMP117, MAX30101) it could show as a sub-mA DC offset. Bench-measure if anything looks off. Disposition: documented.`
- `[DEFECT-004] No software de-jitter / DC-blocker | components/mic_pdm/mic_pdm.c | MED | Raw 16-bit PDM-decoded samples have a non-zero DC offset (typical ~50-500 LSB) which speech pipelines must remove. The driver passes raw samples; downstream code is on its own. The test harness subtracts a per-frame DC estimate before computing RMS so the metric is meaningful. Disposition: by design -- chip layer is raw; preprocessing belongs in a higher module.`
- `[DEFECT-005] No frame-rate verification | test/test_mic_pdm.c | LOW | We log per-read wall time but never verify the wall-clock rate matches 50 Hz (one 20 ms frame). If the DMA is slipping (e.g. PLL not locked), we'd silently get the wrong sample rate. Disposition: a future variant of the test could compare wall time to expected.`
- `[DEFECT-006] Capture buffer in BSS, not heap | test/test_mic_pdm.c:buf | LOW | The 16 KB static buf consumes BSS; for the test app that's fine (no other big users). For production code that wants to record longer than a quick sample, the buffer should be malloc'd from PSRAM. Disposition: documented test-only design.`

## Open questions

1. **Where does mic policy live?** Today nothing calls `mic_pdm_start()`. The expected hook is a "voice command" or "voice memo" trigger -- a future UI module or a wake-word listener (Phase 2+).
2. **Should the mic auto-start on a button hold?** UX call. If yes, `boot_power.c`'s long-press handler routes to `mic_pdm_start()` and a buffer-to-SD pump.
3. **Sample-rate sanity.** The IDF v5 PDM driver clock config defaults are usually fine for 16 kHz; if the bench measurement shows a different effective rate, switch to explicit `clk_cfg` numbers and tune.
4. **Multi-frame burst -- one xQueueOverwrite or N frames?** The current driver returns per-frame. For a buffered VAD pipeline, the caller might prefer batch reads of 5-10 frames. Recommend: keep one-frame for now (simpler), batch in a wrapper layer when needed.
5. **Audio + I2C bus 1 coexistence.** The mic is on I2S0 (own DMA channel). It should NOT contend with I2C bus 1; v7.2 confirms the pins are distinct. Sanity-check on the bench.
