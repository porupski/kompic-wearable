## Summary

The VEML6030 replaces the v5 BH1750 at the ambient-light slot. It is on I2C bus 1 at 0x10, runs continuously, and drives the watch's auto-brightness UX through `broker_light_data_t.auto_brightness`. The chip is substantially more capable than the BH1750: programmable gain (1/8x to 2x), programmable integration time (25 ms to 800 ms), 16-bit ADC, and an interrupt window register that the firmware ignores in v1 but could exploit later. The cost of that capability is complexity in the driver: lux is `raw_count × resolution(gain, IT)`, the resolution table has to live in the code, and auto-ranging has to be implemented to avoid saturating in bright sunlight or losing resolution in dim rooms.

The auto-brightness logic above the driver carries forward unchanged: `broker_light_data_t.auto_brightness` is a 0-100% derived value, computed inside `task_light_fn` and consumed by `light_tile.c` which writes it to the display backlight via `backlight_set_brightness()`. The lux-to-percentage curve is **recomputed** for the wider dynamic range -- a clamped log10 mapping rather than the linear-ish curve the BH1750 used.

Today's deliverable: replaced driver code with auto-ranging state machine, renamed-and-resized tile, perceptual log10 brightness curve, standalone test harness with curve sanity dump.

## Code audit

### What dies

`components/veml6030/bh1750.{c,h}` -- deleted. The BH1750 had:
- A command-byte protocol (no register addresses): one-byte writes selected operating mode; reads returned 2 bytes directly.
- A single fixed lux conversion (1.2 lx/count at H_RES mode).
- No gain / IT controls.
- No interrupt window.

All of that is replaced.

### What carries forward

- **`broker_light_data_t` shape, field-for-field.** Defined now in `veml6030.h`. Same `lux / auto_brightness / last_update_ms / enabled`. `light_tile.c` and any other consumer build with only an include-path + identity-call rename.
- **The auto-brightness contract.** `task_light_fn` computes `broker.auto_brightness` from `broker.lux`; `light_tile.c` applies it via `backlight_set_brightness()` when "Auto Brightness" is enabled. The chain is chip-independent.
- **Task entry point name.** `task_light_fn` is what's wired in `boot_tasks.c`. Kept verbatim.
- **EMA filter pattern.** BH1750 already used a single-pole IIR. Same idea here with α = 0.85, seeded on first valid read.

### What's new

- **16-bit register protocol.** Writes are 3 bytes (reg, LSB, MSB). Reads are 1 byte address + 2 data bytes LSB-first.
- **Auto-ranging state machine.** Start at gain=1/4x, IT=100 ms. Step down to 1/8x on saturation (raw >= 0xFFF0). Step up to 1x / 2x on underflow (raw <= 100). One sample skipped after each range change (the chip needs one full IT to settle).
- **Resolution table** -- `veml6030_resolution_lx_per_count[4][6]` from Vishay AN84367 §Table 5. Driver looks up the lx/count factor by current (gain, IT) tuple.
- **Lux-to-brightness curve recomputed.** Clamped `log10`-based mapping from [1 lux, 5000 lux] -> [5%, 100%]. The BH1750 era had a narrower dynamic range; the new curve handles 0 lux (dark sleeve) to 20+ klux (full sun) without clipping the response.
- **Public `veml6030_lux_to_brightness()`** -- the curve is now a pure function with a public symbol so tests can exercise it.

## Implementation

### `components/veml6030/veml6030.h`

- Wire constants `VEML6030_ADDR = 0x10`.
- Register-address `#define`s for ALS_CONF (0x00), ALS_WH (0x01), ALS_WL (0x02), POWER_SAVING (0x03), ALS (0x04), WHITE (0x05), ALS_INT (0x06). All 16-bit, LSB first.
- `CONF_*` bit/mask defines for ALS_SD, ALS_INT_EN, gain (GAIN_1/2/1_8/1_4), IT (25/50/100/200/400/800 ms).
- `veml6030_gain_t` / `veml6030_it_t` enums for the auto-range state machine.
- `veml6030_resolution_lx_per_count[4][6]` (extern const) -- the lx/count table.
- `broker_light_data_t` -- field-for-field preserved.
- Public API: `init / deinit`, `task_light_fn`, `read_raw`, `set_range`, `current_resolution`, `get_range`, `lux_to_brightness`.

### `components/veml6030/veml6030.c`

- `write_reg16 / read_reg16` -- 16-bit LSB-first wrappers around the modern I2C APIs.
- `veml6030_init`: power-down first (Vishay-recommended), wait 5 ms, then set range (gain=1/4x, IT=100 ms, ALS_SD=0), wait 150 ms for first integration.
- `veml6030_set_range`: writes ALS_CONF, updates `s_gain` + `s_it`, sets `s_skip_next` so the auto-range loop ignores the next stale read.
- `auto_range_step`: decides whether to step up or down based on `raw` vs `SAT_HIGH (0xFFF0)` / `SAT_LOW (100)`. Single-step per cycle so we don't thrash.
- `veml6030_lux_to_brightness`: clamped log10 -- `t = (log10(lux+1) - 0) / (log10(5000) - 0)`, then `pct = 5 + 95*t`, clamped to [5, 100].
- `task_light_fn`: 2 Hz poll, read-before-write, auto-range step under mutex, give mutex, run EMA, compute brightness, broker write.

### `components/veml6030/light_tile.c`

- `#include "bh1750.h"` -> `#include "veml6030.h"`.
- Three identity-function call sites swapped: `bh1750_get_chip_name/desc` -> `veml6030_get_chip_name/desc` (header text + log).
- Layout constants extracted to named locals; y positions scaled from 240x280 to 410x502.
- Switch widths bumped from 44×22 to 60×30. Slider height 14 -> 18.
- All the carried-forward logic in this tile (auto-brightness toggle, brightness slider, theme buttons, blue-light filter) is untouched.

### `components/veml6030/CMakeLists.txt`

- `SRCS bh1750.c` -> `veml6030.c`.

### `test/test_veml6030.c`

Standalone harness:
1. I2C bus 1 init at 400 kHz.
2. `veml6030_init` (gain=1/4x, IT=100 ms).
3. 10 raw reads at 500 ms spacing; print raw, lux, current gain/IT, per-read dt, and "[RANGE CHANGED]" marker if auto-range stepped.
4. Lux-to-brightness curve sanity: print `veml6030_lux_to_brightness()` at 0, 1, 10, 50, 200, 1000, 5000, 20000 lux. Useful for eyeballing whether the curve feels right before bench.

| State | Current | Notes |

### Notes

- **Auto-ranging is the load-bearing complexity.** A bug here turns the auto-brightness curve into a step function (gain hops cause visible brightness flickers). The EMA + the one-cycle skip-after-change should hide most of the discontinuity.
- **The 500 ms poll matches the 100 ms IT comfortably** (5× margin). At lower IT (25/50 ms) the chip could supply data faster, but ambient light doesn't change at >2 Hz in any normal scenario.
- **5% floor on the brightness curve is deliberate.** Dark room or covered sensor should still let the watch be legible.

## Defects discovered

- `[DEFECT-001] Gain / IT bit layouts in ALS_CONF tagged [DSV] | components/veml6030/veml6030.h | HIGH | The CONF_GAIN_* and CONF_IT_* values are taken from Vishay AN84367 conventions. They have NOT been verified against the datasheet in this session. Disposition: confirm before bench bring-up. If wrong, the auto-range state machine will write the chip into an invalid mode and lux readings will be nonsense.`
- `[DEFECT-002] Resolution table tied to gain/IT enum order | components/veml6030/veml6030.c:veml6030_resolution_lx_per_count | MED | The table is indexed [gain][it] with assumed ordering (gain rows: 1/8x, 1/4x, 1x, 2x; IT cols: 25, 50, 100, 200, 400, 800). If the gain enum's ordering is changed without updating the table, lux will be silently wrong. Disposition: add a unit test that cross-checks veml6030_current_resolution() at every (gain, IT) combination once datasheet values are confirmed.`
- `[DEFECT-003] Lux-to-brightness curve is heuristic | components/veml6030/veml6030.c:veml6030_lux_to_brightness | LOW | The 1-5000 lux clamped log10 mapping is a perceptual approximation, not a measured response. Bench data with the actual panel + the actual case window will tell us whether 100% feels right at 5000 lux or if we should push the ceiling higher. Disposition: tune after first bench observation.`
- `[DEFECT-004] Auto-range may oscillate at boundaries | components/veml6030/veml6030.c:auto_range_step | MED | The state machine has no hysteresis. A lux value sitting right at the gain-change threshold could flip back and forth on each poll. The 500 ms poll + EMA hides this in the broker payload, but not in the raw register state. Disposition: add hysteresis (one threshold to step up, a different one to step down) if oscillation is observed on bench.`
- `[DEFECT-005] s_skip_next not protected against multi-task reentry | components/veml6030/veml6030.c | LOW | The task is single-instance pinned to Core 0; nothing else should call into the auto-range path. If a future test path calls veml6030_set_range from another task while task_light_fn is running, the skip flag could race. Disposition: documentation-only; the broker design enforces single-writer.`
- `[DEFECT-006] g_i2c_mutex via extern | components/veml6030/veml6030.c | LOW (pre-existing) | Same as elsewhere.`

## Open questions

1. **Datasheet review.** Like the rest of Phase 2, a single Vishay AN84367 review collapses [DEFECT-001] and locks in [DEFECT-002].
2. **Interrupt window usage.** VEML6030 has ALS_WH / ALS_WL threshold registers and an ALS_INT_EN bit; with INT routed, the chip can wake the ESP only on significant lux changes (good for power-save). v7.2 does NOT route the INT pin, but `ALS_INT` register-polling could still flag a wake event. Question: is this worth the complexity vs just polling at 2 Hz? Recommend: stick with polling for v1.
3. **WHITE channel usage.** VEML6030 also exposes a broadband "WHITE" measurement at REG 0x05. Useful for distinguishing artificial vs natural light or for IR rejection. Not used today. Question: any UI / fusion benefit, or is this future-noise?
4. **Backlight gating during charge.** Should auto-brightness be suppressed (or capped lower) during USB charging? Brightness × charge current × heat is one of those second-order interactions worth flagging. Recommend: not in v1; revisit if thermal or perceived-flicker issues show up on bench.
5. **Calibration at boot.** Should we take a single low-light "dark" reading at boot to characterise the sensor floor (the optical window may not be perfectly opaque to internal LED bleed)? Recommend: not in v1; the EMA absorbs small DC offsets.
