## Summary

The BME688 replaces the v5 BME280 at the environment slot. Same I2C address (0x76), same footprint, but now with a gas sensor: a hot-plate heater (programmable temperature + duration) drives a metal-oxide film whose resistance correlates with VOC concentration. Bosch publishes a managed component (`boschsensortec/bme68x_sensor_api`) that handles the calibration math + heater profile; this driver is a thin wrapper around it.

The broker_env_data_t shape carries forward verbatim from the bme280_drv era. `gas_resistance_ohm` and `gas_valid` were already in the struct (always 0.0 / false on the BME280 stub); the BME688 driver populates them for real.

The big behavioural change from the BME280 driver is the **mutex hold time** in `task_env_fn`. With the gas heater enabled, one forced-mode cycle takes ~200 ms wall time (vs ~40 ms on the BME280). The task runs at prio 2 -- below touch and button -- but it does hold the I2C bus 1 mutex for that whole window. We measure the worst-case hold in the test harness and flag if it ever crosses 250 ms.

Today's deliverable: replaced driver code (Bosch BME68x wrapper, heater profile fixed at 320 C / 150 ms), env_tile identity rename, idf_component.yml for managed-component pull, standalone test harness with heater-blocking measurement.

Single rail; no V_IO split. The chip is happy on a single 3V3.

### Gas-sensor heater

The BME688 has an on-chip resistive heater that drives the metal-oxide film for VOC sensing. Heater profile is programmable: target temperature (0-400 C, set via heater-resistance LUT computed by the Bosch library) and pulse duration (0-4032 ms). For VOC discrimination Bosch recommends 320 C / 150 ms as a single-set-point starting profile. The heater current draw is significant -- datasheet typical is ~12 mA during the pulse, so the duty cycle matters for power.

## Code audit

### What dies

`components/bme688/bme280_drv.{c,h}` -- deleted. The BME280 driver was a thin wrapper around the legacy Bosch BME280 library (a different chip family, not API-compatible with BME68x). All of it goes.

The old `CMakeLists.txt` referenced a `bme280.c` source file that wasn't actually present in the directory (the vendored Bosch BME280 lib was supposed to live in `components/bme280_vendor/` but doesn't exist in this tree). The new build pulls the Bosch BME68x lib via the managed component instead -- cleaner.

### What carries forward

- **`broker_env_data_t` shape, field-for-field.** Defined now in `bme688_drv.h`. `env_tile.c` builds with only the include + identity-call rename (`bme280_get_chip_name/desc` -> `bme688_get_chip_name/desc`).
- **Hypsometric altitude formula.** `bme688_pressure_to_altitude()` is the exact same arithmetic as `bme280_pressure_to_altitude()`. Chip-independent.
- **Read-before-write pattern.** `home_ref_*` is Core-1-written; the task reads-then-writes so it doesn't clobber.
- **Cross-driver event firing.** `XD_EVENT_ENV_UPDATED` fires on every successful read, same as before. Fusion baro-altitude channel listens on this.
- **Task structure.** `task_env_fn` -- 0.5 Hz poll, prio 2, mutex around the whole forced-mode cycle, 500 ms mutex timeout.

### What's new

- **Bosch BME68x library integration.** Managed-component pull via `idf_component.yml` (boschsensortec/bme68x_sensor_api ^4.4.8). Replaces the vendored BME280 lib.
- **`bme68x_dev` configuration.** I2C callbacks wired to ESP-IDF i2c master via the same cmd-link pattern as bme280_drv. Bosch's `amb_temp` field is seeded at 25 C and updated on every read so the heater gas-resistance compensation has a sane reference.
- **Heater profile.** Single-set-point: 320 C / 150 ms. Tuning is Phase 4+ work.
- **`gas_resistance_ohm` + `gas_valid` populated for real.** The struct already had these fields (always 0.0 / false on BME280); now they reflect the chip's measurement, gated on the chip's own `BME68X_GASM_VALID_MSK | BME68X_HEAT_STAB_MSK` status flags.
- **Measurement-time computation.** `bme68x_get_meas_dur()` + heater duration is summed and used as the delay between trigger and read. Replaces the BME280's fixed 40 ms wait.

## Implementation

### `components/bme688/bme688_drv.h`

- `broker_env_data_t` -- field-for-field preserved.
- `BROKER_ENV_TIMEOUT_MS` = 10000 (5x the 2 s poll period).
- Identity strings `"BME688"` / `"T/H/P/Gas"`.
- Public API: `bme688_drv_init / bme688_read_forced / bme688_pressure_to_altitude / task_env_fn`.

### `components/bme688/bme688_drv.c`

- Bosch I2C callbacks `bme_i2c_read / bme_i2c_write / bme_delay_us` -- same shape as the bme280_drv era, returning Bosch-defined error codes (BME68X_INTF_RET_SUCCESS / BME68X_E_COM_FAIL).
- `bme688_drv_init`:
  - Wires the Bosch `struct bme68x_dev` with I2C callbacks + `amb_temp` = 25.
  - Calls `bme68x_init` (chip-id probe + calibration data load).
  - Configures `os_temp = os_pres = os_hum = 1x`, `filter = OFF`, `odr = NONE` (forced mode).
  - Programs the heater profile: 320 C / 150 ms.
- `bme688_read_forced`:
  - Sets forced mode (one-shot trigger).
  - Waits `bme68x_get_meas_dur() + heater_dur` -- chip-computed.
  - Calls `bme68x_get_data` and decodes the result.
  - Sets `gas_valid` only if both `BME68X_GASM_VALID_MSK` and `BME68X_HEAT_STAB_MSK` bits are set in `data.status`.
- `task_env_fn` -- shape preserved from bme280_drv era. Mutex hold time is the only behavioural delta.

### `components/bme688/env_tile.{c,h}`

- `#include "bme280.h"` -> `#include "bme688_drv.h"`.
- Identity-call swap: `bme280_get_chip_name/desc` -> `bme688_get_chip_name/desc` (2 sites: header label + init log).
- Layout unchanged. Zero Height button + delta height row + apply_theme fan-out -- all chip-independent.

### `components/bme688/idf_component.yml`

New file. Pulls `boschsensortec/bme68x_sensor_api ^4.4.8` at build time. This is the per-component manifest pattern; the project root `main/idf_component.yml` is unaffected.

### `components/bme688/CMakeLists.txt`

- `SRCS bme280.c bme280_drv.c env_tile.c` -> `SRCS bme688_drv.c env_tile.c`. The non-existent `bme280.c` reference is gone. The Bosch BME68x lib comes in via the managed component, not via SRCS.

### `test/test_bme688.c`

Five-phase standalone harness:

1. **Phase 1** -- I2C bus 1 init at 400 kHz on GPIO1/2.
2. **Phase 2** -- `bme688_drv_init` (chip-id probe + Bosch lib config + heater profile).
3. **Phase 3** -- 5 forced-mode reads at 2 s spacing. Logs T/P/H, altitude, gas resistance, gas_valid flag, per-read I2C wire time, and total mutex hold time.
4. **Phase 4** -- Worst-case mutex hold quantified across the 5 cycles, warns above 250 ms.
5. **Phase 5** -- Stack high-water + heap.

|---|---|---|

### Current draw estimate

| State | Current | Notes |
|---|---|---|
| Idle (sleep mode, between measurements) | < 1 uA | between forced-mode cycles |
| Active T/P/H only (OSR=1x) | ~3 mA | brief during the measurement window |
| Heater pulse (320 C, 150 ms) | ~12 mA peak, ~1 mA average | dominates the energy budget for this sensor |
| Average over 2 s poll period | ~1 mA | (150 ms heater @ ~12 mA + 1850 ms sleep @ < 1 uA) / 2 s |

### Notes

- **Heater mutex hold is the single biggest design constraint.** 200 ms is 5% of a 4 s wall clock, but if touch / button polling lives on the same I2C bus (it does -- CST9217 + BQ25619 are bus 1 too), starvation matters. The 500 ms `xSemaphoreTake` timeout in `task_env_fn` is generous; if it ever fires we'll know.
- **The 2 s poll is bounded by the heater duty cycle.** Faster polling means hotter chip and more current. Slower polling means less timely VOC data. 2 s is the Bosch-recommended starting point for VOC sniffing.
- **Gas resistance is in ohms (typically tens of kohms to MOhms).** Higher resistance = cleaner air, lower = more VOCs. The absolute number is not meaningful without baseline calibration (Bosch's BSEC library does this). We expose the raw value today and let the UI / fusion layer figure out what to do with it.

## Defects discovered

- `[DEFECT-001] Heater profile (320 C / 150 ms) is a Bosch default, not measured [DSV] | components/bme688/bme688_drv.c:HEATER_TEMP_C/HEATER_DURATION_MS | MED | The plateau temperature and duration come from Bosch application notes for "VOC sniffing". They have not been characterised against any real volatile in our enclosure. If the chip plateaus visibly below 320 C (e.g. due to thermal coupling to the case), gas readings will be unreliable. Disposition: bench-verify with a known VOC source (isopropyl, ethanol); tune profile in Phase 4+.`
- `[DEFECT-002] I2C mutex held for the full ~200 ms forced-mode cycle | components/bme688/bme688_drv.c:bme688_read_forced | HIGH | The heater pulse is on-chip; we could in principle release the mutex during the wait. We don't, because the Bosch library is stateful and re-entering it from another task mid-cycle is undefined. As a result the touch + button drivers on the same bus (CST9217 + BQ25619 + LIS3MDL + LSM6DSV16X + VEML6030) are blocked for 200 ms every 2 s -- 10% bus blocking duty cycle. Disposition: bench-verify that touch latency doesn't degrade; if it does, refactor to split trigger / wait / read into three separate mutex acquisitions.`
- `[DEFECT-003] gas_valid flag check may be too strict | components/bme688/bme688_drv.c:bme688_read_forced | LOW | We require BOTH GASM_VALID_MSK and HEAT_STAB_MSK before declaring gas_valid. Bosch's BSEC library uses a looser policy (heat_stab takes a few cycles to assert after a heater profile change). On the first 2-3 reads after boot, gas_valid will be false even with a working chip. Disposition: documented behaviour; if the UI looks dead at boot, relax to GASM_VALID_MSK only.`
- `[DEFECT-004] amb_temp seeded at 25 C | components/bme688/bme688_drv.c:bme688_drv_init | LOW | Before the first measurement, the heater gas-resistance compensation uses 25 C as the ambient. If the watch is bring-up'd in a hot car or cold mountain, the very first gas reading is off. Self-corrects on the second cycle (s_dev.amb_temp updates with the measured temperature). Disposition: cosmetic; not worth fixing.`
- `[DEFECT-005] Bosch managed component version pinned to ^4.4.8 | components/bme688/idf_component.yml | LOW | The Bosch BME68x lib evolves; pinning to ^4.4.8 follows the principle but means we have no idea what minor changes ship. Disposition: revisit on first failed CI build; don't proactively bump.`
- `[DEFECT-006] g_i2c_mutex via extern | components/bme688/bme688_drv.c | LOW (pre-existing) | Same pattern as cst9217 / bq25619 / etc. Resolves when boot_hw_init.h lands.`

## Open questions

1. **BSEC vs raw gas resistance.** Bosch ships a higher-level library (BSEC) that consumes the raw gas resistance and outputs an IAQ index (0-500 indoor air quality score). BSEC is closed-source-binary, must be linked separately, and has its own licensing. Question: do we want IAQ in v1, or is raw gas resistance enough? Recommend: ship raw only; revisit when fusion-layer work starts.
2. **Heater profile per use-case.** Multiple set-points can be programmed (BME688 supports up to 10), each measured in a sequence. Useful for distinguishing VOC types. Question: worth the complexity for a watch? Recommend: no, single set-point is plenty.
3. **Power budget for the heater.** ~1 mA average draw is significant for a watch (every mA shaves ~10 hours off battery). Question: should the heater pulse fire only on user request (button press), or continuously every 2 s? Recommend: continuous for v1, expose a power-save toggle in v2.
4. **Altitude reference (P0 = 1013.25 hPa).** Same hardcoded sea-level pressure as the BME280 era. Question: should the watch pull current P0 from GPS or weather API? Recommend: not in v1.
5. **Cross-bus contention measurement.** Once Phase 3 is on bench, instrument touch / button latency during a heater cycle to confirm we're not starving them. The 500 ms mutex timeout in `task_env_fn` is the safety net; if it fires, we have a problem.
