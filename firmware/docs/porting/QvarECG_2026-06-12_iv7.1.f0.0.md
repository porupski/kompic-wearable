## Summary

New module -- no v5 carryforward. Mk I exposes a single-lead ECG-class signal via the **LSM6DSV16X's embedded Analog-Hub / Qvar block**. v7.2 §CASE describes the two electrodes:

- **Qvar1 (skin):** pogo pin through a printed sleeve in the case bottom, contacts a pad on the daughter board.
- **Qvar2 (crown):** pogo pin riding a groove on the crown shaft -- contacts a finger from the user's free hand when the crown is touched.

The Qvar channel is a special-purpose high-impedance ADC built into the LSM6DSV16X die. It shares the IMU's I2C address (0x6B on bus 1), its register space, and -- critically -- its bus mutex (`g_i2c_mutex`). This driver is therefore **not standalone**; it depends on the LSM6DSV16X chip being brought up first by the IMU driver.

Three deliverables:

1. **`components/qvar_ecg/qvar_ecg.{c,h}`** -- enables `CTRL7.AH_QVAR_EN`, selects the 235 MOhm ZIN setting, exposes single-sample + capture + contact-detection APIs. Shares `g_i2c_mutex`.
2. **`components/qvar_ecg/ecg_tile.{c,h}`** -- placeholder LVGL tile per the brief ("optional: display ECG waveform strip"). Real rendering deferred to Phase 2+ once the broker payload + tile registry settle.
3. **`test/test_qvar_ecg.c`** -- 8-phase bench harness that brings up bus 1 + the chip's WHO_AM_I + reset, then init / single-sample / 1 s capture (240 Hz) / contact heuristic / deinit. Operator's eyeball: bare electrodes -> "no contact" + railed values; wet finger across both pogo pins -> contact + ~ECG-shaped waveform.

The register addresses + bit positions are tagged `[DSV]` (datasheet-verify) throughout per the project's convention; once bench bring-up confirms the LSM6DSV16X die behaviour, the tags can be retired.

## Hardware spec from v7.2

### Bus + chip

| Item | Value |
|---|---|
| Chip | LSM6DSV16X (same die as IMU) |
| Bus | I2C bus 1 (GPIO1 SDA, GPIO2 SCL, 400 kHz) |
| I2C addr | 0x6B |
| Shared mutex | `g_i2c_mutex` (from lsm6dsv16x.c) |

### Qvar registers (LSM6DSV16X datasheet, [DSV])

| Reg | Addr | Bits | Purpose |
|---|---|---|---|
| CTRL7         | 0x16 | bit 0 `AH_QVAR_EN`, bits 4-5 ZIN selection | enable + impedance |
| OUT_AH_QVAR_L | 0x3A | -- | low byte of 16-bit signed sample |
| OUT_AH_QVAR_H | 0x3B | -- | high byte |

ZIN settings (bits 5:4 of CTRL7) per the datasheet:
| Code | Impedance | Comment |
|---|---|---|
| 00 | 2.4 GOhm | highest impedance -- lowest current, highest input-referred noise |
| 01 | 730 MOhm | -- |
| 10 | 300 MOhm | -- |
| 11 | 235 MOhm | lowest impedance -- best noise floor; **driver default** |

### Electrodes (v7.2 §CASE)

| Pin | Location | Role |
|---|---|---|
| Qvar1 | pogo pin through case bottom, touches a daughter-board pad | skin contact (under-wrist) |
| Qvar2 | pogo pin riding the crown shaft groove | finger contact (free hand) |

The two pogo pins form the differential input to the Qvar channel; v7.2 §CASE flags both as "Qvar electrodes". The actual signal is the charge differential between them.

## Code audit

### What dies

Nothing -- new module.

### What's new

- **Read-before-write on CTRL7.** We preserve any bits the IMU driver may have set in CTRL7 (Blueprint 4 §3 read-before-write). We only own bits 0 (AH_QVAR_EN) and 5:4 (ZIN).
- **ZIN = 235 MOhm.** Lowest impedance; sees pogo-pin contact as a clean low-impedance source. The 2.4 GOhm option is the right pick when the electrodes are non-contact (capacitive sensing through clothing), which is not the Mk I model.
- **Direct OUT_AH_QVAR_L/H read.** 2-byte burst, no FIFO. ECG-class signals fit comfortably in 240 Hz polled mode; FIFO would be over-engineering for v1.
- **Cadence via `vTaskDelayUntil`.** Drift-free 240 Hz sample loop in `qvar_ecg_capture`. The I2C work (< 1 ms at 400 kHz) is < 25% of the 4.17 ms sample period.
- **Contact heuristic.** Capture a 200 ms window; if the majority of samples peg above ±30000 (railed), report "no contact". A floating high-impedance input rails naturally; a real biopotential signal stays well under saturation.

### What's intentionally NOT here

- **No init of the LSM6DSV16X chip.** That's the IMU driver's job. We assume `lsm6dsv16x_init()` ran first; if `g_i2c_mutex` is NULL when we try to init, we bail with `ESP_ERR_INVALID_STATE`.
- **No broker write.** ECG samples are on-demand and bursty (a 30 s strip is 7200 samples = 14 KiB); the broker is the wrong place. Callers can save to SD via `sdcard.c` or stream over BLE.
- **No R-peak detection / HRV math.** Phase 2+ territory.
- **No waveform rendering.** `ecg_tile.c` is a stub per brief.

## Implementation

### `components/qvar_ecg/qvar_ecg.h`

- Bus + chip: `QVAR_I2C_PORT = I2C_NUM_0` (bus 1), `QVAR_I2C_ADDR = 0x6B`.
- Registers: `QVAR_REG_CTRL7 = 0x16`, `QVAR_REG_OUT_AH_QVAR_L/H = 0x3A/0x3B`, tagged `[DSV]`.
- Bit defines: `QVAR_CTRL7_AH_QVAR_EN`, `QVAR_CTRL7_ZIN_235M`, tagged `[DSV]`.
- Capture: `QVAR_ODR_HZ = 240`, `QVAR_DEFAULT_CAPTURE_SEC = 1`, `QVAR_DEFAULT_CAPTURE_N = 240`.
- Contact: `QVAR_FLOAT_THRESHOLD = 30000`, `QVAR_CONTACT_WINDOW_SAMPLES = 48` (200 ms).
- Identity: `"QvarECG"` / `"LSM6DSV16X AH_QVAR @ 240 Hz"`.
- Public API: `_init` / `_deinit` / `_read_sample` / `_capture` / `_has_contact`.

### `components/qvar_ecg/qvar_ecg.c`

- `extern SemaphoreHandle_t g_i2c_mutex` -- same pattern as lsm6dsv16x.c (no `boot_hw_init.h` yet; the encoder + DRV2605 docs flag the same).
- `write_reg` / `read_reg` / `read_regs` -- file-static I2C primitives that take a port and assume the mutex is held.
- `qvar_ecg_init(i2c_num)` -- takes mutex, reads CTRL7, ORs in AH_QVAR_EN + ZIN_235M, writes back, releases mutex, 10 ms settle, sets `s_enabled`.
- `qvar_ecg_deinit()` -- takes mutex, clears AH_QVAR_EN, writes back.
- `qvar_ecg_read_sample(int16_t *)` -- 2-byte burst from 0x3A.
- `qvar_ecg_capture(dst, n, out_n)` -- `vTaskDelayUntil` loop at 240 Hz.
- `qvar_ecg_has_contact()` -- 200 ms window, majority-rail check.

### `components/qvar_ecg/ecg_tile.{c,h}`

Placeholder per brief. `ecg_tile_init()` logs and returns OK. Real LVGL widget lands when:
- A `broker_ecg_data_t` payload type is defined (Tier 1, mutex-guarded).
- The tile registry pattern in `lvgl_ui` is firm.

### `components/qvar_ecg/CMakeLists.txt`

REQUIRES: `driver`, `esp_timer`, `freertos`. No `lsm6dsv16x` requirement -- we share state through `g_i2c_mutex` only, not direct symbols.

### `test/test_qvar_ecg.c`

Self-contained: brings up bus 1, runs a minimal IMU probe (WHO_AM_I + soft reset), then init / 5 sanity samples / 1 s capture at 240 Hz / contact heuristic / deinit. Defines its own `g_i2c_mutex` so the test app stands alone without the rest of the firmware.

### Boot-time cost

| Phase | Component | Time | Method |

| Operation | Time | Notes |

| Per-sample loop overhead | < 1 ms | vTaskDelayUntil + qvar_ecg_read_sample |
| `qvar_ecg_has_contact` (200 ms window) | ~200 ms wall | drives the capture path |

### Memory

| Metric | Value | Notes |
|---|---|---|
| Driver state | ~8 bytes | s_enabled + s_i2c_num |
| 1 s capture trace | 480 bytes | int16_t * 240 |
| `qvar_ecg_has_contact` window | 96 bytes | int16_t * 48 (on stack) |

### Current draw estimate

| State | Current | Notes |
|---|---|---|
| Qvar disabled (CTRL7.AH_QVAR_EN=0) | 0 mA additional | chip itself runs normally |
| Qvar enabled, ZIN=235 MOhm | < 0.5 mA additional [DSV] | the highest-current ZIN setting |
| Qvar enabled, ZIN=2.4 GOhm | sub-uA [DSV] | -- |
| I2C polling overhead at 240 Hz | well under 1 mA | 240 * ~50 us of bus activity per second |

### Notes

- **240 Hz is conservative for cardiology.** ECG diagnostic guidelines require 500 Hz; clinical wearables are happy at 100-250 Hz. 240 Hz is the LSM6DSV16X's Qvar ODR that's high enough to be useful and low enough to be sustained by I2C polling. If a future use case wants 480/960 Hz, the driver can be bumped -- but the I2C bus 1 sharing with TMP117 / VEML6030 / MAX30101 / etc. starts to bite.
- **Calibration is a future problem.** Raw Qvar units are not millivolts -- they're "LSB of the AH channel". A real ECG display needs a per-unit gain calibration against a known signal (or just normalise the strip). v1 ships uncalibrated.
- **Contact detection is a heuristic.** The "majority rails -> no contact" rule works in the typical case but a slow drift through the threshold could flap. If false-positive contact detection becomes an issue, add hysteresis.

## Defects discovered

- `[DEFECT-001] Register addresses tagged [DSV] | components/qvar_ecg/qvar_ecg.h | MED | OUT_AH_QVAR_L/H at 0x3A/0x3B and CTRL7 bit definitions are family-conventional but not bench-verified for the Mk I unit. On the bench, read CTRL7 / write a known pattern / read back; if the chip doesn't echo, the addresses are wrong. Disposition: verify on first hardware unit.`
- `[DEFECT-002] g_i2c_mutex extern depends on lsm6dsv16x driver having run | components/qvar_ecg/qvar_ecg.c | MED | If qvar_ecg_init() is called before lsm6dsv16x_init() in production, g_i2c_mutex is NULL and we bail. Disposition: documented ordering requirement; bring-up code must enforce.`
- `[DEFECT-003] FUNC_CFG_ACCESS page select not exercised | components/qvar_ecg/qvar_ecg.h:QVAR_REG_FUNC_CFG_ACCESS | LOW | Some LSM6DSV16X Qvar config bits live on the embedded-function page (bank-switch via REG 0x01). The driver enables Qvar via the main-page CTRL7 only; advanced Qvar features (filter cutoff, FIFO routing) need the embedded-page sequence. Disposition: defer until a downstream consumer needs them.`
- `[DEFECT-004] qvar_ecg_capture holds the timing loop in caller's task | components/qvar_ecg/qvar_ecg.c | LOW | The capture is synchronous -- the caller's task blocks for the full duration. For UI integration, the caller should run capture on a dedicated worker task. Disposition: documented; tile-layer concern.`
- `[DEFECT-005] No FIFO mode, no DRDY interrupt | components/qvar_ecg/qvar_ecg.c | LOW | Polled-only. For a high-rate trace (480/960 Hz) the polling cadence will jitter under bus 1 contention. The DRDY-driven path would be cleaner but needs an INT line, which v7.2 does not route for the LSM6DSV16X's secondary INT pins. Disposition: limited by hardware.`
- `[DEFECT-006] Contact heuristic could flap on slow drift | components/qvar_ecg/qvar_ecg.c:qvar_ecg_has_contact | LOW | The majority-rail test has no hysteresis. A signal slowly drifting through the threshold could oscillate the contact bit. Tile-layer should debounce if it ever drives UI state from this. Disposition: documented.`
- `[DEFECT-007] ecg_tile is a stub | components/qvar_ecg/ecg_tile.c | LOW | By design -- brief says the tile is "optional: Phase 2+". The placeholder satisfies the deliverable list and gives the future tile a place to land. Disposition: documented Phase 2+ work.`

## Open questions

1. **Verified register map?** The Qvar register addresses ([DSV] tags throughout) need bench confirmation. Once one capture works end-to-end, the tags retire.
2. **Co-existence with the IMU's poll loop.** The IMU runs at 20 ms cadence and grabs the mutex briefly. The Qvar capture grabs the mutex every 4.17 ms. Will the IMU starve / lag? Recommend: capture session is foreground UX (user is looking at the ECG screen), the IMU poll can slip a frame or two without consequence. Bench-validate.
3. **What's the calibration story?** Raw Qvar samples have an arbitrary gain. For a meaningful "mV" axis on the UI strip, the firmware needs a per-unit cal step or a normalised display. UX call.
4. **Filter cutoff?** The Qvar block has internal low-pass options. v1 leaves the default; if the strip is too noisy, add a config bit.
5. **Storage policy.** Should ECG traces be auto-saved to `/sd`? Recommend: NO by default (privacy), opt-in via a UI toggle.
