## Summary

NVS-persistence layer: magnetometer hard-iron calibration, GPS last-known date
(RTC cold-boot seed), height reference (barometric zero), UI settings async
save queue + task. This is the only Phase 6 module that touches flash directly.

Battery NVS keys formerly served by `app_battery.c` are out of scope — the v7.2
BQ25619 driver owns battery state, and the `"battery"` NVS namespace is currently
unused (see DEFECT-002 below).

## Pinout / bus from v7.2

N/A. Internal flash via the standard ESP-IDF NVS partition.

## Code audit (v5 → v7.2)

- `CMakeLists.txt`: dropped `app_battery.c` from SRCS (file does not exist).
  Dropped `driver` and `esp_adc` from REQUIRES (BQ25619 reads battery via I2C
  now, not ADC). Kept `nvs_flash`, `data_broker` (for `ui_settings_t` consumers),
  `lvgl_ui` (for the `ui_settings_t` typedef itself, which lives in
  `lvgl_ui/ui_broker.h`).
- Header comment: `"battery"` namespace ownership note rewritten to point at
  `components/bq25619/` and cross-reference the broker porting doc.
- Mag-cal API (`app_nvs_save_mag_calibration`, `..._load_mag_calibration`,
  `..._clear_mag_calibration`) carries forward unchanged — LIS3MDLTR stores
  hard-iron offsets in the same X/Y/avg shape as the v5 QMC5883P. Verified by
  reading `lis3mdl_calibration_t` struct in `lis3mdl.h`.
- GPS-date API carries forward unchanged — MAX-M10S NMEA + UBX dates feed the
  same "strictly-newer" guard.
- Height-reference API carries forward unchanged; activates once `fusion.c`
  references the user-zeroed altitude.

## Test harness

`firmware/test/test_app_nvs.c`:

1. `nvs_flash_erase()` + `app_nvs_init(&cal)` — confirm safe defaults loaded
   when keys are absent.
2. Save mag-cal floats (e.g. `offset_x = 1.234`, `scale_avg = 0.987`); load
   them back; assert the round-trip is exact within the `×1000` quantisation
   tolerance.
3. Save GPS date `2026-06-13`; attempt to save `2026-06-12` — assert
   `ESP_ERR_INVALID_STATE` (older-than-stored guard).
4. Save GPS date `2026-06-14`; load; assert the new date came back.
5. Save UI settings (`brightness=42`, `theme=LIGHT`); load; assert round-trip.
6. Create the async settings queue via `app_nvs_settings_queue_create()`;
   confirm it has depth 1 with overwrite semantics (push twice, drain once,
   assert second value is what comes out).

### Notes
- NVS writes happen only from `task_settings_saver` (Core 1 → queue → unpinned
  worker), never from the LVGL refresh task. The async pattern is preserved
  bit-for-bit from v5 and remains the correct shape for v7.2.
- Magnetometer-cal scale endpoints (`×1000` quantisation) give three decimal
  places — verified sufficient for LIS3MDLTR's typical 0.146 mGauss/LSB
  range. Reconfirm with datasheet in the Phase 20 pass.

## Defects discovered

[DEFECT-001] `ui_settings_save_async` declaration commented out in app_nvs.h | components/app_logic/app_nvs.h:87 | LOW | The function is declared (in header) and defined (in `ui_broker.h` / `ui_broker.c` under the lvgl_ui component) but the legacy app_nvs.h block still has the prototype commented out. Disposition: keep commented — function lives in ui_broker now; the comment is a navigation breadcrumb, not a bug.

[DEFECT-002] BQ25619 lifetime min/max voltage NOT persisted to NVS | components/bq25619/bq25619.c:138-153 | MED | The v5 `"battery"` NVS namespace tracked battery curve endpoints across reboots. `app_battery.c` is gone; BQ25619 keeps `s_vbat_min_mv` / `s_vbat_max_mv` in static RAM only with a `TODO: ... persist to NVS` note at line 147. Lifetime-extreme curve adaptation is therefore reset on every boot. Disposition: file as Phase-1 BQ25619 follow-up. **Do not** silently move into app_logic — keep the chip driver's persistence local to its own file.
