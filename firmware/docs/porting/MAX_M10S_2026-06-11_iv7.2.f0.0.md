## Summary

The MAX-M10S replaces the legacy TU10F GPS module. Both are u-blox M10 silicon (TU10F was the M10050-KB inside a module package), so the NMEA-side parser carries forward verbatim. The new ground in this port is the **UBX-NAV-TIMEUTC** path -- a binary frame the chip emits as soon as the receiver has timing lock, BEFORE any position fix. That collapses the GPS-time-seeding latency from "first valid position fix" (often 30-90 s on cold start) to "first valid timing-only fix" (typically < 15 s, sometimes < 5 s with AGPS-like ephemeris cache). The system clock + PCF85063 RTC are seeded from whichever path fires first.

The other new ground is the **1PPS** TimePulse line on GPIO46. Today we install a rising-edge ISR that just counts edges and timestamps the most recent edge with `esp_timer_get_time()`. The Phase 2+ work (RTC discipline) will use the edge timestamp to slew-correct the PCF85063 -- not in scope this batch.

The `broker_gps_data_t` shape is preserved field-for-field so `gps_tile.c` builds with only an include-path swap and three identity-call renames. The ATOMIC SYNC button and NMEA debug overlay keep working.

Today's deliverable: new driver (NMEA + UBX demux + 1PPS ISR), tile refactor (identity rename only -- layout unchanged), standalone test harness with hardcoded NMEA + UBX validation, old `tu10f.{c,h}` deleted.

## Hardware spec from v7.2

### Pinout (v7.2 §GPIO ASSIGNMENT)

| Signal       | GPIO | Direction | Notes |
|---|:--:|---|---|
| GPS_RX (ESP→GPS) | 17 | UART1 TX | feeds GPS data line |
| GPS_TX (GPS→ESP) | 18 | UART1 RX | RX pullup enabled in init |
| GPS_PPS          | 46 | input    | TimePulse, rising-edge ISR, pull-down |

UART1 is a hard-wired pair: no other peripheral shares the lines. The chip default baud is **9600** (datasheet); the legacy TU10F shipped at 38400 and we re-configured on boot. For MAX-M10S we start at 9600 and assume the chip is unmodified -- if the bench unit was preconfigured higher, init will succeed but `max_m10s_update()` returns `ESP_ERR_NOT_FOUND` and the parser sees no sentences. Flagged as [DEFECT-001].

### 1PPS line (v7.2 §GPIO ASSIGNMENT)

GPIO46 carries the TimePulse output. MAX-M10S TimePulse is push-pull with a programmable polarity; chip default after cold start is "rising edge at the top of each UTC second, pulse width ~100 ms". We configure GPIO46 as input + pull-down (idle LOW between pulses) and trigger on the rising edge. The ISR is IRAM-resident, count-only -- no I2C, no logs.

### Power

| Rail | Connected | Notes |
|---|---|---|
| 3V3  | MAX-M10S VCC, V_BCKP | TPS62840 buck; same rail as IMU, mag, light. Single rail keeps the supply rejection simple. |

## Code audit

### What dies

`components/max_m10s/tu10f.{c,h}` -- deleted. The TU10F driver was a u-blox M10 NMEA-only path; functionally equivalent on the NMEA side, but had no UBX parser and no 1PPS handler.

### What carries forward

- **`broker_gps_data_t` shape, field-for-field.** Defined now in `max_m10s.h`. `gps_tile.c` builds with only the include swap + three identity-call renames (chip name/desc + debug-sentence accessor).
- **NMEA parser logic.** `parse_gga`, `parse_rmc`, `nmea_to_decimal`, `parse_time_str`, `parse_date_str`, the bulk-read UART pump, the binary-junk filter on the NMEA line discipline -- verbatim from `tu10f.c`. These chip-independent helpers have years of bench time behind them.
- **System-clock + RTC seeding logic.** `timegm()` + `settimeofday()` + `pcf85063_sync_utc()` under `g_i2c_mutex`. Same `g_gps_time_seeded` one-shot.
- **Cross-driver event firing.** `XD_EVENT_GPS_TIME_VALID`, `XD_EVENT_GPS_FIX_VALID`, `XD_EVENT_GPS_FIX_LOST` fire on the same conditions as before.
- **Task entry point name.** `task_gps_fn` is what `boot_tasks.c` wires. Kept.

### What's new

- **UBX state machine.** Two-byte sync (0xB5, 0x62), class+id, little-endian payload length, payload, Fletcher-8 checksum (`ck_a`, `ck_b`). The implementation is a single byte-at-a-time switch on `ubx_state_t`. Reset on bad sync, length overflow, or checksum failure. `max_m10s_feed_ubx_byte()` is exposed in the header so the test harness can drive it without UART.
- **UBX-NAV-TIMEUTC handler.** When a complete frame with class=0x01 id=0x21 arrives, the payload is decoded into `s_gps.time_utc` and `time_valid` is set -- but only if the chip's own `valid` byte has bit 2 (`validUTC`) set. That guards against the receiver emitting a placeholder frame before the timing-lock is real.
- **UART demux.** Each incoming byte is routed to either the UBX state machine or the NMEA line discipline. The rule: if UBX is mid-frame, all bytes go to UBX. Outside a UBX frame, `0xB5` enters UBX, printable ASCII extends NMEA, anything else resets the NMEA line.
- **1PPS ISR.** Rising-edge IRAM ISR on GPIO46, increments `s_pps_count`, captures `esp_timer_get_time()` into `s_pps_last_us`. `gpio_install_isr_service()` is idempotent across drivers (we tolerate `ESP_ERR_INVALID_STATE` because cst9217 / lis3mdl may have installed it first).
- **Init log line.** Mentions UART, 1PPS pin, baud explicitly so first-boot logs leave a breadcrumb.
- **Seed log line.** When the RTC seed fires, the log message distinguishes "NMEA RMC" vs "UBX-NAV-TIMEUTC" depending on whether a position fix was already present -- useful for measuring how often UBX wins the race.

- `broker_gps_data_t` -- field-for-field unchanged.
- `BROKER_GPS_TIMEOUT_MS` = 5000.
- `max_m10s_debug_sentences_t` -- typedef preserved (renamed only).
- Public API: `init`, `deinit`, `update`, `get_snapshot`, `has_fix`, `flush`, `get_debug_sentences`, `get_utc_time`, `get_pps_count`, `get_last_pps_us`, `feed_ubx_byte`, `task_gps_fn`.

### `components/max_m10s/max_m10s.c`

- Two file-static parser states: `s_gps` (NMEA-derived snapshot) and `s_ubx` (UBX state machine).
- `process_nmea_sentence()` -- routes to `parse_gga` / `parse_rmc`, also captures raw sentence into `s_debug.gga/rmc`.
- `handle_ubx_nav_timeutc()` -- 20-byte payload decode, validates `valid & 0x04` (validUTC), commits to `s_gps.time_utc` + flips `time_valid`. Stamps `last_update_ms` so the broker sees the time-only update as fresh.
- `max_m10s_feed_ubx_byte()` -- pure CPU state machine, no I2C/UART; exposed for test harness.
- `pps_isr` -- IRAM, increments + timestamps. `pps_install()` tolerates the already-installed ISR service from sibling drivers.
- `max_m10s_update()` -- bulk drain, byte-by-byte demux. Mirrors the tu10f.c pump but routes UBX bytes to the new state machine.
- `task_gps_fn` -- same shape as before (`broker_gps_hw_alive`, `broker_gps_get_enabled`, 200 ms poll, RBW broker write, cross-driver fires, RTC seed). Log message at seed-time identifies the path that won the race.

### `components/max_m10s/gps_tile.{c,h}`

- `#include "tu10f.h"` -> `#include "max_m10s.h"`.
- Three identity-call swaps: `tu10f_get_chip_name/desc` -> `max_m10s_get_chip_name/desc`, `tu10f_get_debug_sentences` -> `max_m10s_get_debug_sentences`.
- Comment block updated to reference the new chip name.
- Layout unchanged. The tile is read-only with respect to chip-specific data, and the broker shape is identical, so no other edits were needed.

### `components/max_m10s/CMakeLists.txt`

- `SRCS tu10f.c` -> `SRCS max_m10s.c`. `gps_tile.c` unchanged. REQUIRES list unchanged.

### `test/test_max_m10s.c`

Five-phase standalone harness:

1. **Phase 1** -- `max_m10s_init()`. Logs UART/1PPS pins and wall-time of the call.
2. **Phase 2** -- NMEA loopback. Writes a hardcoded `$GPGGA` + `$GPRMC` pair (Ljubljana, 14 Feb 2026, 14:32:07 UTC) via `uart_write_bytes()` and calls `max_m10s_update()`. Requires a TX18<->RX17 jumper on the dev board; if absent, the test logs a warning and continues.
3. **Phase 3** -- UBX direct feed. Hand-built `NAV-TIMEUTC` payload (year=2026, month=2, day=14, hh=14, mm=32, ss=07, valid=0x07), checksum computed on-the-fly. Bytes fed via `max_m10s_feed_ubx_byte()`, snapshot inspected, time_valid asserted.
4. **Phase 4** -- Live observation. 10 s window pumping `max_m10s_update()` at 5 Hz and watching the 1PPS edge count. Useful for bench triage of a real GPS module.
5. **Phase 5** -- Stack high-water mark + heap report.

### Notes

- **The UBX-TIMEUTC path is the headline feature.** First valid UTC arrives many seconds before a position fix; with a clear sky we'd expect the RTC seed log to nearly always say "UBX-NAV-TIMEUTC", not "NMEA RMC". If the log persistently says NMEA, the chip likely isn't emitting UBX frames at all -- check UBX-CFG defaults.
- **1PPS counts but doesn't slew yet.** The edge timestamp is captured with ~us precision via `esp_timer_get_time()` from inside IRAM. The actual RTC discipline (compare edge-to-RTC-tick offset, write `OFFSET` register to PCF85063) is deferred to a later phase.
- **The NMEA loopback test needs a board jumper.** Without TX18<->RX17 shorted, Phase 2 logs a warning but Phase 3 (UBX) still validates the parser via direct feed.

## Defects discovered

- `[DEFECT-001] MAX_M10S_BAUD_RATE = 9600 (datasheet default) [DSV] | components/max_m10s/max_m10s.h:51 | HIGH | The legacy TU10F shipped at 38400 baud and we reconfigured at boot. MAX-M10S datasheet default is 9600. If the bench unit was preconfigured to a different baud by a previous firmware, init will succeed (UART driver doesn't care) but the parser sees no sentences and reports SENSOR_OFFLINE within 5 s. Disposition: confirm chip baud on bench; consider auto-detect by trying 9600 then 38400 then 115200.`
- `[DEFECT-002] UBX-CFG not sent on boot | components/max_m10s/max_m10s.c:max_m10s_init | MED | The MAX-M10S powers up with a chip-default UBX message rate that may NOT include NAV-TIMEUTC. The datasheet says the default protocol mix is "UBX + NMEA" but the default UBX-NAV message subscription varies by firmware revision (M10 ROM 5.10 vs 5.20+). If NAV-TIMEUTC is silent, the UBX path never fires and the RTC seed falls back to NMEA only -- functionally identical to the old TU10F. Disposition: add a UBX-CFG-MSG send during init to explicitly subscribe NAV-TIMEUTC; deferred until bench confirms which firmware rev ships.`
- `[DEFECT-003] No UBX frame timeout / resync | components/max_m10s/max_m10s.c:max_m10s_feed_ubx_byte | MED | If the chip sends a partial UBX frame and then stops, the state machine stays parked mid-frame and consumes subsequent NMEA bytes as UBX payload (silently corrupting both streams). Disposition: add a 200 ms inactivity reset (call ubx_reset() if no bytes arrived since the frame started). Low priority because the chip emits one full frame per second and partial frames are uncommon.`
- `[DEFECT-004] 1PPS ISR shares gpio_isr_service with siblings | components/max_m10s/max_m10s.c:pps_install | LOW | gpio_install_isr_service() can only succeed once per app. We tolerate ESP_ERR_INVALID_STATE on the second-and-later install. If a sibling driver later changes its install flags (e.g. drops ESP_INTR_FLAG_IRAM), 1PPS ISR could end up not-in-IRAM. Disposition: centralise gpio_isr_service install in boot_hw_init.c once that file lands. Same root cause as [DEFECT-006] in Phase 2.`
- `[DEFECT-005] timegm() may not be available in IDF newlib | components/max_m10s/max_m10s.c:task_gps_fn | LOW | The Phase 14 tu10f.c had a commented-out timegm() line ("// time_t epoch = timegm(&utc_tm);") and used mktime() with a TZ=UTC0 trick. Phase 15 fix replaced with timegm() per its commit message. We re-use timegm() here. If the IDF newlib variant we're building against does NOT export timegm(), the link fails. Disposition: confirm in CI; if missing, fall back to the setenv("TZ","UTC0")+tzset()+mktime() trick.`
- `[DEFECT-006] g_i2c_mutex via extern, g_gps_time_seeded via extern | components/max_m10s/max_m10s.c | LOW (pre-existing) | Same pattern as cst9217 / bq25619 / lsm6dsv16x / lis3mdl / veml6030. Resolves when boot_hw_init.h lands.`

## Open questions

1. **Baud rate auto-detect.** Worth implementing now, or accept that bench bring-up will confirm the right baud and we hardcode it? Recommend: hardcode for v1, auto-detect only if bench shows it's a moving target.
2. **UBX-CFG-MSG init sequence.** Should we explicitly subscribe to NAV-TIMEUTC on every boot, or rely on the chip's flash-saved config from a one-time factory provisioning step? Recommend: send UBX-CFG-MSG on boot to be self-contained; the cost is one short I2C-style write at startup.
3. **AGPS / AssistNow.** MAX-M10S supports AssistNow Online (ephemeris fetched over IP) which collapses TTFF to single-digit seconds. Out of scope without a network stack, but worth flagging.
4. **1PPS interrupt latency.** With IRAM ISR + `esp_timer_get_time()` we should get sub-us jitter on the timestamp. Bench measurement will confirm. If the jitter is > 10 us, the RTC discipline math has to account for it -- might want a hardware timer capture instead.
5. **Power management.** MAX-M10S has a software-controlled low-power mode (UBX-RXM-PMREQ). When the watch goes to sleep, should the GNSS be put in cyclic-tracking mode (sips power, keeps fix warm) or fully off (cold start on wake)? Recommend: defer to Phase 2+ power-management batch.
