### Bit-field decoding

**Control_1 (0x00)** [p. 6–7, Table 5]
- `[7] EXT_TEST` — 0=normal, 1=external clock test mode (replaces internal 32 kHz with CLKOUT-as-input). **Never set in production firmware.**
- `[5] STOP` — 0=RTC running, 1=upper prescaler held in reset, no 1 Hz ticks generated. Used to load exact time before start. After clearing, first tick fires **~507 ms later** [p. 9, Figure 7].
- `[4] SR` — Software reset. Writing 0x58 to the whole register clears all regs to defaults. Read-only returns 0.
- `[2] CIE` — Correction interrupt enable. 1 = pulse on INT every offset-correction cycle (debug only).
- `[1] 12_24` — 0=24-hour mode (default), 1=12-hour mode.
- `[0] CAP_SEL` — 0=7 pF crystal load, 1=12.5 pF crystal load.

**Control_2 (0x01)** [p. 11, Table 8]
- `[7] AIE` — Alarm interrupt enable; 1=INT pin follows AF flag.
- `[6] AF` — Alarm flag; read 1 = alarm matched. **Write 0 to clear** (write-1 leaves it unchanged). Software-clear only.
- `[5] MI` — Minute interrupt enable (INT every 60 s).
- `[4] HMI` — Half-minute interrupt enable (INT every 30 s).
- `[3] TF` — Timer flag; 1 = countdown timer expired. Software-clear.
- `[2:0] COF[2:0]` — CLKOUT frequency [p. 14, Table 10]: 000 = 32 768 Hz (default), 001 = 16 384 Hz, 010 = 8 192 Hz, 011 = 4 096 Hz, 100 = 2 048 Hz, 101 = 1 024 Hz, 110 = 1 Hz, 111 = CLKOUT = LOW.

**Offset (0x02)** [p. 14–18]
- `[7] MODE` — 0 = normal (correction every 2 hours, 4.34 ppm/LSB, lower-power); 1 = fast (correction every 4 minutes, 4.069 ppm/LSB, slightly higher power)
- `[6:0] OFFSET[6:0]` — Two's-complement, range +63 to −64 LSB
- Calibration achieves **±2 ppm** accuracy with workflow shown on [p. 18, Figure 11]; ±1 ppm ≈ 0.0864 s/day drift

**Seconds (0x04) OS bit** [p. 20, §7.3.1.1]
- `[7] OS` — Set to 1 by hardware whenever oscillator stops (power loss, weak supply, OSCI shorted). FW must clear it by writing 0 to the bit after the oscillator is confirmed stable (200 ms – 2 s after power-on). If OS cannot be cleared, the oscillator is **not running** — use as health check.

**Timer_mode (0x11)** [p. 27, Table 34; p. 28, Table 35]
- `[4:3] TCF[1:0]` — timer source clock: `00`=4096 Hz, `01`=64 Hz, `10`=1 Hz, `11`=1/60 Hz (default at reset)
- `[2] TE` — 1=timer running
- `[1] TIE` — INT enable on timer expire
- `[0] TI_TP` — 0=INT follows TF flag (level); 1=INT is a pulse on expiry, TF independent

Countdown period = T / SourceClockFrequency. With T ∈ [1, 255]: **244 µs to 4 h 15 min** [p. 27, Table 35].

## Initialization sequence

[Synthesized from p. 10, §7.2; p. 20, §7.3.1.1; p. 23, §7.4]

```
1. (Hardware) +3V3 rises; ML621 backup cell trickle-charges via 1N4148 + 10kΩ
2. Wait for oscillator stable: 200 ms to 2 s, depends on crystal + temp [p. 20]
3. (Optional) Software reset for clean state:
     Write 0x58 to Control_1 (0x00). Restart I²C with STOP+START.    [p. 10]
4. Read Seconds (0x04). If bit 7 (OS) = 1:
     → Oscillator was stopped/restarted.
     → Clear OS by writing seconds value with bit 7 = 0.
     → Re-check after some seconds; if OS stays 1 → oscillator dead.
5. If crystal CL = 12.5 pF (not 7 pF): write Control_1 bit 0 (CAP_SEL) = 1.
6. (Optional) Configure CLKOUT: write Control_2 COF[2:0]. v7.2 default
   is "RTC CLKOE = GND" — keep COF = 111 (CLKOUT off) to save power
   unless the ESP needs an external 1 Hz/32 kHz feed.
7. (One-time) Set time from a known-good source:
     Single I²C write starting at register 0x04, len = 7 (S, M, H, D, W, Mo, Y).
     **Must complete in one < 1 s transaction to avoid carry-corruption.**
8. (Optional) Program alarm:
     a. Write all five 0x0B–0x0F alarm regs with AEN_x bit 7:
        - 0 = include this field in match
        - 1 = "match always" (ignore this field)
     b. Set Control_2 bit 7 (AIE) = 1
     c. Make sure Control_2 bit 6 (AF) is 0 (any pending alarm cleared)
9. (Optional) Program countdown timer (Timer_value, Timer_mode).
10. RTC is ready. Read time via the recommended sequence [p. 24]:
    - Write target+W, send 0x04, repeated-start, target+R, read 7 bytes,
      STOP.  All in one transaction.
```

**Critical timing constraint:** during a multi-byte read/write of time regs (0x04–0x0A), the time counter is **frozen**. The transaction must complete in **< 1 s** [p. 23, Figure 15], else at most one missed tick will be applied at release; multiple missed ticks corrupt the count. At 400 kHz I²C, a 7-byte read takes ~250 µs — comfortably within budget.

## Operation

### Reading the time (canonical sequence) [p. 24]
```
S · 0xA2 (W) · ACK · 0x04 · ACK · SR · 0xA3 (R) · ACK ·
  byte0 (Seconds + OS) · ACK ·
  byte1 (Minutes) · ACK ·
  byte2 (Hours) · ACK ·
  byte3 (Days) · ACK ·
  byte4 (Weekdays) · ACK ·
  byte5 (Months) · ACK ·
  byte6 (Years) · NAK · P
```

### Setting time
- Same shape, write instead of read.
- **Must be one I²C transaction** (no STOP between fields), or risk minute/hour rolling over between sub-writes.

### Alarm comparison logic [p. 26, Figure 16]
- For each of {sec, min, hour, day, weekday}: the alarm register's AEN_x bit gates whether that field participates in the match.
- AF (Control_2 bit 6) is set to 1 the moment **all enabled** alarm fields match the current time **simultaneously** during an increment.
- AF persists until cleared (write 0); even if AIE=0, AF still gets set.
- INT pin follows AF when AIE=1.
- **A field with AEN_x=1 means "this field always matches"** (= "wildcard"). To alarm at "every minute at second 30", set Second_alarm = 0x30, AEN_S=0; set all four other AEN_x=1.

### Timer [p. 27–29]
- Source clock × countdown value T (1–255) = period (244 µs to 4 h 15 min)
- **First period is uncertain by up to one tick** [p. 28, Figure 17; Table 36] — because enable is asynchronous to the source clock
- Reading Timer_value while running returns the **current count**, not the loaded T — read twice to confirm stable
- INT pulse-width depends on source clock and TI_TP setting [p. 29, Table 37]

### Power-up behavior
- "It is required that V_DD start at zero volts at power up or upon power cycling to ensure that there is no corruption of the registers." [p. 10] — if power glitches without dropping fully, FW should issue a SW reset before trusting register state.
- Internal POR is generated automatically when V_DD rises through the threshold.

### Backup-cell switchover (implied)
- The datasheet does not explicitly describe a "switch-to-backup" pin or mechanism. The standard topology used by v7.2 (1N4148 from +3V3 + 10 kΩ trickle-charge to ML621, with the ML621 also feeding V_DD via its own series diode) achieves automatic switchover passively: when +3V3 drops below the ML621 voltage minus a diode drop, the cell takes over.

## Interrupts

| Signal | Pin (v7.2) | Polarity | Trigger sources | De-assert | Notes |
|---|---|---|---|---|---|
| INT | **GPIO15** | **Open-drain, active LOW** [p. 4 footnote 1] | Alarm (AF), countdown timer (TF), minute (MI), half-minute (HMI), offset-correction (CIE) | All software-clear via Control_2 writes (write 0 to AF / TF; or reset enable bits) | Needs external pull-up (or internal ESP-S3 WPU) — see WARN-01 |

**Interrupt sharing:** If AIE + TIE (via Control_2.bit3 via TF — see TI_TP) are both enabled, INT can be asserted by either. FW must read Control_2 to disambiguate AF vs TF on each interrupt.

**TI_TP modes** [p. 13, §7.2.2.3]:
- `TI_TP = 0` (default): INT level-follows the TF flag. INT stays low until FW clears TF.
- `TI_TP = 1`: INT pulses on each timer expiry; pulse width per Table 37; TF still set but does not gate INT. **Useful for repeating timer wake-ups** because FW does not need to clear TF every cycle.

**Wake-from-deep-sleep on GPIO15** [v7.2 line 128]: required wake source. The ESP-S3 deep-sleep wake-on-GPIO uses falling edge detection — pairs naturally with the active-low INT. Make sure WPU is enabled (internal or external) before entering deep-sleep, else INT line floats and edges may be missed.

### Battery-budget calc (v7.2 cross-check)

ML621 capacity ≈ **5.5 mAh** [v7.2 BOM]. At 220 nA typical (25 °C), runtime ≈ **5.5 mAh / 0.22 µA = 25 000 h ≈ 2.85 years**. v7.2 claim "~4 years" is correct as an **order-of-magnitude estimate**; cold-temperature operation (< 25 °C average) gets there. At wrist-warm 30–35 °C average, expect closer to 2–3 years per full charge — which is still acceptable for a device with USB-rechargeable backup.

**Interface-active cost:** if firmware polls every 1 s at 100 µs/transaction, average current = 18 µA × (100 µs / 1 s) = 1.8 nA additional — **negligible**. If firmware polls every 100 ms, average = 18 nA — still negligible. The 18 µA peak only matters during the actual transaction window for instantaneous battery droop, not for the budget.

### Off-budget
- **No deep-sleep / shutdown** mode for the RTC itself — it's always running. Powering it off means losing time (the very purpose this part exists to prevent). The ML621 backup is the design's "off" state.

## Datasheet notes

- **Time range**: 2000-01-01 (Sat) to 2099-12-31 — Years register is BCD 00–99, with century not tracked. **Y2K100 problem** in the year 2100. Not a concern for any reasonable firmware lifetime.
- **Leap year auto-handling**: years divisible by 4 (including 00) add Feb 29 [p. 21, Table 22, footnote 1]. Correct for 2000-2099 — exception that century years (1900, 2100) not divisible by 400 should NOT be leap years is not handled, but 2100 is past our scope.
- **CLKOUT push-pull, 32 768 Hz default** [p. 14, Table 10]. **v7.2 default = disabled via JP5 (GND)**. To save power, leave COF[2:0] = 111 (CLKOUT off in software) too.
- **Address auto-increment wraps at 0x11 → 0x00** [p. 5]. A bulk write past timer registers will overwrite Control_1 — be careful with multi-byte writes that cross 0x11.
- **Address pointer is preserved across STOP** as long as power is maintained [implicit from p. 5–6 description]. A read after a write to set the pointer is the canonical pattern.
- **5 V tolerant I²C** [p. 35, footnote 5]: SDA/SCL can be pulled to 5 V even if V_DD = 3.3 V. Not relevant for v7.2 (everything is 3.3 V).
- **ESD**: HBM ±5 kV; CDM ±1.75 kV (ATL), ±2.0 kV (AT/ATT) [p. 33, Table 39]. Comfortable for wearable handling.
- **Latch-up**: 200 mA at max ambient [p. 34, Table 39 footnote 4].
- **Oscillator series resistance**: typical 60 kΩ, max 100 kΩ [p. 35]. The generic 32 768 Hz crystal in v7.2 BOM (LCSC C2842180) needs to satisfy R_s ≤ 100 kΩ and have CL = 7 pF or 12.5 pF.
- **Oscillator startup time**: 200 ms to 2 s depending on crystal, temperature, supply voltage [p. 20, §7.3.1.1]. FW should not assume time is valid for at least 2 s after power-on.
- **VDD min for I²C active**: 1.8 V at 400 kHz, but characterization shows it works down to 1.71 V (1.8 V − 5%) [p. 35, footnote 2]. For pure timekeeping V_DD can drop to **0.9 V** — well within ML621's discharge curve.
- **CLKOUT 32 768 Hz duty cycle** is **not exactly 50:50** [p. 14, Table 10, footnote 2] — typical 60:40 to 40:60. Other frequencies are 50:50. Relevant only if the CLKOUT is used as a clock for another chip's PLL (not the v7.2 use case).
- **Datasheet revision date**: 26 Aug 2025 (Rev. 7.2). Current and authoritative.

## Warnings & Errata

| ID | Title | Source | Severity | Description / Mitigation |
|---|---|---|---|---|
| WARN-01 | **INT (GPIO15) is open-drain, needs a pull-up** | [p. 4, Table 3 footnote 1] | **HIGH** | The INT pin is open-drain. The 5.1 kΩ pull-ups on the v7.2 bus 1 are for SDA/SCL, **not for INT**. If GPIO15 floats high, the line is undefined; if it floats low, the ESP sees a permanent "alarm". **Mitigation:** enable ESP-S3 internal weak pull-up (~45 kΩ) on GPIO15 in FW init. Equivalently, add a 100 kΩ external pull-up to +3V3 — adds ~30 nA drain at idle, well below the RTC's own budget. Confirm during board bring-up. v7.2 does **not** call out a pull-up on GPIO15. |
| WARN-02 | **CAP_SEL default may mismatch crystal** | [p. 6, Table 5, bit 0] | **HIGH** | After power-on, Control_1.CAP_SEL = 0 → internal load = 7 pF. | WARN-03 | **OS flag must be cleared on every cold boot** | [p. 20, §7.3.1.1] | MED | OS bit 7 of Seconds (0x04) defaults to 1 after any oscillator stop. If OS re-asserts after a fresh write, oscillator is dead — fall back to monotonic clock and report fault. |
| WARN-04 | **Stop bit release introduces ~507 ms delay** | [p. 9, Figure 7; p. 9 text] | MED | After clearing STOP (Control_1 bit 5), the first time-increment fires **between 0.507813 s and 0.507935 s later** — not exactly 1 s. If FW uses the STOP/clear pattern to load a known absolute time, the first second is off by 0.5 s. **Mitigation:** load time *without* using STOP (just direct write to 0x04–0x0A), which avoids this transient at the cost of up to a 1-tick error. |

## Open questions

- **Q1 — Crystal load capacitance.** What is the actual CL spec on LCSC C2842180 (or whatever 32 768 Hz crystal v7.2 ends up with)? Drives the CAP_SEL bit choice during init [WARN-02]. **Action:** Ivan — pull the crystal datasheet.
- **Q2 — NXP package confirmation.** Is the part on the v7.2 BOM the **PCF85063ATL** (DFN2626-10, with CLKOE pin)? JP5 implies yes, but the manifest doesn't say. **Action:** Ivan — confirm LCSC code.
- **Q3 — GPIO15 pull resistor strategy.** Internal ESP-S3 WPU (45 kΩ) is sufficient if the trace is short. But during deep-sleep, the WPU draw is ~75 µA continuous (3.3 V / 45 kΩ) — *much more than the RTC itself*. **Action:** prefer an external 100 kΩ to +3V3 (30 µA draw) added in next board rev. For Mk I, accept the WPU cost and quantify on bench [WARN-01].
- **Q4 — JP5 default.** v7.2 says JP5 default = GND (CLKOE=LOW → CLKOUT disabled). The datasheet says CLKOE only valid for DFN package, and even then "CLKOUT can be disabled by setting COF[2:0] to 111 or by setting CLKOE LOW" [p. 14]. **Action:** software-disable CLKOUT in FW init regardless of the JP5 default, to be robust against jumper changes.
- **Q5 — Backup-cell trickle-charge current.** v7.2's "10 kΩ charge limit" gives roughly (3.3 V − 0.6 V diode − 3.0 V cell) / 10 kΩ = ~30 µA charging current. ML621 cells typically accept 0.1–1 mA — 30 µA is well within spec but slow (5.5 mAh / 30 µA ≈ 7.6 days for full charge). **Action:** verify charge-rate expectation; bench-measure cell voltage after typical charging duration to confirm full charge is achievable.
- **Q6 — Use of internal RAM byte (0x03).** The free 8-bit RAM_byte persists across power-cycles only if the backup cell holds; not across full cold-boot. **Action:** if FW wants to flag "boot from RTC wake" vs "boot from fresh power-on", consider using this byte — but document that it's not durable storage. NVS flash is the right place for permanent state.
