## Initialization sequence

[driver source: `cst9217.c:cst9217_init`, `cst9217_hw_reset`]

```
1. Drive GPIO44 (RST) LOW for ≥ 5 ms                  [CST9217_RST_LOW_MS]
2. Drive GPIO44 HIGH; wait 50 ms for chip to settle   [CST9217_RST_HIGH_MS]
3. (Under g_i2c_mutex) Read 1 byte at register 0xD000
   Expect 0xAB. Anything else → ESP_ERR_NOT_FOUND
4. Configure GPIO6 as input, internal pull-up, falling-edge interrupt
5. Install GPIO ISR service (idempotent — INVALID_STATE means already installed,
   which is benign)
6. Register the per-pin ISR handler
7. Driver is ready; task_touch_fn blocks on ulTaskNotifyTake(portMAX_DELAY)
```

**Reset pulse widths:** 5 ms low / 50 ms high are first-pass conservative values from generic touch-IC practice [DEFECT-003 in the porting log]. Datasheet confirmation pending.

**No software sleep command** is issued during init. The chip powers up with the rest of the device (no enable pin on +3V3 [v7.2]).

## Operation

### Read path (ISR-driven)

[driver source: `cst9217.c:cst9217_isr`, `task_touch_fn`]

1. Touch event → CST9217 drives GPIO6 LOW (falling edge).
2. `cst9217_isr` (IRAM, ≤ 1 µs typical): `vTaskNotifyGiveFromISR()` + `portYIELD_FROM_ISR()`. No I2C, no logging.
3. `task_touch_fn` on Core 0 wakes, takes `g_i2c_mutex` with **50 ms ceiling**.
4. Burst-read 8 bytes at `0xD000`: write `[0xD0, 0x00]`, restart, read 8 bytes.
5. Release mutex, parse into `cst9217_point_t {x, y, fingers, gesture, t_us}`.
6. `xQueueOverwrite(g_touch_q, &p)` — depth-1, last-write-wins.
7. LVGL indev callback (in `lvgl_ui`) does non-blocking `xQueuePeek` / `xQueueReceive(0)`.

**Polling path: explicitly avoided.** Latency, not throughput, is the design constraint [porting log §Profiling].

### Wire-time floor at 400 kHz [driver source comments]

| Operation | Bytes | Wire time |
|---|---|---|
| ACK probe (write 2-byte reg + read 1) | 3 wr + 2 rd + START/STOP | ~80 µs |
| Full report (write 2-byte reg + read 8) | 3 wr + 9 rd + START/STOP | ~280 µs |

Add ~50–150 µs driver overhead (mutex, I2C setup) → typical end-to-end latency **well under 1 ms** at the typical case. Worst case is bounded by the 50 ms mutex ceiling (set in `task_touch_fn`).

### Sleep / low-power mode

[no datasheet — bench TODO]. CST816-family chips support a software "deep sleep" register write. The CST9217 may or may not — driver does not exercise it. If the chip stays awake during ESP light-sleep, this is **the** quiescent-current contributor on bus 1 — quantify on bench.

**Calculation notes:** None possible from documentation. The CST816 family typically draws 1.5–2 mA active and ~10–100 µA in software sleep; treat that as the order-of-magnitude expectation until bench data exists.

**Board-level implication:** because there is no HW shutdown, the touch controller's idle current is a *fixed* contributor to the always-on battery budget. It cannot be gated except by removing the +3V3 rail entirely (which also kills the display, RTC, sensors, etc.). For a smartwatch in deep sleep, the CST9217 idle current is the cost of having "tap-to-wake".

## Datasheet notes

- The chip is from the **Chipone/Hynitron family**. LCSC traceability code is not in the manifest. Vendor naming is inconsistent in source docs — confirm one canonical name before ordering replacements.
- The driver was deliberately written as a **custom component** (not via Espressif's managed `esp_lcd_touch_cst816s`) because (a) the wire-level facts are 16-bit-reg / 0x5A / 0xAB-ACK / 0xD000-base — all different from CST816, and (b) no `esp_lcd_touch_cst9217` managed package exists [porting log §Code audit].
- **GPIO44 boot quirk**: GPIO44 is the **boot-log RX** pin. The ~3 ms boot-log activity at boot is harmless because TP-RST is a driven output that ignores the input blip [v7.2 line 138]. **Never put an I²C line on GPIO43/44.**
- **GPIO6 deep-sleep wake**: tap-to-wake is a Required wake source [v7.2 §Sleep wake sources, line 126]. Anything that masks GPIO6 during deep sleep prep will break the watch's primary wake mechanism.
- The chip lives **on the FPC ribbon**, not on the main PCB — so a chip-level swap requires a panel swap. Treat the chip choice as locked at panel selection time.
- **No public datasheet** appears to be available from Chipone or Hynitron at the time of writing. The vendor distributes the spec under NDA. Adding it to `docs/datasheets/` requires manual contact with the vendor — `datasheets_manifest.md` flags this as a To-Do.

## Open questions

- **Q1 — Datasheet acquisition.** Single biggest unknown. Can Ivan obtain the CST9217 datasheet under NDA from Chipone or Hynitron? Without it, [WARN-01], [WARN-02], [WARN-03], [WARN-04], [WARN-05] all stand. **Action:** Ivan — contact vendor.
- **Q2 — Multi-touch.** The chip family supports two-finger gestures. The current driver is single-point only. Do we want to expose a second contact now? Recommend deferring until LVGL needs it.
- **Q3 — Gesture consumption strategy.** Should gestures be consumed in firmware (driver → LVGL gesture event), or derived in LVGL from raw x/y/down/up? **Recommend the latter** — keeps the driver dumb and the gesture surface uniform across platforms.
- **Q4 — SW low-power mode.** If a deep-sleep register exists (CST816-style), should the firmware drop the chip into it whenever the display is off? Off-budget for now, but quantify the savings on bench first — could move the needle if idle is ~1.5 mA.
- **Q5 — GPIO6 wake handoff.** Coordination between the runtime ISR (this driver) and the deep-sleep wake path (yet to be implemented in `boot_power`). Out of scope here; tag for the boot_power refactor.
- **Q6 — Address-collision sanity.** v7.2 routes DRV2605 to 0x5A on bus 2 — same address, different bus, no collision. Confirm in HW debug that the driver code uses the right `i2c_port_t` consistently. The current driver hardcodes `I2C_NUM_0` for bus 1 [driver source `task_touch_fn`, line 244]; this is correct but fragile if a future refactor splits the bus.
