### Strapping pins [CHIP p. 32, Table 3-1]

| Pin | Default config | Bit value | Function |
|---|---|---|---|
| GPIO0 | Weak pull-up (WPU) | 1 | Chip Boot Mode (1 = SPI Boot, 0 with GPIO46=0 = Joint Download Boot) |

| GPIO46 | Weak pull-down (WPD) | 0 | ROM message printing control |

**Strapping latch timing** [CHIP p. 33, Table 3-2]: t_SU ≥ 0 ms, t_H ≥ **3 ms** after EN/CHIP_PU release before strap pins can be driven as regular IO.

### Pins with power-up glitches [CHIP p. 18–19, Table 2-2]
- **GPIO 1–14, 17, XTAL_32K_P/N**: low-level glitch ~60 µs during chip power-up
- **GPIO 18**: both low and high glitches ~60 µs
- **GPIO 19 (USB D−)**: low + high glitch; total disturbance ~**3.2 ms**
- **GPIO 20 (USB D+)**: pull-down + high glitch; total disturbance ~**2 ms**

**v7.2 impact:** TP-RST on GPIO44 must tolerate the U0RXD boot blip (v7.2 already documents this). The 60 µs low-glitch on GPIO 1, 2, 4, 5 means the I²C lines momentarily go low at the chip boot — should be harmless as the 5.1 kΩ pull-ups re-establish idle within microseconds and the I²C peripherals are not yet initialized.

## I2C / Bus

**Two I²C controllers** on the SoC, configurable to **either** master or slave [MOD p. 18]. ESP32-S3 supports:
- Standard mode: 100 kbit/s
- Fast mode: **400 kbit/s** (v7.2's choice on both buses)
- Up to **800 kbit/s** (constrained by external pull-ups) [MOD p. 18]
- 7-bit and 10-bit addressing
- Double-addressing mode (slave + register address) — required for CST9217 (16-bit reg)

**Internal pull-up resistor** (when enabled): **R_PU = 45 kΩ typical** [CHIP p. 65, Table 5-4]. **v7.2 uses external 5.1 kΩ → 3V3** on both buses (mandatory at 400 kHz; 45 kΩ internal pull-up is too weak for fast mode and would cause SDA droop).

### QSPI / SPI controllers [CHIP p. 53; MOD p. 19]

- **SPI0/SPI1**: dedicated for the in-package flash/PSRAM (octal PSRAM on R8). **Pins 28, 30, 31, 32, 33, 34, 35** (chip pin numbers; module-internal). **Do not use for anything else.** [CHIP p. 26, Priority 4 (P4)]
- **SPI2 (FSPI)**: general-purpose SPI; supports 1/2/4/8-line modes; full IOMUX on GPIO 9–14 (v7.2's display QSPI). **Max clock 80 MHz with IOMUX, 40 MHz with GPIO matrix** [TRM reference — not extracted from datasheet]. Per CO5300 datasheet at 50 MHz max for RGB888 QSPI → comfortably within ESP32-S3's IOMUX path capability.
- **SPI3**: general-purpose SPI; routed only via GPIO matrix. Unused on v7.2.

### Native USB

- **USB Serial/JTAG controller**: GPIO19/20 hardwired (default after reset: USB_PU, IE) [CHIP p. 17, footnote 6; CHIP p. 22, Table 2-4]. Provides CDC + JTAG over a single USB connection — v7.2 relies on this for programming and debug (no UART bridge IC) [v7.2 §MCU].
- **USB 2.0 OTG (full-speed)**: same physical pins (GPIO19/20), runtime-selectable [CHIP p. 22]

## Initialization sequence [MOD p. 16, Table 4-6]

```
1. +3V3 rises; VDDA, VDD3P3, VDD3P3_RTC, VDD3P3_CPU stabilize
2. Wait t_STBL ≥ 50 µs after rails stable                  [CHIP p. 30; MOD p. 16]
3. Release EN/CHIP_PU (drive HIGH or let pull-up rise it)
4. Wait t_H ≥ 3 ms — strapping pins are latched; do not change them    [CHIP p. 33]
5. ROM bootloader runs; reads strap latches; chooses boot mode
6. After ROM boot: pins free up as regular IO
   - GPIO0 latch held internally; pin available as DRV_EN output
   - GPIO45 latch held; pin available as QSPI_TE input
   - GPIO46 latch held; pin available as TimePulse input
   - GPIO3 latch held; pin available as QSPI_RESET output
7. SPI-loaded firmware starts; ESP-IDF runs
```

**Reset pulse on EN**: t_RST ≥ **50 µs** [CHIP p. 30, Table 2-13; MOD p. 16, Table 4-6].

**Boot-log on UART0 (GPIO43/44)** at 115200 Bd default — produces ~3 ms of activity per v7.2 measurements (datasheet does not quantify). To disable: burn EFUSE_UART_PRINT_CONTROL or EFUSE_DIS_USB_SERIAL_JTAG_ROM_PRINT [CHIP p. 32, §3.3].

## Operation

### Clock & CPU [CHIP p. 3, p. 41]
- Two Xtensa LX7 cores, configurable up to **240 MHz**
- 384 KB ROM, 512 KB SRAM, **16 KB SRAM in RTC domain**
- 16 MB external Quad-SPI flash + 8 MB Octal-SPI PSRAM (R8 variant)
- L1 cache; supports 32-bit and 128-bit data access instructions
- 5-stage pipeline, FPU (single-precision), SIMD (128-bit data bus)
- CoreMark: **1329.92 @ 240 MHz dual-core, 5.54 CoreMark/MHz** [CHIP p. 3]

### Power Management Unit (PMU) — operating modes [CHIP p. 5, p. 67–68; MOD p. 28–30]

| Mode | What's on / off | Wake sources |
|---|---|---|
| Active | Full SoC + RF | — |
| Modem-sleep | Wi-Fi clock-gated; CPU/peripherals on | — |
| Light-sleep | VDD_SPI + Wi-Fi powered down; **GPIOs Hi-Z**; clocks stop | RTC GPIO, RTC timer, ULP, UART |
| Deep-sleep | Only RTC domain on (optional ULP) | RTC GPIO, RTC timer, ULP, touch |
| Power-off | EN = LOW; chip dead | EN rising edge (reset) |

### RTC GPIO wake [CHIP p. 23, Table 2-6]
- RTC IO MUX is available **only on GPIO0–GPIO21** (chip pin numbers 5–27, including XTAL_32K and USB_D±).
- v7.2's wake pins **6, 7, 8, 15, 16, 21** all qualify. GPIO 38–48 cannot wake — confirmed.
- During deep-sleep, the regular IO MUX is off; only RTC IO MUX functions. **Driver setup must use `esp_sleep_enable_gpio_wakeup` / `ext0` / `ext1` API, not regular `gpio_install_isr_service`**, for wake-from-deep-sleep.

### Special IO drive strength [CHIP p. 18, footnote 5]
- GPIO17, GPIO18: default 10 mA drive
- GPIO19, GPIO20: default **40 mA drive** (USB lines)
- All others: default 20 mA

v7.2 GPIO 41 (flashlight LED via MOSFET gate) and GPIO 42 (WS2812 data) use 20 mA default — sufficient for both (series resistors handle the current limiting).

### USB Serial/JTAG vs USB OTG (mutually exclusive on GPIO19/20)
- Default after reset: **USB Serial/JTAG** controller active, with internal pull-up (USB_PU). ESP-IDF prog/log uses this path.
- For USB OTG (e.g., HID device, MSC): firmware switches the mux at runtime. ESP-IDF's TinyUSB integration handles this; no GPIO config change needed.

### Native USB CDC vs UART bridge
v7.2 ships **without a UART-to-USB bridge IC**. Programming uses native USB CDC. **Practical implication:** if firmware crashes such that USB CDC enumeration fails, the only recovery is the **Joint Download Boot mode** — pull GPIO0 + GPIO46 low at reset (manual hardware pads, not present on v7.2 — see WARN-03 below).

## Interrupts (high-level)

- **Interrupt Matrix** routes any peripheral interrupt to any CPU [CHIP p. 42]
- **5 priority levels**, NMI support
- **GPIO interrupts**: edge / level / any-edge; configurable per pin via `driver/gpio.h`
- **RTC GPIO interrupts**: only available in light/deep sleep wake paths

## Datasheet notes

- **Octal PSRAM consumes GPIO 33–37** internally on the chip [CHIP p. 31, Table 2-14]. On the WROOM-1U-N16R8 module, GPIO 28, 30, 31, 32 are also dedicated (in-package Quad SPI flash). The module exposes only GPIO 35, 36, 37 in this group; v7.2 (and the datasheet footnote) correctly mark them unavailable. **Do not attempt to use them.** [MOD p. 12 footnote b]
- **GPIO19/20 are by default USB**; to use as plain GPIOs, firmware must clear `IO_MUX_MCU_SEL` [CHIP p. 25, §2.3.4]. v7.2 uses them as USB → no action required.
- **Pad capacitance**: 2 pF typical [CHIP p. 65] — relevant for 5.1 kΩ pull-up timing at 400 kHz I²C (RC = 5100 × ~10 pF including PCB = ~51 ns, well under SDA rise-time budget).
- **ADC calibration**: hardware + software calibrated; total error ±5–50 mV depending on ATTEN setting [CHIP p. 66, Table 5-6]. v7.2 does not use the ADC.
- **Operating frequency** for in-package SPI flash and PSRAM: 80 MHz [MOD p. 30 Table 6-8]. v7.2 should set `CONFIG_SPIRAM_SPEED_80M` (octal mode at 80 MHz is the supported max).
- **Flash data retention**: 20 years [MOD p. 30, Table 6-8].
- **Flash program/erase cycles**: 100 000 [MOD p. 30] — relevant for OTA / NVS write budgeting.
- **Flash chip-erase time** (16 MB part): typ 7 s, max **20 s** [MOD p. 30 Table 6-8].
- **Reliability** [CHIP p. 69, Table 5-13]: ESD HBM ±2000 V, CDM ±1000 V; latchup trigger ±200 mA / 1.5×VDDmax; MSL3 preconditioning required (192 h @ 30 °C / 60 % RH before reflow). **v7.2 §MCU already documents MSL3 — "bake 125 °C / 24h if open > 1 week".**
- **PSRAM 3.3V variants**: VCC 2.7–3.6 V (typ 3.3 V), F_C 80 MHz [MOD p. 31 Table 6-9].
- **Wi-Fi RX sensitivity** [CHIP p. 71, Table 6-4]: −98.4 dBm at 802.11b 1 Mbps; **−104.5 dBm at BLE 125 Kbps** [CHIP p. 5, p. 74 Table 6-14]. Very good for a wearable.
- **BLE TX power range**: −24 to +20 dBm (1 Mbps PHY) [CHIP p. 73, Table 6-8]. v7.2 has no specific TX power target, but the upper bound matters for emissions compliance.

## Warnings & Errata

| ID | Title | Source | Severity | Description / Mitigation |
|---|---|---|---|---|
| WARN-01 | **+65 °C ceiling for N16R8 (octal PSRAM)** | [MOD p. 3, Table 1-2] | MED | The WROOM-1U-**N16R8** variant is rated **−40 to +65 °C**, not the −40 to +85 °C of QSPI-PSRAM variants. A wearable on a wrist in direct sunlight can hit 50 °C surface temperature, and self-heating during sustained BLE TX adds ~10 °C. v7.2 does not call this out. **Mitigation:** add a thermal note to v7.2 §MCU; firmware should monitor the internal temp sensor and throttle CPU/RF if T > 60 °C. |
| WARN-02 | **Module dimensions in v7.2 wrong** | [MOD p. 42] | LOW | v7.2 §MCU states "18 × 25.5 mm module". WROOM-1U is actually **18 × 19.2 mm**. The 25.5 mm dimension belongs to WROOM-1 (with the PCB antenna keepout area). **Mitigation:** correct v7.2. PCB layout is likely already correct (Ivan would have used the actual footprint), but the doc is wrong. |
| WARN-03 | **No manual download-mode pads on v7.2** | [v7.2 §GPIO]; [CHIP p. 33] | MED | If firmware crashes such that USB-Serial-JTAG enumeration breaks (Wi-Fi stack panic, watchdog loop), the *only* recovery is Joint Download Boot, which requires pulling **GPIO0 + GPIO46 low at reset**. GPIO0 is wired to DRV_EN (no test pad), GPIO46 is wired to 1PPS (no test pad). **Mitigation:** consider adding two 0402 test pads (GPIO0/GND, GPIO46/GND) for future revisions. For Mk I, USB Joint Download via `esptool.py` works through the USB-Serial-JTAG handshake *if* the controller is still alive. |
| WARN-04 | **GPIO3 floating strap, panel-dependent** | [CHIP p. 32, Table 3-1; p. 34, §3.4] | MED | GPIO3 has **no internal pull**. v7.2 line 86 says "Held by panel + FW output" — but the panel is unpowered before the ESP boots, so during the brief window before the ESP drives GPIO3, the pin floats. EFUSE_STRAP_JTAG_SEL=0 (default) → JTAG goes to MTDI/MTCK/MTMS/MTDO (chip GPIO 39–42 — v7.2 uses these as SD/flashlight/WS2812). With JTAG hardwired to those pins instead of USB Serial/JTAG, **JTAG is fighting with the v7.2 peripheral assignments at boot**. **Mitigation:** Confirm EFUSE_DIS_PAD_JTAG=1 burn at production (disables external JTAG, forces USB-Serial-JTAG path). Until then, GPIO 39–42 may glitch during the first 3 ms after reset. |
| WARN-05 | **GPIO46 v7.2 wording ambiguous** | [v7.2 line 134; CHIP p. 32] | LOW | v7.2 says "no pull at all" for GPIO46 — meant as "no external pull", but the **internal WPD is the strap-setting mechanism**. A future maintainer might add an external pull-down "to be safe", which is harmless. But adding an external **pull-up** would break the strap → ROM-print would be misconfigured. **Mitigation:** clarify v7.2 line 134 to "no external pull; internal WPD sets boot-msg strap to LOW". |
| WARN-06 | **U0RXD/U0TXD default-bold (boot-log)** | [CHIP p. 22, Table 2-4] | INFO | v7.2 already documents this (line 111-112): GPIO43/44 are the boot-log pins by default. The datasheet's bold formatting (F0 default) confirms it. The "boot-log ~3 ms" duration is empirical; the datasheet only confirms function, not pulse length. The MTCK pin (GPIO39 — v7.2 SD_CMD) is **also bold (F0 = MTCK)** — meaning JTAG is the default function on this pin at boot. Combined with WARN-04, this is the reason firmware should re-mux 39 immediately on startup. |
| WARN-07 | **Power-up glitches up to 3.2 ms on USB lines** | [CHIP p. 19, footnote 2] | LOW | GPIO19 (USB D−) shows low-level + high-level glitches totaling **3.2 ms** during chip power-up; GPIO20 (USB D+) shows pull-down + high glitches totaling 2 ms. Host PC USB stack handles this via standard reset/enumeration sequence. **Mitigation:** none needed — USB-CDC enumeration timing is forgiving. |
| WARN-08 | **eFuse write requires VDD3P3_CPU ≤ 3.3 V** | [CHIP p. 64, footnote 3] | LOW | "If writing to eFuses, the voltage on VDD3P3_CPU should not exceed 3.3 V" — the eFuse program circuitry is sensitive. v7.2 runs at 3.2 V (TPS62840 setpoint), so this is comfortably honored. **Mitigation:** do not bump the buck to 3.3 V exact during production-time eFuse burns; keep the 3.2 V margin. |
| WARN-09 | **Recommended output current ≥ 500 mA** | [CHIP p. 64, footnote 4] | LOW | The chip can draw 500 mA peak (Wi-Fi TX 355 mA + display + sensors). TPS62840 at 750 mA → 250 mA headroom. If a firmware version turns on the +5V boost simultaneously with peak Wi-Fi TX, the BQ25619 PMID boost (1 A rated) also draws from the battery — battery droop becomes the limit, not the +3V3 buck. **Mitigation:** firmware coexistence — don't trigger flashlight max + Wi-Fi TX at the same instant on a low battery. |
| WARN-10 | **MSL3 — must be baked before reflow** | [CHIP p. 69; v7.2 §MCU] | MED | If the WROOM-1U bag has been open > 1 week before reflow, the module must be baked at 125 °C / 24 h or it will crack/delaminate during the thermal profile. Already documented in v7.2 §MCU. **Mitigation:** verify reflow workflow in the Manual Assembly guide. |

## Open questions

- **Q1 — JTAG strap behavior on the v7.2 design.** With GPIO39–42 used as SD_CMD/SD_DAT0/Flashlight/WS2812, and EFUSE_DIS_PAD_JTAG defaulting to 0, JTAG is hardwired to those pins at boot until firmware re-muxes them. **Action:** Ivan — confirm whether you intend to burn EFUSE_DIS_PAD_JTAG=1 at production. If yes, document in v7.2 and the production checklist. If no, document the 3 ms boot-glitch window (matching the boot-log notes for GPIO43/44).
- **Q2 — PSRAM speed config.** Datasheet supports 80 MHz octal PSRAM. ESP-IDF `CONFIG_SPIRAM_SPEED_80M` is the right choice. **Action:** during sdkconfig review, verify this is set.
- **Q3 — Internal temp sensor for thermal throttling.** N16R8 is +65 °C max; a wrist-worn device in sun + active Wi-Fi will approach that. **Action:** add a `temp_sensor` driver task that throttles CPU freq (via `esp_pm_configure`) when T > 60 °C internal.
- **Q4 — Light-sleep current with PSRAM.** Datasheet says 240 µA base + 140 µA for 8 MB 8-line PSRAM = ~380 µA. **Action:** bench-measure on the first board. This is the most-impactful number for "always-on idle" battery life.
- **Q5 — RF emissions at +20 dBm BLE.** BLE TX peak is 344 mA at 20 dBm. CE/FCC have masks at this power level; v7.2's antenna and matching are flagged as "not yet recalculated for 1 mm stackup" [v7.2 Open Issues §1]. **Action:** before any +20 dBm TX, fix the antenna trace width. Default ESP-IDF BLE TX power is +9 dBm (202 mA peak) — safer until antenna is verified.
- **Q6 — Connection to v7.2 EN pin.** Datasheet says "Do not leave CHIP_PU floating" [CHIP p. 28]. v7.2 §POWER doesn't explicitly call out the EN circuit (presumed RC + pull-up to +3V3). **Action:** confirm EN is properly conditioned (typically 10 kΩ to +3V3 + 100 nF to GND + tactile button to GND for manual reset). The BQ_BUTTON on GPIO16 also acts as a wake signal — separate from EN.
