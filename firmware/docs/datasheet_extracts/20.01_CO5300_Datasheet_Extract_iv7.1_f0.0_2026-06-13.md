**Vendor / Doc:** Chipone Technology (Beijing) Co., Ltd. · "OLED Smart Watch Display Driver IC Specification CO5300" V0.00, 2023-03-28 (marked CONFIDENTIAL)

The CO5300 is a wearable AMOLED driver IC with internal RAM (1,843,200 bits / 1.8 Mb GRAM), on-chip DC/DC for panel rails (AVDD/VCL/VGH/VGL), MIPI-DSI and QSPI host interfaces, OTP for ID/gamma/GOA timing, and a deep-standby state. Maximum panel area is 480 × 480 RGB; the v7.2 panel uses 410 × 502 (column offset 22) [v7.2 Display]. [p. 6]

## Registers (firmware-critical)

CO5300 uses **DCS-style commands**, not memory-mapped registers. After HW/SW reset the **User Command Set (CMD1)** is active. Manufacturer Command Sets (CMD2 P0–P0EXT / Gamma1–3 / Page0/PWR / GOA / OLED IP) are reached via `0xFEh = page-number, 0xF4h=0x5Ah, 0xF5h=0x59h` to unlock [p. 117]. **Firmware should not touch CMD2 — it holds factory-tuned gamma/GOA and is OTP-burned by Chipone/panel vendor.**

### MADCTL (0x36) bit fields [p. 172]
- `[7] MY` — Row Address Increment: 0 = vertical-increasing, 1 = vertical-decreasing
- `[6] MX` — Column Address Increment: 0 = horizontal-increasing, 1 = horizontal-decreasing
- `[5:4]` reserved (write 0)
- `[3] RGB` — 0 = RGB, 1 = BGR
- `[2:0]` reserved (write 0)
- Note: bits ML/MH are documented in the flow chart but **not** in the parameter table — datasheet uses only MY/MX/RGB. Default 0x00.

### COLMOD (0x3A) bit fields [p. 176]

- `[3]` reserved
- `[2:0] IFPF[2:0]` — format selector for SPI path

## Initialization sequence

### Power-on timing [p. 100]

| Symbol | Description | Min | Unit |
|---|---|---|---|
| T\_on1 | VDDI on → VCI on delay | > 0 | µs |
| T2 | VDDI on → RESX high | 10 | ms |
| T3 | RESX high → first command | 10 | ms |
| T4 | SLPOUT received → DISPON received | 60 | ms |

### Reset timing [p. 116]
- **t\_RESW** (RESX low pulse width): ≥ 10 µs (5–10 µs may be accepted; < 5 µs rejected)
- **t\_RESET** (reset complete time): ≤ 5 ms when in Sleep In; ≤ 120 ms when in Sleep Out
- After releasing RESX, wait **≥ 5 ms before any command**, and **≥ 120 ms before SLPOUT** [p. 116 Note 5]
- ESD spikes < 50 ns on RESX are rejected by the chip [p. 116 Note 4]

### Recommended firmware boot (QSPI, RGB888)
```
1. Bring up +3V3 → VDDI and VCI energize (Ton1 > 0)
2. Wait ≥ 10 ms after VDDI rises                  [T2]
3. Drive RESX low for ≥ 10 µs, then release       [t_RESW]
4. Wait ≥ 5 ms after RESX rises                   [t_RESET, Sleep In]
5. (Optional) Verify RDDDBS (A1h) → SID=0x3311    [p. 208]
6. SLPOUT (0x11). Wait ≥ 5 ms; do not issue SLPIN for 120 ms.
7. COLMOD (0x3A) = 0x77        ; RGB888 / 24bpp   [v7.2 mandate; p. 176]
8. MADCTL (0x36) = 0x00         ; default orientation, RGB order
   (or 0x08 for BGR if v7.2 panel wired BGR — confirm on bench)
9. CASET (0x2A) = SC=22..(22+409)=431              [v7.2 col offset 22; panel 410 wide]
10. RASET (0x2B) = SP=0..501                       [panel 502 tall]
11. TEON (0x35) = 0x00          ; TE pulses at V-blank only
    (or STESL (0x44) = N        ; TE at scan line N — Mode 3, finer than V-only)
12. WRDISBV (0x51) = 0x00       ; start dark
13. WRCTRLD (0x53) = 0x28       ; BCTRL=1 enables WRDISBV; DD=1 enables dimming
14. Wait until T4 = 60 ms has elapsed since SLPOUT
15. DISPON (0x29)
16. RAMWR-continuous (QSPI: instruction 0x32, addr 0x003C00) for frame upload
17. WRDISBV ramp to target brightness
```
**Wait budget from SLPOUT to first visible pixel: ≥ 60 ms.** [p. 100, T4]

## Operation

### Pixel-write path (QSPI 4-wire, RGB888)
- Frame writes use **instruction `0x32`** followed by 24-bit address `0x003C00` (= RAMWRC at register 0x3C) and then a stream of 3-byte pixels (R, G, B) on all four data lines [p. 22].
- CASET/RASET/MADCTL define the destination rectangle; pixel stream auto-increments within that window and wraps [p. 159, 161, 178].
- For RGB888 in QSPI 4-wire: each pixel = 3 bytes; SI0..SI3 carry nibbles per the timing diagram on p. 22.
- **Restriction: RAMWR must be preceded by CASET/RASET/MADCTL each time the window changes**, else data lands at undefined memory addresses [p. 178].
- The window width and height (EC−SC+1 and EP−SP+1) **must each be divisible by 2** [p. 159, 161].

### Tearing-Effect (TE) [p. 31–33, 170, 180]
- TE is a digital output (VDDI–VSSI levels); pulse rate is tied to internal V-sync (and optionally scan-line N).
- Three modes:
  - **Mode 1** (TEON M=0): V-blank only — single high pulse per frame
  - **Mode 2** (TEON M=1): V-blank + H-blank — every scanline pulses during V-porch
  - **Mode 3** (STESL with N≠0): pulse when scanning reaches line N (1 ≤ N ≤ 1280+porch)
- Signal levels: t\_r, t\_f ≤ 15 ns; thresholds 0.2/0.8 × VDDI; 60 Hz nominal frame rate [p. 33]
- **Active low when in Sleep In** (with TE still enabled) [p. 32, 170]
- v7.2 GPIO45 (QSPI\_TE) is **VDD\_SPI-strap LOW at reset** — the panel must not drive TE high before ESP-S3 reset is released. The internal pull-down on GPIO45 holds this LOW until the panel powers up [v7.2 §Critical GPIO warnings]. The CO5300 datasheet confirms TE is low whenever the command is not activated [p. 14].

### Sleep / wake [p. 143–146]
- **SLPIN (0x10)**: stops DCDC, internal oscillator, panel scan. Takes **~120 ms** to reach booster-off state. Booster-off can be polled via `RDDST (0x09)` bit 31. **Auto-blank**: display goes black on its own — no `0x28 DISPOFF` strictly required first, but v7.2 sequence (`0x28` → `0x10`) is conservative and safe.
- **SLPOUT (0x11)**: restarts oscillator → DCDC → charges panel offset voltage → "Sleep Out mode" after **~120 ms**. Must wait ≥ 5 ms for register reload, ≥ 120 ms before SLPIN can be re-issued. **Full re-init (COLMOD, MADCTL, CASET, RASET, TEON, WRDISBV, WRCTRLD) is required** because user-set values may be reloaded from default during the 5 ms self-diagnostic [p. 145].

### Deep Standby (0x4F DSTB=1) [p. 183]
- Lowest-power state. Exit **only** by driving RESX low > 3 ms.
- For MIPI hosts: also pull HSSI\_CLK/D0/D1 P&N to GND after the command. (Not applicable to v7.2 — QSPI mode.)
- **Datasheet silent on QSPI deep-standby current.** [datasheet silent on QSPI DSTB current; recommend bench test]

### Power-OFF timing [p. 101]

| Symbol | Description | Min | Unit |
|---|---|---|---|
| T\_off1 | VCI off → VDDI off delay | > 0 | µs |
| T5 | DISPOFF received → SLPIN received | > 0 | µs |
| T6 | SLPIN received → RESX low | 83 | ms (@ 60 Hz) |
| T7 | MIPI ULP mode → RESX low | 0 | µs |
| T8 | RESX low → VCI off | 0 | µs |

Note: **VGH ≥ VCI must be maintained during power-off** [p. 101 Note 2]. The on-board Schottky on VGL (recommended D4 = RB520G-30 [p. 225]) handles this.

## Interrupts

CO5300 has **no host-bound interrupt pin.** Only async signal back to the host is the **TE** (Tearing-Effect) line — already documented above.

| Signal | Pin (v7.2) | Polarity | Trigger | Latency | Notes |
|---|---|---|---|---|---|
| TE | GPIO45 | Active HIGH pulse (low at idle / in Sleep In) | V-blank (Mode 1), V+H-blank (Mode 2), or scan line N (Mode 3) | tied to frame rate (~60 Hz → ~16.6 ms period) | t\_r/t\_f ≤ 15 ns [p. 33] |
| TE1 | not connected | (second TE output) | same | same | "If not used, please open this pin." [p. 14] |

**TE clearing:** automatic — TE is a level/pulse output tied to internal V-sync, not a latched interrupt. No host write required to de-assert.

## Datasheet notes

- **GRAM is 1,843,200 bits ≈ 230 400 bytes = 480 × 480 × ⅓ × 24 bits** [p. 6]. The v7.2 panel uses 410 × 502 ≈ 205 820 pixels — within capacity.
- Internal oscillator: **22 MHz nominal, ±2 % over temperature** [p. 103 Note 3].
- VDDI logic-level thresholds: V\_IH ≥ 0.8 × VDDI, V\_IL ≤ 0.2 × VDDI; leakage I\_LIH/I\_LIL ≤ 1 µA at Vin = 0..VDDI [p. 103].
- **BSTM pin = 0 selects 2-PWR boost mode (VDDI + VCI)** — AVDD and VCL come from the internal charge pumps. BSTM = 1 is reserved. Tie BSTM = 0 on the FPC (or via a strap). [p. 14]
- **SWIRE / OLED_EN are outputs** to an *external* power IC, not inputs. With the v7.2 panel — which has no external power IC and runs entirely from +3V3 — both pins are left open per p. 14 ("If not used, please open this pin").
- **MIPI lanes can be programmed to 1- or 2-lane** via CMD2 register xxh; max 500 Mbps/lane with 24-bit data [p. 8]. Not relevant to v7.2 (QSPI path).
- **Operating temperature: −40 °C to +85 °C** [p. 102] — covers wearable use comfortably.
- **Schottky diode required on VGL to GND** (RB520G-30 recommended) for the power-up sequence to work correctly [p. 100 Note 2, p. 225].
- **Self-diagnostic during SLPOUT**: chip auto-runs ESD detect and low-voltage detect during the 5 ms register-reload after SLPOUT [p. 145, §2 Features].
- **OTP / MTP**: device ID, gamma 1–5, GOA timings, panel power settings are stored in OTP — set by the panel vendor (PY206-W38-V2 in v7.2) and **not** to be re-burned by FW [p. 8].
- **External cap requirements (panel-side, not firmware)** [p. 225]: VCI = 2.2 µF, VDDI = 2.2 µF, VREF = 22 nF, VDD = 1.0 µF, VREFP/VREFN = 1.0 µF, ELVDD/ELVSS = 2.2 µF, AVDD = 2.2 µF, VCL = 2.2 µF, VGH = 2.2 µF, VGL = 2.2 µF (max 25 V), plus charge-pump bridge caps C11/12/21/31/41/42 = 1.0 µF each. v7.2's "panel runs from +3V3 only" assumption holds — these caps live on the panel/FPC, not the main PCB.

## Warnings & Errata

| ID | Title | Pages | Severity | Description / Mitigation |
|---|---|---|---|---|
| WARN-01 | **COLMOD reset default is invalid (IFPF=000)** | p. 176 | HIGH | After POR/SWRESET/HWRESET, COLMOD reads `0x70` → IFPF=000, which is not a listed format. FW **must** write `0x3A 0x77` (or another valid format) before the first RAMWR; otherwise pixel decode is undefined. **Mitigation:** include COLMOD=0x77 in every reset/SLPOUT recovery path, never assume the chip "remembers" pixel format across HW reset. |
| WARN-02 | **WRDISBV requires BCTRL** | p. 184, 186 | HIGH | `0x51 WRDISBV` writes a brightness value, but the **dBV[7:0] value is gated by BCTRL bit in `0x53 WRCTRLD`**. If BCTRL=0 (the POR default 0x00), brightness writes have no effect. **Mitigation:** during init, write `0x53 0x28` (BCTRL=1, DD=1) before relying on `0x51`. |
| WARN-03 | **SLPOUT → DISPON minimum 60 ms (T4)** | p. 100 | HIGH | Datasheet specifies T4 ≥ 60 ms from SLPOUT command to DISPON command during power-on. Skipping this delay risks a blank or glitched first frame. Independent of the separate 5 ms / 120 ms restriction-window after SLPOUT. **Mitigation:** budget 60 ms in the wake path, run init writes (COLMOD/MADCTL/CASET/RASET/TEON) during that wait. |
| WARN-04 | **CASET/RASET window must be even** | p. 159, 161 | MED | (EC−SC+1) and (EP−SP+1) must each be **divisible by 2**, due to the source-driver MUX architecture. **Mitigation:** firmware partial-redraw helpers must round window width/height up to even numbers. v7.2 panel 410×502 is even on both axes; arbitrary dirty rectangles are not. |
| WARN-05 | **120 ms wait between SLPIN ↔ SLPOUT** | p. 143, 145 | MED | Either command requires ≥ 120 ms before the opposite command can be sent. Quick toggle sequences will be silently ignored. **Mitigation:** state-machine in FW must rate-limit sleep transitions. |
| WARN-06 | **Deep-standby exit requires RESX low > 3 ms** | p. 183 | MED | Once DSTBON (0x4F DSTB=1) is issued, the **only** way out is a hardware reset pulse. Software cannot rescue the part. **Mitigation:** if FW uses deep standby, GPIO3 (QSPI\_RST) must remain alive during whatever low-power state the ESP is in. v7.2 has GPIO3 as a strap with no internal pull and the panel side driving it — verify that the ESP's RTC GPIO state keeps GPIO3 controllable during light-sleep. |
| WARN-07 | **No published active-current figure** | p. 103 | MED | The DC characteristics table only lists Sleep-In current. Active-mode current draw must be characterized empirically — battery budget cannot be closed from datasheet alone. **Mitigation:** plan a bench measurement during bring-up (open issue v7.2 §5 already lists SLPIN; add active states too). |
| WARN-08 | **Reset spike tolerance** | p. 116 | LOW | RESX pulses 5–10 µs land in the "Reset Start" zone with **indeterminate** behavior. Below 5 µs are rejected; above 10 µs always reset. **Mitigation:** drive RESX low for ≥ 20 µs to stay clear of the ambiguous zone. |
| WARN-09 | **MIPI lanes must be pulled to LP-00/VSSI for deep standby** | p. 13, 183 | LOW | Affects MIPI hosts only. v7.2 ties HSSI\_* to HVSSAM permanently (QSPI mode) — already compliant. |
| WARN-10 | **Sleep-In TE polarity** | p. 32, 170 | LOW | While Sleep In is active and TE is on, TE output is **active low**. Reverses sense vs. active operation. **Mitigation:** FW should mask TE interrupts (or ignore TE polarity) during sleep transitions. |
| WARN-11 | **Confidential datasheet** | p. 1–2 | INFO | The PDF is marked "CHIPONE CONFIDENTIAL" V0.00 (preliminary, 2023-03-28). Subject to silent revision. **Mitigation:** keep the cached copy in `docs/datasheets/`; do not assume a newer public revision matches. |

## Open questions

- **Q1 — RGB565 QIO lane mapping**: v7.2 states "RGB565 QIO lane mapping is broken on CO5300". The datasheet documents RGB565 4-wire write as the `0x32` opcode with two bytes per pixel [p. 24], without flagging any constraint. Is this a CO5300 silicon issue, or a panel-specific (PY206-W38-V2) byte-order issue? **Action:** Ivan — confirm on bench whether RGB565 is fundamentally broken (CO5300) or only the FPC byte-order is wrong on this panel. If only byte-order, RGB565 might still be viable for ½ memory bandwidth.
- **Q2 — Active-mode current**: missing from datasheet. **Action:** add to v7.2 Open Issue list ("Measure CO5300 active-mode current at 0/25/50/75/100 % brightness for battery budget").
- **Q3 — Deep-standby current vs SLPIN current**: datasheet publishes 135 µA total for SLPIN-DSI-LP but says nothing about DSTBON quiescent. **Action:** if FW ever needs to use DSTB instead of SLPIN, bench-characterize.
- **Q4 — VCI = +3V3 vs spec 2.7–3.6V**: spec range includes 3.3 V, but the typical wearable design point is closer to 3.0 V (VCI for analog). Running VCI at v7.2's deliberately-low 3.2 V (TPS62840 setpoint) is comfortably mid-range. No action needed — flagged for completeness.
- **Q5 — `0x0B` MADCTR read returns ML/MH bits?** The MADCTL write parameter table at p. 172 only lists MY/MX/RGB, but the flow chart references ML/MH. The TEON description [p. 170] says "not affected by changing MADCTL bit ML." **Action:** if firmware ever needs to flip ML for vertical-refresh ordering, consult the panel vendor — datasheet is internally inconsistent on this bit.
